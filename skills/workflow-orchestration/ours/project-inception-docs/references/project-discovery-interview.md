# Project Discovery Interview

项目初始化不总是写文件动作。用户在空目录、新目录或只有想法时说“初始化项目”，应先进入需求访谈与项目分类，再决定是否生成启动文档和工程地基。

## 触发语义

出现以下语义时进入访谈模式：

- “我准备开发一个...”
- “初始化项目”，但目录为空、近似空或只有少量占位文件
- “开始问我问题”
- “先收集需求”
- “多轮对话收集需求”
- “不要臆测”
- “按角色提问”
- “用户端 / 管理端 / 租户端 / 授权端分别问我”

若用户明确说“直接打地基”“先生成模板”“先别问，按当前信息初始化”，可跳过深度访谈，但所有未知项必须写成 `待确认`，不能伪造成已确认需求。

## Project Classification Gate

收到“初始化项目 / 新项目 / 给项目打地基 / 接入工作流”后，先判断项目状态：

| 状态 | 判定信号 | 默认动作 |
|---|---|---|
| 空目录新想法 | 目录为空、只有 `.git/`、只有少量占位文件，且无可信 README/docs/src | 先进入 Discovery Interview，不立刻生成完整文档包 |
| 半初始化项目 | 有 README/docs/package/src 的一部分，但目标和状态不清 | 先 scan + State Restore，再问缺口问题 |
| 已有项目后补 workflow | 有代码、历史文档、构建入口或 Git 历史 | 先扫描和恢复现状，补 workflow，不重新做产品访谈 |
| 明确想法输入 | 用户已描述产品、角色、平台或业务流程 | 进入角色化访谈，围绕已给信息追问 |
| 直接地基请求 | 用户明确要求先生成基础文件 | 生成最小地基，未知项标 `待确认`，并把下一步设为继续访谈 |

不要因为用户说“初始化项目”就默认写大量模板。初始化是项目启动流程入口，写文件只是后半段。

## Discovery Interview Rules

- 先声明当前处于“需求访谈模式”，暂不写文件、不初始化完整文档、不开发代码。
- 每轮问题按角色和主题分组，尽量具体，避免开放式大空题。
- 不替用户做产品、商业、合规、技术栈决策；需要推荐时先标明“建议”。
- 每轮结束输出四段：`已确认`、`待确认`、`冲突/风险`、`下一轮问题`。
- 一轮不要无限追问；优先问能决定 MVP、数据模型、权限边界和技术地基的问题。
- 当用户回答后，更新需求摘要，再进入下一轮，直到用户说“可以生成文档 / 初始化项目 / 开始开发”。
- 用户要求“尽量多提问”时，可以一次给较多问题，但仍要分角色分组，便于逐项回答。

## Question Framework

按项目实际角色选择问题组。没有对应角色时跳过，不要硬套。

### 用户端

- 目标用户是谁？他们为什么会打开这个产品？
- 用户第一版必须完成哪 3 个核心动作？
- 用户是否需要登录？登录后有什么权益变化？
- 用户如何浏览、搜索、筛选、收藏、下载、分享或投稿内容？
- 是否有付费、会员、积分、广告、邀请或分销？
- 用户数据是否需要同步到多设备？

### 租户端

- 租户是谁？个人、商家、机构还是内部团队？
- 租户能管理哪些内容、用户、配置和数据？
- 租户之间数据是否完全隔离？
- 租户是否有套餐、容量、功能权限或有效期？
- 租户管理员有哪些角色和操作权限？

### 平台管理端

- 平台管理员要管理哪些租户、用户、内容、订单、配置和风控项？
- 平台是否需要审核流、封禁、下架、申诉、操作日志？
- 平台需要哪些统计报表和运营看板？
- 哪些动作需要二次确认或审计记录？

### 授权与权限端

- 授权对象是什么：租户、用户、功能、内容、API、设备还是域名？
- 授权如何创建、续期、暂停、撤销和过期？
- 授权粒度是套餐、功能开关、数量额度、时间范围还是内容范围？
- 授权异常时用户看到什么，后台如何处理？

### 内容来源与采集端

- 内容来源是否合法、稳定、可追溯吗？
- 采集是手动、半自动还是定时自动？
- 采集字段有哪些：标题、图片、作者、来源链接、发布时间、标签、版权说明？
- 如何去重、归类、打标签、审核、下架和更新？
- 采集失败、来源变更、重复内容、侵权投诉如何处理？

### 商业化与运营

- MVP 先验证增长、留存、转化、内容供给还是收入？
- 是否需要广告、会员、付费下载、企业定制、SaaS 租户套餐？
- 关键指标是什么：DAU、收藏、分享、下载、转化率、租户数、采集成功率？
- 运营人员需要哪些配置入口？

### 合规与风险

- 是否涉及版权、隐私、未成年人、UGC、支付、广告或平台政策？
- 用户协议、隐私政策、内容投诉和下架机制是否需要第一版就有？
- 哪些内容必须敏感词、图片审核或人工审核？
- 数据保留、删除、导出和账号注销如何处理？

### 技术与交付

- 平台形态是什么：微信小程序、H5、App、Web 管理后台、API？
- 是否已有技术栈偏好、云服务、数据库、对象存储、CDN、搜索方案？
- 第一版是否需要管理后台？是 Web 还是小程序内后台？
- 是否需要多环境、日志、监控、备份、导入导出？
- 验收方式是什么：原型、可运行 demo、MVP 上线还是完整生产版？

## Exit Criteria

满足以下任一条件时，才进入文档/地基生成：

- 用户明确说“可以生成文档 / 初始化 / 开始写”。
- 已确认项目目标、目标用户、核心角色、MVP 范围、主要数据对象、权限边界和首轮验证方式。
- 用户要求先按当前信息生成，并接受未知项保留为 `待确认`。

进入生成阶段时，先把访谈结果整理成：

- 项目目标
- 角色与权限
- 核心用户旅程
- MVP / 非目标
- 数据对象
- 内容来源
- 风险与合规
- 待确认问题
- 第一轮工作包
