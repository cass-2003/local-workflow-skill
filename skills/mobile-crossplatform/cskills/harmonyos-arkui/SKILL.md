# HarmonyOS ArkUI 开发

HarmonyOS ArkUI 技能负责把 ArkTS 声明式 UI 需求落到可维护、可验证、可迁移的页面和组件。它覆盖 ArkUI 页面、组件、布局、导航、弹窗、状态管理 V1/V2、V1->V2 状态迁移、MVVM 架构、性能和稳定性，不替代后端接口、支付、地图、系统服务、移动安全或普通 Android/iOS 原生开发技能。

定位：先确认项目版本、页面入口、状态 owner、组件边界、路由方式和验证命令，再实现或审查 ArkUI 代码。

铁律：未读真实 ArkTS/ArkUI 文件、未确认 API/SDK 版本和既有状态管理体系、未跑可用验证命令时，不报告“已完成”。

## 触发边界

必须使用本技能的场景：

- 用户明确提到 HarmonyOS/鸿蒙/ArkUI/声明式 UI/Stage 模型页面/ArkUI 组件/ArkUI 状态管理，并且动作是新增、修改、排查、审查或迁移页面组件。单独 ArkTS/ets 语法、并发或运行时问题交给 `harmonyos-arkts`。
- 新增或修改 ArkUI 页面、业务组件、通用组件、表单、列表、网格、瀑布流、Tabs、Navigation、弹窗、模态页、动画、手势或视觉适配。
- 实现或排查 `@Component`、`@ComponentV2`、`@State`、`@Local`、`@Param`、`@Event`、`@ObservedV2`、`@Trace`、`@Provider/@Consumer`、`@Monitor`、`@Computed` 等状态装饰器。
- 处理 V1 状态管理到 V2 的迁移，包括组件状态、父子传值、跨层级共享、应用级状态、Repeat、组件复用和 animateTo 行为差异。
- 审查 ArkUI 性能、长列表、渲染刷新、build 副作用、内存泄漏、空指针、生命周期和 MVVM 分层。

条件触发：

- 只出现 `HarmonyOS` 但实际是 Map Kit、Push Kit、Wallet、支付、蓝牙、相机、文件、系统权限或原生服务集成时，本技能只处理 ArkUI 展示层，核心实现交给对应领域技能。
- 只出现 `.ets` 文件但用户目标是普通 ArkTS 语法、后端脚本、构建发布或安全审计时，先按任务语义选择相邻技能，本技能最多作为 UI 影响面补充。
- 需求涉及接口字段、数据库、登录鉴权、发布上架或性能压测时，本技能负责 UI 侧契约和状态闭环，必须联动 API、DB、安全、发布或测试技能。

## Anti 场景

不要因为以下情况自动使用本技能：

- 只读学习 HarmonyOS 概念、整理课程、写介绍文案、做竞品调研，不需要实现、修改、排查、审查或迁移 ArkUI 代码。
- 仅目录、README、依赖或文件名里出现 ArkUI/HarmonyOS，没有真实页面、组件、状态或 UI 问题。
- Android 原生 Activity/Compose、iOS SwiftUI/UIKit、Flutter、UniApp、小程序、Vue/React Web 页面，不涉及 ArkUI。
- 只做后端 API、数据库迁移、支付回调、地图 SDK、推送、账号权限、加密、逆向、安全测试或部署发布。
- 只改 UI 文案、海报、营销落地页、设计稿审美，不触碰 ArkUI 页面组件。
- 用户明确要求先不改、只盘点、只审查时，保持只读，不写盘、不自动迁移。

## 起手流程

1. 读用户原话：提取动作、目标页面/组件、涉及状态、迁移意图、验证要求和禁止范围。
2. 读项目证据：查看目标 `.ets`、`module.json5`、路由配置、页面目录、组件目录、状态模型、资源文件和测试/构建脚本。
3. 判定版本体系：确认项目使用 V1、V2、混用还是迁移中；新代码默认 V2，但既有 V1 项目不得为单个小需求强行全量迁移。
4. 建 UI 契约：列页面入口、数据来源、状态 owner、组件 props/events、路由参数、空态、加载态、错误态、权限态、提交态和回退路径。
5. 定最小改动：优先复用现有组件、资源、状态模型、路由方式和样式约束；跨模块改动先说明影响面。
6. 实现后验证：至少跑项目可用的编译、lint、测试或预览验证；无法运行时说明原因和未覆盖风险。

## 快速总则

- 新 ArkUI 代码优先 `@ComponentV2` 和 V2 状态装饰器；维护 V1 代码时先尊重当前体系。
- 同一组件内不要混用 V1 和 V2 状态装饰器；混用只允许作为分批迁移过渡，并要写清边界和回归点。
- 状态必须有唯一 owner：组件内部、父组件输入、事件输出、跨层级共享、应用级状态、ViewModel、路由参数或持久化存储。
- `build()` 保持纯 UI 描述，避免请求、耗时计算、持久化写入、订阅注册和会导致循环刷新的副作用。
- 子组件不直接访问父组件；父到子用输入参数，子到父用事件，跨层级共享用明确的 provider/consumer 或 ViewModel。
- 列表项必须有稳定 key；动态列表不要用 index 作为 key。
- 长列表、网格、瀑布流和高频刷新优先考虑 Repeat、懒加载、缓存数量、组件复用和局部刷新。
- 弹窗和模态页必须区分阻断流程、轻提示、菜单选择、底部半模态、全屏模态和页面级导航。
- 复杂页面使用 MVVM；简单局部组件不为套架构而拆层。
- 性能优化先定位再改动；不要用大范围重构掩盖状态归属不清。

## 状态管理决策

V2 默认选择：

- `@Local`：组件内部状态，必须本地初始化，禁止从外部初始化。
- `@Param`：父到子输入，子组件不直接改写；无法提供默认值时配合 `@Require`。
- `@Param @Once`：外部只初始化一次，随后组件维护本地副本。
- `@Event`：子到父通信；替代 V1 `@Link` 的隐式双向同步。
- `@ObservedV2` + `@Trace`：类、嵌套对象、数组元素和需要属性级刷新的模型。
- `@Provider/@Consumer`：跨层级状态共享；明确 alias，避免隐式匹配。
- `@Monitor`：监听状态变化；说明触发源、更新目标和防循环策略。
- `@Computed`：派生值、校验结果、汇总统计等缓存计算；不要在 getter 内做副作用。

V1 维护选择：

- `@State`：组件内部状态；必须初始化。
- `@Prop`：父到子单向传递；注意本地修改语义和同步范围。
- `@Link`：父子双向绑定；迁移时拆成 `@Param` + `@Event`。
- `@Observed/@ObjectLink`：嵌套对象观测；迁移时改为 `@ObservedV2/@Trace`。
- `@Provide/@Consume`：跨层级共享；迁移时改为 `@Provider/@Consumer`。
- `@Watch`：状态变化监听；迁移时改为 `@Monitor`。

状态建模检查：

- 每个字段写清 owner、初始化来源、更新者、重置时机、是否持久化、是否跨页面共享。
- API 响应不要直接散落进模板；先映射成稳定 view model。
- 表单要区分输入态、校验态、提交态、保存态、错误回填和取消恢复。
- 嵌套对象需要深度观测时，不要只替换装饰器；同步补 `@ObservedV2` 和需要追踪的 `@Trace` 字段。
- 数组、Map、Set、Date 等集合状态要确认变更 API 是否能触发刷新；不能靠重新赋值蒙混过关。

## 强子流程：V1 -> V2 状态迁移

触发条件：

- 用户要求升级、迁移、V1 到 V2、装饰器替换、状态不刷新、双向绑定变化、Repeat 迁移、AppStorageV2、PersistenceV2 或 animateTo V2 兼容。
- 代码中出现 V1 装饰器，且当前任务需要改状态、组件边界、列表渲染或性能稳定性。

迁移前门禁：

1. 确认 API/SDK 版本是否满足 V2 能力；不满足先停止并说明升级前置。
2. 扫描目标范围内的 V1 装饰器和状态来源，形成文件清单。
3. 从叶子组件开始，按功能模块分批迁移；不要一次跨页面全量替换。
4. 标记混用边界：父 V1 子 V2、父 V2 子 V1、共享模型、应用级状态和路由传参都要单独回归。
5. 每批迁移都要有编译和关键交互验证；失败两次仍无法收敛时停手复盘。

核心映射：

| V1 | V2 | 迁移要点 |
| --- | --- | --- |
| `@Component` | `@ComponentV2` | 页面入口 `@Entry` 保持；确认组件内装饰器一起迁移 |
| `@State` | `@Local` 或 `@Param @Once` | 本地状态用 `@Local`；外部初始化一次用 `@Param @Once` |
| `@Prop` | `@Param` | V2 `@Param` 只读；需要本地修改用副本或事件 |
| `@Link` | `@Param` + `@Event` | 显式表达输入和输出，不保留隐式双向 |
| `@Observed` | `@ObservedV2` | 类级观测，属性需按需 `@Trace` |
| `@ObjectLink` | `@ObservedV2` + `@Trace` + `@Param` | 保留对象传递，同时补深度观测 |
| `@Track` | `@Trace` | 属性级精确观测 |
| `@Provide/@Consume` | `@Provider/@Consumer` | alias 作为匹配 key，避免依赖属性名 |
| `@Watch` | `@Monitor` | 检查回调签名、before/now 和循环更新 |
| `@Reusable` | `@ReusableV2` | 复核复用生命周期和列表场景 |
| `LocalStorage` | `@ObservedV2` 单例或页面级模型 | 明确页面隔离和同步方向 |
| `AppStorage` | `AppStorageV2` | 明确跨 Ability 共享和 connect 方式 |
| `PersistentStorage` | `PersistenceV2` | 明确持久化 key、默认值和恢复时机 |
| `Environment` | `UIAbilityContext.config` | 不把环境变量伪装成 UI 状态 |
| `ForEach` | `Repeat` | V2 列表优先 Repeat，保留稳定 key |
| `LazyForEach` | `Repeat` + virtual scroll | 大数据列表迁移时复核首次渲染、更新、拖拽和复用 |
| `$$` | `!!` | 只迁移明确双向绑定语义 |

特殊迁移卡：

- `@State` 外部初始化：不要直接改成 `@Local`；使用 `@Param @Once` 或重新设计父子状态 owner。
- `@Prop` 本地修改：V2 中不能假装仍可双向；用本地副本、`@Param @Once` 或 `@Event` 回传。
- `@Link`：必须写出父组件更新函数；子组件只发事件，不直接改父状态。
- 嵌套对象：类加 `@ObservedV2`，参与 UI 刷新的属性加 `@Trace`；只换组件装饰器会漏刷新。
- `@Provide/@Consume`：迁移时统一 alias；避免 V1 属性名匹配规则迁移后失效。
- `@Watch`：改成 `@Monitor` 后检查是否会回写同一状态形成循环。
- Repeat：使用 `RepeatItem.item` 取值，补稳定 key；大量数据结合虚拟滚动和缓存策略。
- 内置对象：List/WaterFlow 的内置对象、modifier 或 updater 迁移后如不刷新，检查是否需要可观测包装。
- animateTo：V2 中动画前的状态预设可能不会先刷新；按 API 版本选择同步刷新方案，再调用 `getUIContext().animateTo`。

迁移输出必须包含：

- 迁移范围、V1 装饰器清单、V2 映射清单、混用边界。
- 每批次改动文件、状态 owner 变化、父子通信变化、应用级状态变化。
- 编译结果、关键交互验证、长列表/动画/弹窗/导航回归结果。
- 未迁移项、保留 V1 的原因和后续迁移顺序。

## 页面与组件

新增页面：

- 先确认页面路由、Ability/模块位置、资源依赖、入口参数、返回路径、权限和生命周期。
- 页面组件负责布局、路由和组合；业务逻辑进入 ViewModel 或领域服务，不塞进 `build()`。
- 页面必须覆盖初始、加载、成功、空、错误、无权限、提交中、提交失败和恢复路径。

新增组件：

- 先定义组件职责、输入、输出、slot/builder、默认值、禁用态、可访问性和样式边界。
- 组件只接收必要数据；不要把整个页面 model 下传给小组件。
- 可复用组件不要依赖页面路由、全局状态或业务接口。
- `@Builder`、`@BuilderParam`、`@LocalBuilder` 只用于 UI 内容复用，不能隐藏业务状态 owner。

组件审查：

- 查 `build()` 是否有副作用、耗时计算或直接请求。
- 查状态是否多处竞争，是否有父子互相修改、兄弟组件直接访问或全局变量绕过。
- 查生命周期订阅、定时器、观察者、动画和异步任务是否在离开页面或组件销毁时清理。

## 布局、列表与渲染

布局选择：

- Row/Column：简单水平或垂直排列。
- Flex：弹性尺寸、换行、对齐和轻量响应式。
- Stack：重叠、浮层、徽标、遮罩和局部分层。
- RelativeContainer：复杂相对定位和锚点关系。
- GridRow/GridCol：多设备断点和栅格适配。
- List：长列表、分组、粘性标题、侧滑、刷新和滚动监听。
- Grid：规则网格、跨行跨列和自适应列数。
- WaterFlow：不等高卡片和瀑布流。
- Tabs：同级内容切换；不要替代页面级导航。

渲染选择：

- 小型静态列表可用 ForEach。
- 大量数据或滚动懒加载场景用 LazyForEach 或 V2 Repeat + virtual scroll。
- V2 新代码优先 Repeat；保留稳定 key，避免 index。
- 高频刷新列表要检查缓存数量、组件复用、局部更新和图片加载。
- 不在循环里做复杂计算、网络请求、格式化重活或临时创建大对象。

## 导航、弹窗与交互

导航：

- 页面级导航优先 Navigation + NavPathStack；参数解析集中处理。
- 确认系统路由表、命名路由、跨包跳转、返回参数和页面生命周期。
- 直接访问、刷新、返回、替换、非法参数和重复跳转都要有兜底。
- 前端导航守卫只做体验，真实权限由服务端或系统能力兜底。

Tabs：

- Tabs 用于同页面内并列内容；状态是否保留、是否懒加载、是否同步路由 query 要明确。
- 底部 tab、顶部 tab、可滚动 tab 和自定义 tabBar 要分别验证焦点、无障碍和屏幕适配。

弹窗与模态：

- AlertDialog：确认、警告、不可忽略的阻断操作。
- ActionSheet/Menu：操作选择和上下文菜单。
- Popup/Toast：非阻断提示和即时反馈。
- CustomDialog：需要自定义内容但仍是弹窗语义。
- bindSheet：底部半模态、短流程编辑、可保留父视图上下文。
- bindContentCover：全屏模态、临时完整流程。
- 复杂长流程优先 Navigation 页面，不要用半模态承载多步业务。
- 弹窗必须处理关闭、二次确认、重复触发、层级冲突、键盘/焦点和返回键。

事件与手势：

- 点击、触摸、拖拽、焦点、键盘、鼠标和悬停事件要说明目标组件和冒泡/拦截策略。
- 手势冲突先判断优先级、并行/互斥关系和触摸热区，不用大范围禁用交互硬压。
- 动画与手势衔接时，先保证状态更新时序，再追求动效。

## MVVM 与数据流

使用 MVVM 的条件：

- 页面有复杂业务状态、接口聚合、表单、列表编辑、跨组件共享、缓存或多入口复用。
- 需要测试 ViewModel 逻辑，或需要降低页面组件体积和副作用。

分层规则：

- Model：数据结构、数据访问、存储和业务规则；不直接操作 UI。
- ViewModel：页面状态、命令、派生值、加载和错误处理；可用 V2 可观测模型驱动 UI。
- View/Page：UI 展示、用户输入、路由和组件组合；通过 ViewModel 方法改变业务状态。
- View 不直接调用 Model；Model 不依赖 View；兄弟组件不互相访问。

目录建议：

- `model/` 放领域模型和 DTO 映射。
- `viewmodel/` 放页面或业务模块 ViewModel。
- `view/` 放业务组件。
- `pages/` 放页面入口。
- `common/` 放不依赖业务状态的通用组件。

## 性能与稳定性

性能门禁：

- 减少组件嵌套和重复布局；复杂布局先测量再重排。
- 长列表使用懒加载、缓存、稳定 key、图片尺寸约束和局部刷新。
- 计算密集逻辑放到 ViewModel、缓存或异步流程；不要在 `build()` 或循环生成器里重复执行。
- 图片、视频、富文本和自定义绘制要控制尺寸、生命周期和资源释放。
- 动画优先使用 transform/opacity 等低成本属性；避免频繁触发布局重计算。

稳定性门禁：

- 所有外部输入、路由参数、接口字段和资源读取都要处理 null、undefined、空数组、非法枚举和旧数据。
- 异步请求要处理取消、重复点击、页面离开、旧响应覆盖新状态和失败重试。
- 订阅、定时器、观察者、长动画、弹窗控制器和外部资源必须有清理路径。
- 表单提交要防重复，失败后保留用户输入并回填字段错误。
- 错误态和空态不能互相替代；权限态不能只靠按钮隐藏表达。

## 实现场景卡

新功能：

- 先接页面骨架和状态模型，再接真实数据和交互。
- 每个状态字段必须能回答：谁初始化、谁更新、谁读取、何时重置、失败如何恢复。
- UI 完成不等于功能完成；必须验证交互、异常、返回、刷新和设备适配。

修改现有页面：

- 先定位目标组件和状态 owner，不做跨目录风格化重构。
- 修改 props/events 前先查所有调用方。
- 改共享组件时列影响页面和回归路径。

修 bug：

- 先复现，再按状态、渲染、生命周期、路由、弹窗层级、异步竞态、资源和版本差异分类。
- 禁止只加延时、强制刷新或全量重建组件来掩盖根因。
- 修复后跑原失败场景和至少一个相邻正常场景。

代码审查：

- 输出问题按严重程度排序，给文件/行号、影响、触发条件和修复建议。
- 重点查状态 owner、V1/V2 混用、列表 key、build 副作用、资源泄漏、错误态缺失和验证缺口。

## 输出要求

实现类任务输出：

- 改动文件和改动点。
- 状态 owner、组件边界、路由/弹窗/布局选择和 V1/V2 决策。
- 验证命令、结果和未覆盖风险。
- 如未运行验证，明确写“未跑”，并说明原因。

迁移类任务输出：

- V1 装饰器清单、V2 映射清单、迁移批次和保留项。
- 混用边界、风险点、回归页面、失败恢复策略。
- 编译/交互/性能验证结果。

审查类任务输出：

- 先列问题，再列证据；不要先写总结。
- 每个问题包含位置、影响、复现或触发条件、建议修复。
- 若没有发现问题，说明已查范围和仍未覆盖的验证。

无法完成时：

- 写清卡点是缺文件、缺版本信息、缺运行环境、缺依赖、接口不通还是需求冲突。
- 不编造 API、路径、验证结果或官方能力。

## 提交前自检

- frontmatter 是否只有 `name` 和 `description`。
- 是否确认 ArkUI 版本体系、页面入口、状态 owner、路由方式和验证命令。
- 是否避免同一组件混用 V1/V2 装饰器。
- V1->V2 迁移是否走了清单、映射、批次、混用边界和回归验证。
- 列表是否有稳定 key，长列表是否考虑懒加载或 Repeat。
- `build()` 是否保持纯 UI 描述，无请求、订阅、持久化和重计算副作用。
- 弹窗、模态和导航是否有关闭、返回、重复触发和异常路径。
- MVVM 是否按复杂度使用，没有为了简单组件过度拆层。
- 性能和稳定性是否覆盖空值、异步竞态、资源释放和错误恢复。
- 输出是否包含改动文件、关键取舍、验证结果和未覆盖风险。
