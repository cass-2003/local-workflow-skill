# 天地图

首次自称：天地图（tianditu-map，兼容 slug: tianditu-map）。

## 定位/适用范围

本技能用于在真实项目中设计、开发、调试、测试或发布天地图相关能力，目标是让一次实现更接近可上线标准，而不是只给示例链接。

适用场景：

- Web 端或移动端接入天地图 JavaScript API、H5 容器、原生 WebView、第三方 GIS 框架中的天地图底图。
- 使用天地图 WMTS/WMS 服务作为底图、影像、注记、地形或专题图层来源。
- 后端接入、网关转发、内网代理、缓存、审计、限流、可观测性和发布验证。
- 政务、自然资源、测绘、应急、城管、交通、水利、农业、园区等系统中的地图展示与业务叠加。
- 处理行政区、底图、影像、注记、矢量、地形、地名、兴趣点、路线、专题图层与业务空间数据的叠加。
- 处理 CGCS2000、WGS84、Web Mercator、EPSG:4490、EPSG:4326、EPSG:3857 等边界问题。
- 处理官方 JS API、WMTS、WMS、服务域名、图层命名、c/w 瓦片矩阵、token、归属标识和审图信息的工程落地。

不适用场景：

- 只学习天地图概念、只做只读解释、入门导览或 onboarding，不改代码、不调试、不测试、不发布。
- 仅凭 README、依赖包名、package lock、路由名或配置注释出现 map/tianditu 字样，但没有实际修改、排障、验证或发布动作。
- 只问 Google/高德/百度/腾讯/Mapbox/ArcGIS/Leaflet 的独立接入，且没有天地图迁移、互操作、替换、兼容或对比需求。
- “source map、hash map、roadmap、mind map、site map、map-filter-reduce、map/reduce、Map 数据结构”等非地理地图语义。

## 铁律

1. 先确认坐标系和瓦片矩阵集，再写图层代码；不要凭经验混用 CGCS2000、WGS84 和 Web Mercator。
2. 天地图 token/tk 只能按官方授权使用；不得绕过 token、盗用 token、硬编码泄露生产 token。
3. 不抓取、批量下载、离线囤积或重分发天地图瓦片；缓存只能服务于授权业务、稳定性和性能目标。
4. 不隐藏、裁剪、遮挡或误导天地图及审图号、版权、来源归属等标识。
5. 政务和测绘系统先查数据来源、密级、公开范围、脱敏要求、坐标转换资质和成果发布边界。
6. 所有后端代理必须有鉴权、限流、来源校验、日志审计和缓存边界；禁止开放式公共瓦片代理。
7. 测量、面积、长度、边界判定、规划审批、权属认定等功能不得暗示具备法定测绘或审批效力，除非项目具备对应资质、数据和流程。
8. 发布前必须提供可复现证据：关键图层截图、网络请求、坐标校验、token 策略、异常回退、缓存策略和合规说明。
9. 遇到坐标系不明、官方能力未确认、token 授权不明、涉密/敏感数据边界不清、代理安全无法收敛时，先停止实施并要求补证据，不用猜测补齐。
10. 任何“临时能用”的地图改法都必须说明生产禁止条件；不能把测试 token、公开代理、未审图数据、未授权缓存作为上线方案。

## 官方端点速查

- 服务域名：`t0.tianditu.gov.cn` 到 `t7.tianditu.gov.cn` 共 8 个负载均衡子域；同一瓦片源建议按 `(x+y) % 8` 或 `z % 8` 分发，避免单域名浏览器并发上限（HTTP/1.1 一般 6）。
- WMTS KVP 模板：`https://t{0-7}.tianditu.gov.cn/{layer}_{matrix}/wmts?SERVICE=WMTS&REQUEST=GetTile&VERSION=1.0.0&LAYER={vec|img|ter|cva|cia|cta|ibo|eva}&STYLE=default&TILEMATRIXSET={c|w}&FORMAT=tiles&TILEMATRIX={z}&TILEROW={y}&TILECOL={x}&tk={token}`，路径段中的 `{layer}_{matrix}` 与查询参数 `LAYER`、`TILEMATRIXSET` 必须三处一致。
- XYZ 模板只在框架或网关明确提供 z/x/y URL 适配时使用；不得把 WMTS KVP 参数随手改名为 XYZ。核对 z/x/y 顺序、行列原点、级别起点、TILEMATRIXSET、子域名和 token 位置；`w` 矩阵更接近常见 Web Mercator XYZ，`c` 矩阵是经纬度瓦片，必须显式配置投影和瓦片网格。
- 图层代码：`vec` 矢量底图、`img` 影像底图、`ter` 地形晕渲、`cva` 矢量中文注记、`cia` 影像中文注记、`cta` 地形中文注记、`eva` 矢量英文注记、`eia` 影像英文注记、`ibo` 全球境界。
- 矩阵集 `c`：EPSG:4490/CGCS2000 经纬度，原点 `(-180, 90)`，0 级 2 张瓦片（每瓦 180°），通常支持 1-18 级，`TILEMATRIX` 直接用整数级别。
- 矩阵集 `w`：EPSG:3857 Web Mercator，原点 `(-20037508.3427892, 20037508.3427892)`，0 级 1 张瓦片，通常支持 1-18 级，与 Google/OSM 切片体系一致。
- Web 服务（地名/逆地理/路径/行政区）域名为 `api.tianditu.gov.cn`，常见路径：地名搜索 `/v2/search`、地理编码 `/geocoder`、逆地理 `/v2/geocoder`、行政区 `/administrative`、驾车 `/v2/drive`、公交 `/v2/transit`、步行 `/v2/walk`；统一通过 `tk` 鉴权。
- Web 服务错误不要只看 HTTP 状态；同时记录响应体 status/msg/infocode 或官方返回信息码。地名、行政区、逆地理等服务要把 401/403/429/5xx、参数错误、无结果、越界、配额不足和 key 限制不匹配拆开处理。
- JS API v4.0 命名空间 `window.T`，与 v3.0 在控件、覆盖物、坐标对象上不完全兼容；混用 v3/v4 脚本会导致原型方法缺失。

## 快速总则

接到天地图任务后，按以下顺序收敛方案：

1. 明确业务动作：新增接入、修复偏移、叠加图层、代理瓦片、迁移底图、发布上线、性能优化、合规审查。
2. 明确消费者：天地图 JS API、OpenLayers、ArcGIS JS、Cesium、Leaflet 适配、移动 WebView、后端服务、内网大屏。
3. 明确服务类型：JS API、WMTS、WMS、地名搜索、行政区、路线规划、地理编码、专题服务或自有 GeoServer/MapServer 服务叠加。
4. 明确坐标链路：采集坐标、存储坐标、服务坐标、前端渲染坐标、用户输入坐标、导入导出坐标。
5. 明确图层组合：矢量底图 vec、影像底图 img、地形 ter、中文注记 cva/cia/cta，以及业务专题图层。
6. 明确瓦片矩阵：CGCS2000 经纬度矩阵通常对应 c，Web Mercator 通常对应 w；二者 URL、分辨率、级别、原点和切片编号不可混搭。
7. 明确安全边界：token 放置位置、域名白名单、服务端签发或代理、缓存 TTL、日志脱敏和异常告警。
8. 明确官方依据：以天地图官方文档、服务能力响应、GetCapabilities、项目授权和单位制度为准；不要复制网上旧 URL 或未知 token 示例。
9. 明确验收证据：和官方或权威底图对齐、行政区边界对齐、点位偏移阈值、跨缩放级别一致、弱网和错误码可恢复。

## 强制流程

### 1. 需求定界

- 写下用户要完成的真实动作：开发、调试、迁移、接入、部署、测试、审计或修复。
- 标出涉及对象：底图、影像、注记、行政区、专题图层、瓦片代理、坐标转换、token、CORS、缓存。
- 判断是否政务/测绘/涉密/公共发布；若是，先加入合规和归属要求，不等到上线前补。

### 2. 坐标与投影确认

- 记录数据源坐标：CGCS2000、WGS84、GCJ-02、BD-09、本地投影、地方独立坐标或未知。
- 记录展示坐标：天地图 JS API、WMTS c、WMTS w、第三方 GIS 框架的 view projection。
- 区分概念边界：CGCS2000 常用 EPSG:4490 表达，经纬度轴和 EPSG:4326/WGS84 接近但不能默认等同（EPSG:4326 基于 WGS84，与 4490 椭球参数不同但日常米级业务可近似互用，测绘成果不可省略转换）；Web Mercator 使用 EPSG:3857，对应 `w` 矩阵；`c` 矩阵按经纬度瓦片组织，分辨率以度为单位（0 级 ≈ 0.703125°/px）。
- WMS 轴顺序陷阱：1.3.0 + EPSG:4490/4326 是 `lat,lon`；1.1.1 + EPSG:4326 是 `lon,lat`；BBOX 写反在中国境内会偏移到南极/印度洋方向，且无报错。
- 对未知坐标不得盲转；必须用已知控制点、行政边界或权威样例验证。
- 对中国境内互联网地图偏移问题保持敏感：业务数据、天地图底图和第三方底图若来源不同，可能存在加密、纠偏或投影差异。
- 后端存储建议保留原始坐标字段、坐标系字段、转换版本、转换参数来源和转换时间，避免二次转换不可追溯。

### 3. 服务与图层设计

- JS API 优先用于天地图原生交互、控件、覆盖物、搜索、行政区等能力。
- WMTS 优先用于标准 GIS 框架、后端地图服务拼接、跨端统一底图和大屏渲染。
- WMS/自有专题服务用于动态专题、查询和样式控制；上线前核对 GetCapabilities、CRS/SRS、BBOX 轴顺序、LAYERS、STYLES、FORMAT、TRANSPARENT、QUERYABLE 和比例尺限制，不要把动态业务查询伪装成无限瓦片抓取。
- 底图和注记分层加载：影像 img 配中文注记 cia，矢量 vec 配中文注记 cva，地形 ter 配地形注记 cta；服务命名常见为 img_c/img_w、vec_c/vec_w、ter_c/ter_w、cia/cva/cta 与对应矩阵集组合，最终以官方文档和服务能力为准；不要把注记当作可随意隐藏的版权层。
- 图层命名、样式、格式和级别范围必须从官方文档、服务响应或项目配置中二次核对；遇到 vec/img/cva/cia 与 c/w 组合不一致，优先怀疑配置而不是坐标数据。
- 行政区边界、业务面、热力、轨迹、聚合点应明确层级、样式、透明度、点击优先级和最大数据量。

### 4. Token 与安全设计

- 开发、测试、生产 token 分离；生产 token 不提交到仓库，不写入 README、截图或前端源码常量。
- 区分浏览器端 key 与服务端 key：浏览器端 key 走域名/Referer 白名单，服务端 key 走 IP 白名单，移动端 key 走 Bundle ID/包名校验；不要把服务端 key 直发前端，也不要让浏览器端 key 在无 Referer 的后端场景使用，否则会稳定 403。配额、并发、日调用量、商用授权和政务授权以控制台、合同或项目批复为准；上线前必须按实际图层数、缩放行为、用户量和缓存命中率估算容量。
- 前端直连必须使用官方支持的域名、referer 或应用限制；若无法保护 token，评估服务端签发、短期配置下发或受控代理。
- 代理服务必须校验用户身份、业务权限、目标白名单、参数白名单、频率和来源；禁止任意 URL 转发。
- 日志中脱敏 tk/token、坐标敏感点、内部行政区编码和用户查询条件。
- 失败时不要自动切换到未授权 token、第三方盗链瓦片或无 attribution 的镜像源。

### 5. 前端实现

- 初始化地图前先确定容器尺寸、设备像素比、缩放级别、最小/最大级别、中心点和投影。
- 加载 WMTS 时逐项核对 SERVICE、REQUEST、VERSION、LAYER、STYLE、FORMAT、TILEMATRIXSET、TILEMATRIX、TILEROW、TILECOL、tk。
- 明确选用 WMTS KVP（GetTile?SERVICE=WMTS&...）或 RESTful URL 模板两种调用方式之一；OpenLayers/Leaflet/ArcGIS/Cesium 各框架对 URL 模板、TileMatrix 命名、origin、行列起点的默认值不同，需以服务 GetCapabilities 为准显式配置，不要混用两种风格导致 404 或错位。
- 使用 XYZ 风格 URL 时，把它当作适配层而不是新数据源：必须写清楚真实上游是天地图 WMTS、官方 RESTful 瓦片还是受控代理；缓存键仍包含 layer、matrix、z、x、y、style、format、语言和 token 环境。
- 对 t0 到 t7 子域名做合理轮询或由库处理；不要无限扩大并发。OpenLayers/Leaflet 的 `subdomains` 配置直接传 `['0','1','2','3','4','5','6','7']`，URL 模板里写 `{s}`/`{0-7}` 占位。
- JS API v4 与 v3 不通用：v4 的 `T.Map`、`T.TileLayer.WMTS`、`T.LngLat` 等接口签名与 v3 不同，引入 `<script src=".../api?v=4.0&tk=...">` 后不要再混用 v3 示例代码；插件（搜索 `LocalSearch`、路线 `DrivingRoute`、行政区 `ProvinceData` 等）必须再引一次 `&type=...` 子模块或对应插件 JS。
- 业务点位叠加前先做一组固定控制点测试：省会、县区中心、项目现场、边界附近点。
- 注记层和业务标注需要处理遮挡：行政区名、道路名、项目点名不可互相覆盖导致误读。
- 移动端注意触摸手势、WebView 内核、离线弱网、瓦片内存、定位权限、HTTPS 混合内容和高 DPI 模糊。

### 6. 后端/代理实现

- 只代理明确允许的天地图服务域名、路径和参数；拒绝任意外部 URL、file、内网地址和重定向跳转。
- 建立缓存键：服务类型、图层、矩阵集、级别、行列、样式、格式、语言、token 所属环境；不要把用户身份误混进公共瓦片缓存，也不要把敏感业务图层当公共缓存。
- 设置合理 TTL、失败缓存、熔断、重试、超时、并发上限和上游错误透传。
- CORS 只开放业务域名；不要使用星号配合凭证；预检请求要覆盖真实方法和请求头。
- 保留审计日志：调用方、图层、级别、行列范围、状态码、耗时、命中缓存、错误类型，但脱敏 token。
- 内网部署要确认 DNS、证书、NTP、代理出口、防火墙、网闸、IPv6/IPv4 和备用链路。

### 7. 政务/测绘合规

- 明确天地图数据、项目自有数据、第三方数据、用户上传数据的授权边界。
- 涉及测绘成果、地理信息公共服务、行政区划、敏感设施、精确坐标公开时，按单位制度和法律法规走审查。
- 对外公开地图不得擅自发布未审图底图、未授权专题图或可反推敏感位置的数据。
- 对内部系统也要区分可视化参考、业务辅助、审批依据、法定成果；不同等级需要不同证据和责任链。
- 页面、报告、导出图片、打印件、截图接口都必须保留必要的来源和归属信息。

### 8. 测试、发布和回滚

- 本地、测试、预生产、生产分别验证 token、域名、HTTPS、CORS、缓存、限流和错误处理。
- 至少覆盖：首次加载、缩放、平移、切换底图、叠加注记、点击查询、行政区定位、移动端旋转屏、弱网、上游 4xx/5xx。
- 真机验收必须覆盖目标 Android/iOS/WebView 或政务终端：定位权限拒绝、后台返回、横竖屏、弱网切换、HTTPS 证书、缓存清理、高 DPI、触控惯性、内存回收和长时间运行；模拟器、桌面浏览器和开发者工具不能替代真机结论。
- 发布包中记录版本、配置差异、token 配置方式、缓存刷新方式、回滚开关和监控面板。
- 出现大面积瓦片失败时优先降级业务图层或提示服务异常，不得自动切到未授权源。

## 场景执行卡

### 场景 A：天地图 JS API 接入 Web 页面

- 先确认 API 版本、官方 script 地址、tk 获取方式、域名限制和 HTTPS 策略。
- 地图初始化要固定中心点、缩放级别、最小/最大级别和控件；避免默认值导致全国视角或空白容器。
- 覆盖物统一封装生命周期：创建、更新、隐藏、销毁、事件解绑。
- 若叠加业务点偏移，先检查坐标来源和转换链，不要用 CSS 或像素偏移硬修。
- 验收证据：页面截图、网络请求、token 未泄露策略、点位对齐截图、错误态截图、归属/审图信息可见截图。

### 场景 B：WMTS 接入 OpenLayers/ArcGIS/Cesium/Leaflet 适配层

- 明确使用 c 还是 w 矩阵集，并让 View/Scene 的 projection、resolutions、matrixIds、origin、extent 与之匹配。
- 图层名、注记层和矩阵集成对出现：vec_c 配 c，vec_w 配 w；img、cia、cva、ter、cta 同理按服务命名核对。
- 若框架默认 EPSG:3857，而服务使用 CGCS2000 经纬度矩阵，必须显式配置投影和瓦片网格。
- 若框架默认从 0 级开始而服务级别或矩阵 ID 不同，必须显式映射 matrixIds。
- 验收证据：多级别截图、瓦片 URL 样例、matrixIds/resolutions 配置来源、边界点对齐、无 404/跨域错误、c/w 错配回归用例。

### 场景 C：后端瓦片代理或内网网关

- 只允许固定上游：天地图官方域名和项目批准的自有地图服务。
- 参数白名单包括图层、矩阵集、级别、行列、格式、样式；拒绝任意 URL、任意 header 注入和路径穿越。
- 代理隐藏服务端 token 时，前端不得还能传入任意 tk 覆盖。
- 对高频瓦片加缓存和限流，对专题查询加鉴权和审计。
- 验收证据：安全测试、SSRF/重定向拦截、缓存命中率、TTL 和清理策略、上游 403/429/5xx 降级、日志脱敏、CORS 配置。

### 场景 D：政务专题图层叠加

- 明确专题数据来源、更新时间、公开范围、字段敏感性和坐标系。
- 点、线、面分别控制简化、抽稀、聚合和样式；大数据量不直接全量塞到前端。
- 行政区边界用于筛选或展示时，标注版本和来源；不要把视觉边界当法定权属边界。
- 导出图片、打印、报告要保留底图来源、专题数据来源、时间和用途限制。
- 验收证据：数据字典、坐标验证、权限验证、导出样例、审查意见或合规说明。

### 场景 E：坐标偏移与图层错位修复

- 收集至少三个已知控制点，覆盖不同区域和缩放级别。
- 判断偏移是常量偏移、缩放相关偏移、投影错误、经纬度顺序错误、GCJ/BD 混用、瓦片矩阵错误还是数据源错误。
- 检查经纬度顺序：经度 longitude/x 在前，纬度 latitude/y 在后；但某些接口或数据库字段可能相反。
- 不用 magic number 修正生产坐标；修复应落在坐标转换、投影配置、矩阵集配置或数据清洗层。
- 验收证据：修复前后对比、控制点表、转换函数版本、回归测试。

### 场景 F：移动端或大屏性能优化

- 先定义目标设备、真机型号、浏览器/WebView、网络、屏幕分辨率、运行时长和可接受指标；低端机、国产化终端和政务大屏不能只用开发机浏览器代替。
- 限制同时可见图层数量、瓦片并发、标注数量和矢量要素数量。
- 移动端优先聚合、简化和按视窗加载；大屏优先缓存、预热和分层降级。
- 监控 FPS、内存、瓦片失败率、首屏时间、交互延迟和接口耗时。
- 避免在 pan/zoom 高频事件中做全量空间计算；使用节流、索引、worker 或后端切片。
- 验收证据：性能基线、优化前后指标、低端机或目标大屏实测。

### 场景 G：从其他地图源迁移到天地图

- 列出原地图源的坐标、投影、瓦片矩阵、缩放级别、样式、注记、行政区和 API 能力差异。
- 不假设 zoom、tile row/col、extent、origin、分辨率完全兼容。
- 将 provider 封装层与业务图层解耦，避免把天地图 token 或 URL 散落在业务组件中。
- 对原有 POI、路线、逆地理编码能力逐项替代或保留互操作方案。
- 验收证据：迁移差异表、关键页面截图、坐标对齐、功能回归、合规更新。

### 场景 H：WMS 或自有专题服务叠加

- 先读取 GetCapabilities 或项目服务说明，确认 VERSION、CRS/SRS、LAYERS、STYLES、FORMAT、TRANSPARENT、BBOX、WIDTH/HEIGHT、INFO_FORMAT 和 QUERYABLE。
- WMS 1.3.0 + EPSG:4326/4490 BBOX 顺序是 `minLat,minLon,maxLat,maxLon`；WMS 1.1.1 + EPSG:4326 是 `minLon,minLat,maxLon,maxLat`；EPSG:3857 永远是 `minX,minY,maxX,maxY`（米）。错配在中国境内通常表现为图层飞到非洲或南极，且服务端不报错。
- 动态专题图层要设置比例尺、视窗过滤、权限过滤、超时和空结果提示；禁止用全域高频 WMS 请求替代授权瓦片缓存。
- 验收证据：GetCapabilities 摘要、GetMap/GetFeatureInfo 请求样例、坐标/轴顺序验证、权限验证、性能和错误态截图。

## 验证门禁

上线或交付前必须通过以下门禁：

- 坐标门禁：已记录所有输入/存储/输出坐标系；至少三个控制点在目标缩放级别对齐；无经纬度顺序错误。
- 图层门禁：底图、影像、注记、地形、专题图层的 LAYER、TILEMATRIXSET、STYLE、FORMAT 配置可追溯。
- Token 门禁：生产 token 不在仓库、日志、截图、浏览器可枚举配置中泄露；域名或应用限制已启用。
- 安全门禁：代理无 SSRF、开放转发、任意 header、任意 tk 覆盖、无限并发、无限缓存问题。
- 合规门禁：来源归属、审图号/版权标识、数据授权、公开范围、导出物标识已确认。
- 稳定性门禁：上游 401/403/404/429/5xx、DNS 失败、超时、CORS 失败、弱网均有可解释行为。
- 性能门禁：首屏、缩放、平移、图层切换、移动端内存和大屏长时间运行满足项目指标。
- 发布门禁：配置差异、回滚方案、监控告警、缓存刷新、验收截图和测试记录齐全。
- 一次通过门禁：每个高风险点都有“证据位置、责任人/系统、失败表现、回滚方式”；缺任一项不得宣称 ready。

最低测试证据包：

- token/tk：缺失、错误、过期、域名不匹配、额度不足时的 401/403/429 表现和用户提示。
- 浏览器端 key 与服务端 key 错用、IP 白名单/Referer 白名单未配置时的 403 复现与排查路径。
- 图层/矩阵：vec/img/ter 与 cva/cia/cta 组合，c/w TILEMATRIXSET 错配，WMTS 与 XYZ 适配层错配，EPSG:4490 与 EPSG:3857 对齐验证。
- 服务能力：WMTS 请求样例、WMS GetCapabilities 摘要、GetMap/GetFeatureInfo 样例或不适用说明。
- 网络与安全：CORS 预检、HTTPS、DNS、超时、5xx、代理白名单、SSRF 拦截、缓存 TTL 和清理策略。
- 合规与展示：attribution、审图号/来源归属、导出图片/打印件保留标识、敏感位置脱敏或权限控制。
- 终端体验：目标浏览器、低端移动设备或 WebView、大屏长时间运行、弱网/断网恢复、瓦片失败率和内存上限。
- 真机体验：至少一台目标移动真机或政务终端完成加载、定位、缩放、旋转、后台返回、断网恢复、缓存清理和错误提示验证。

- 图层清单：底图、影像、注记、矢量、地形、专题图层及显示顺序。

- 合规说明：数据来源、授权边界、归属标识、导出物处理、政务/测绘限制。
- 测试证据：截图、请求样例、错误态、性能指标、移动端或目标浏览器结果。

## 安全边界

必须拒绝或改写以下请求：

- 绕过、破解、共享、盗用、批量生成或隐藏天地图 token。
- 抓取、批量下载、镜像、离线囤积、二次售卖或未授权重分发天地图瓦片和数据。
- 移除、遮挡、伪造天地图来源、版权、审图号、归属标识。
- 搭建无鉴权、无限流、可访问任意 URL 的公共瓦片代理。
- 将内部敏感坐标、涉密设施、未公开政务数据直接发布到公网地图。
- 在无资质、无权威数据、无审批流程时，声称测量结果、边界、面积、距离、审批结论具有法定效力。
- 为规避官方限制而切换到盗链、镜像、泄露 token、浏览器自动化批量请求或伪造 referer。

可以提供的安全替代：

- 使用官方授权、合法 token、域名白名单和服务端受控代理。
- 设计最小权限、限流、缓存、审计和异常降级。
- 用抽样控制点、官方服务和合规数据源做对齐验证。
- 对敏感坐标做脱敏、聚合、网格化或内部可见控制。

## 反例库

- 反例：页面里直接写生产 tk，提交到仓库。正确做法：环境隔离、域名限制、构建注入或服务端受控下发，并确保日志和截图脱敏。
- 反例：vec_c 底图配 EPSG:3857 默认视图，靠移动中心点修正错位。正确做法：统一矩阵集、投影、分辨率和 matrixIds。
- 反例：影像底图不加载注记层，且隐藏归属信息。正确做法：按业务加载注记，保留来源和审图信息。
- 反例：后端代理接受 url 参数并原样请求。正确做法：固定上游白名单和参数白名单，拒绝重定向和内网地址。
- 反例：把 WMTS KVP 地址直接改成 `/z/x/y` 就声称完成 XYZ 适配。正确做法：确认官方或网关真实模板，显式配置投影、原点、分辨率、matrixIds、行列顺序和缓存键。
- 反例：把行政区边界截图用于权属或审批结论。正确做法：标明可视化参考，法定用途需权威数据和审批流程。
- 反例：遇到 403 自动切到网上找到的公共 token。正确做法：检查授权、域名、额度、环境配置和官方服务状态。
- 反例：为了性能预下载全国瓦片。正确做法：按授权缓存必要范围，设置 TTL、限流和清理策略。
- 反例：大屏一次渲染几十万业务点。正确做法：聚合、抽稀、视窗加载、后端切片或专题瓦片化。
- 反例：只在桌面 Chrome 或模拟器验收移动地图。正确做法：目标真机验证 WebView、权限、弱网、横竖屏、内存、后台返回和触摸体验。

## 自检清单

- 是否明确天地图服务类型和具体图层？
- 是否明确 CGCS2000/WGS84/Web Mercator/EPSG 边界，并验证控制点？
- 是否区分 c 与 w 瓦片矩阵集，并区分 WMTS KVP、RESTful 和 XYZ 适配层，避免 URL 和前端投影混搭？
- 是否覆盖底图、影像、注记、矢量、地形、专题图层的组合关系？
- 是否处理 token、域名限制、环境隔离、日志脱敏和泄露风险？
- 是否处理 CORS、缓存、代理、限流、重试、超时和上游错误？
- 是否保留天地图来源、版权、审图号或必要归属信息？
- 是否考虑政务、测绘、敏感坐标、公开范围和数据授权？
- 是否覆盖 Web、移动端、后端、内网或大屏的实际消费者？
- 是否完成目标移动真机、WebView、政务终端或大屏实测，而不是只用模拟器和桌面浏览器？
- 是否有测试证据、发布证据、回滚方案和监控指标？
- 是否避免 token 绕过、瓦片抓取、无授权重分发和不当测绘声明？
- 是否把 source map/hash map/roadmap/mind map/site map/map-filter-reduce 等误触发排除？
- 是否在证据不足时停止并列出缺口，而不是用默认值、旧示例或猜测继续？

## 相邻技能边界

- 地图 GIS 核心开发 / map-gis-core（map-gis-core）：通用 GIS 架构、空间索引、投影和图层模型优先交给该技能；涉及天地图官方服务、tk、WMTS 参数和政务接入时使用本技能。
- JavaScript / TypeScript 开发 / javascript-typescript-development（jsts）：复杂几何拓扑、缓冲区、相交、裁剪、合法性修复使用 jsts；本技能只规定天地图展示和合规边界。
- Vue 开发 / vue-development（vue）：Vue 组件生命周期、状态管理、路由和构建问题使用 vue；地图图层、坐标、token 和天地图服务配置使用本技能。
- API 工程 / api-engineering（api）：通用接口设计、鉴权、错误码使用 api；瓦片代理、地图服务参数白名单和上游稳定性使用本技能。
- 测试验证 / test-engineering（tst）：测试框架和自动化测试实现使用 tst；本技能给出地图专项验收门禁和证据要求。
- 代码审计 / code-audit（aud）：审计、合规报告和发布审查使用 aud；天地图归属、测绘边界和地图数据合规要求由本技能提供输入。
- Web 安全 / web-security（wsec）：Web 安全、SSRF、CORS、密钥保护使用 wsec；天地图 token、代理白名单和瓦片安全边界由本技能细化。
- 数据库工程 / database-engineering（db）：空间数据存储、索引、迁移和查询优化使用 db；坐标字段、来源记录和转换链路需与本技能一致。
- 性能工程 / perf-engineering（pfe）：前端性能优化使用 pfe；地图瓦片、标注、移动端和大屏专项性能由本技能约束。
- 发布部署 / release-engineering（rls）：发布流程、回滚、变更记录使用 rls；地图上线门禁和证据由本技能补充。
- 可观测性 / observability（obs）：监控、日志、告警和链路追踪使用 obs；瓦片失败率、上游状态、缓存命中和地图交互指标由本技能定义。
