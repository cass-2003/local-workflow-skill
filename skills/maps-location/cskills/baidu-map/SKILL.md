# 百度地图

首次自称：百度地图（baidu-map，兼容 slug: baidu-map）。

## 定位/适用范围

本技能用于在真实产品中设计、实现、调试或发布百度地图开放平台能力，提高一次接入成功率，覆盖：

- 百度地图 JavaScript API GL / WebGL 与传统 JS API：地图展示、控件、覆盖物、图层、定位、检索、路线、行政区、坐标转换和加载失败处理。
- 百度地图 WebService / LBS API：地点检索、地点详情、地理编码、逆地理编码、路线规划、坐标转换、行政区划、静态图、批量或服务端代理调用。
- Android SDK 与 iOS SDK：地图、定位、检索、路线、坐标转换、鹰眼/轨迹、生命周期、隐私合规、包名/Bundle ID/签名限制。
- 轨迹与鹰眼服务：终端、轨迹点、纠偏、停留点、围栏、里程、实时位置、历史轨迹和服务端管控。
- 小程序与移动 Web 边界：宿主小程序原生 map 组件、WebView/H5、浏览器定位、百度 WebService 后端配合、坐标体系衔接。
- 常见业务链路：地图选点、附近门店、地址录入、配送路线、到店导航、考勤定位、围栏打卡、轨迹回放、区域筛选、行政区联动。

不适用：

- 百度搜索、SEO、站内搜索、百度统计、百度营销、百度网盘、百度智能云通用能力；除非任务明确是百度地图开放平台。
- source map、hash map、roadmap、mind map、site map、map-filter-reduce 或“地图”隐喻，不触发本技能。
- 只读学习、概念解释、入门介绍、README/依赖文件中仅出现 baidu-map 字样且没有修改、调试、测试、部署动作，不触发本技能。
- 通用 GIS 几何、拓扑、投影、空间索引算法优先使用 map-gis-core 或 jsts。
- 仅涉及 Google、AMap/高德、腾讯、天地图、Mapbox、ArcGIS、Leaflet/OpenLayers 且无百度迁移或互操作需求时，使用对应相邻技能。
- 仅做通用 API、安全、测试、发布、前端性能或移动端框架问题时，与 api、wsec、tst、aud、pfe、appl、andr、kt、fltr、wcmp、rls、obs 配合。

触发确认：只有当任务要求设计、修改、排障、验证或发布百度地图开放平台能力，或明确涉及与百度地图的迁移、坐标互通、数据互通、ak/配额/权限问题时，才使用本技能。

## 铁律

1. 坐标体系先定性再写代码：百度地图展示和百度大多数返回坐标按 BD-09/百度坐标处理；国测局坐标为 GCJ-02；GPS 原始点为 WGS84；禁止裸 lng/lat 混用。
2. 中国大陆位置进入百度地图展示、检索周边、路线或轨迹前，必须明确是否需要 WGS84→GCJ-02→BD-09 或 GCJ-02→BD-09 转换；转换只能在明确输入类型后执行。
3. ak 按平台隔离：浏览器端、服务端 WebService、Android、iOS、小程序/后端代理、测试和生产不得共用同一个高权限 ak。
4. ak 必须配置最小化安全限制：浏览器 referer，服务端 IP 白名单/数字签名 sn，Android 包名与 SHA-1 发布签名指纹，iOS Bundle ID；禁止把服务端 ak 当作前端 ak 使用；开启签名校验的服务端 ak 调用必须按官方算法计算 sn 并随请求传入。
5. 不把 ak、签名密钥、企业账号、鹰眼服务 ID 的管理权限、用户位置明文批量导出到前端包、App 可反编译资源、日志、截图或错误上报。
6. 定位必须有用户授权、用途告知、拒绝授权兜底、最小化采集和保留周期；后台定位、轨迹、围栏和考勤必须有更高强度合规证据。
7. WebService 调用必须处理配额、QPS、状态码、空结果、模糊结果、行政区变化、坐标偏移、限流、超时和重试退避，不只判断 HTTP 200；只有响应体 status=0 才算业务成功，非 0 status 必须按 message/错误码分类处理。
8. WebGL/JS API 加载必须有超时、错误态和降级路径；容器尺寸、版本、插件、事件解绑、实例销毁和海量覆盖物性能必须显式处理。
9. 禁止规避百度地图服务条款：不绕过 ak、不抓取或批量囤积 POI、不采集瓦片、不隐藏版权归属、不伪造来源、不拆分账号逃避配额。
10. 发布前必须提供目标环境证据：真实浏览器/真机、权限拒绝、弱网、ak 限制、配额预估、监控告警、回滚方案和隐私合规检查。

## 快速总则

- 先问平台：WebGL/JS API、WebService、Android、iOS、H5/WebView、小程序宿主、服务端批处理、鹰眼轨迹。
- 先问对象：地图展示、定位、POI、地理编码、逆地理编码、路线、行政区、轨迹、围栏、导航跳转、静态图。
- 先问坐标：输入、存储、展示、传给百度、从百度返回、传给其他地图的坐标系分别是什么。
- 先问 ak：使用哪个应用类型、限制是否打开、是否区分环境、是否可轮换、泄露后如何止血。
- 先问合规：用户是否授权、是否采集精确位置、是否后台定位、是否上传轨迹、是否跨境或给第三方共享。
- 先问容量：日调用量、峰值 QPS、地图首屏压力、marker 数量、轨迹点数量、缓存策略和降级策略。
- 先问证据：官方文档核对、成熟实践核对、测试矩阵、发布门禁、监控看板和事故回滚。

## 强制流程

1. 需求与边界确认
   - 写清业务链路、端类型、用户动作、地图能力、是否实时定位、是否落库、是否需要长期轨迹。
   - 明确是否只展示百度地图，还是与高德/腾讯/Google/自有 GIS 数据互通；互通时必须列坐标转换边界。
   - 区分“地图 SDK 展示能力”和“WebService 数据能力”，避免在前端直接承担高频、批量或敏感查询。

2. 官方能力核对
   - 对照百度地图开放平台官方文档确认 API/SDK 名称、版本、ak 类型、参数、返回码、限制、配额、计费、废弃状态。
   - WebGL 优先按 JavaScript API GL 当前文档实现（全局命名空间为 BMapGL，不是 BMap；BMapGL 与传统 BMap 的 Convertor、覆盖物、控件 API 并非全部一一对应，迁移时必须逐项核对）；历史 JS API 2.0/3.0 项目改动时先确认是否可迁移到 GL 或保持兼容。
   - JS API 加载使用官方 api?v=xxx&ak=xxx&callback=init 异步加载方式（GL 用 //api.map.baidu.com/api?v=1.0&type=webgl&ak=xxx&callback=init），callback 必须挂全局；禁止 document.write 同步插入。
   - 在控制台确认对应服务已启用；浏览器端 JS API 只使用浏览器 ak，服务端 ak 不直接放进页面。
   - 鹰眼（Yingyan/轨迹服务 v3）按 service_id + entity_name 模型管理，service_id 在控制台创建，entity 表示一个终端（人/车/设备），轨迹点写入需挂在 entity 下；管理类接口（创建 service、删除 entity、批量纠偏等）只能服务端调用并配 sn 签名。

3. 坐标与数据设计
   - 数据库字段必须带坐标系后缀，例如 bd09_lng、bd09_lat、gcj02_lng、gcj02_lat、wgs84_lng、wgs84_lat。
   - 百度地图展示、百度 POI 结果、百度逆地理编码返回坐标不要默认为 WGS84；需要给其他地图或算法时先转换并标注。
   - GPS 设备、浏览器原始定位、移动端系统定位、第三方地图、历史库数据进入百度前，要验证来源坐标系和转换链路。
   - 批量坐标转换要做去重、批大小控制、失败重试、结果顺序校验和转换前后偏移抽样检查。

4. ak 与安全设计
   - 每个平台独立 ak：WebGL/JS 用浏览器端 ak；WebService 用服务端 ak；Android/iOS 用移动端 ak；鹰眼管理接口只在服务端使用。
   - Web 端配置 referer 白名单并覆盖生产域名、测试域名和本地调试域名；不要用星号长期放开；注意 file://、about:blank、空 referer 和子域名匹配规则。
   - 服务端配置 IP 白名单、网关鉴权、限流、缓存、审计和密钥轮换；前端只调用自家后端；启用 sn 数字签名时按官方算法：参数按 key 升序排序、按 GET 形式拼成 path?k=v&k=v、URL encode（注意空格转 %20 不是 +）、拼接 sk 后做 MD5 小写得到 sn，随请求 sn= 参数传入；sn 启用后请求里不能再带 IP 白名单依赖。
   - Android 绑定包名和 release 签名 SHA-1 指纹，区分 debug/release，渠道包和加固后包要重新核对 SHA-1；iOS 绑定 Bundle ID，区分开发、TestFlight、生产。
   - 发现 ak 泄露时立即收紧限制、轮换 ak、查调用量、封禁异常来源并补充日志脱敏。

5. 权限与合规设计
   - Web/H5 定位要求 HTTPS、浏览器授权、拒绝授权兜底、手动输入地址和低精度降级。
   - Android/iOS 在隐私政策、权限弹窗、SDK 初始化、后台定位、精确定位、轨迹上传、第三方共享清单中保持一致。
   - 小程序按宿主平台配置位置权限用途；宿主 map 组件通常不等同于百度地图 SDK，和百度 WebService 结合时要处理 GCJ-02/BD-09 边界。
   - 考勤、风控、儿童、医疗、车联网、物流轨迹等高敏场景必须最小化采集、加密传输、访问审计、保留期限和删除机制。

6. 性能与稳定性设计
   - WebGL/JS API 异步加载、固定版本、按需加载库/插件、地图容器先有尺寸、首屏不加载非必要图层。
   - marker/覆盖物数量大时使用聚合、抽稀、服务端裁剪、视口加载、瓦片化或 Canvas/WebGL 图层，避免一次性渲染全量点。
   - 路线、POI、逆地理编码和行政区查询做防抖、缓存、并发限制、超时、退避、错误分级和空结果提示。
   - 移动端控制定位频率、轨迹采样、电量、后台运行、生命周期释放和离线/弱网重试。

7. 测试与发布
   - 覆盖成功、失败、ak 未授权、referer/IP/包名/Bundle 限制错误、配额不足、弱网、超时、权限拒绝、定位关闭。
   - 覆盖坐标偏移：WGS84、GCJ-02、BD-09 样本点，城市内外、边界点、历史库数据和第三方地图互通。
   - 覆盖端差异：Chrome/Safari/Edge、移动浏览器、WebView、Android 主流版本和厂商权限、iOS 精确/模糊定位。
   - 发布材料必须包含 ak 限制截图或配置说明、调用量预估、监控指标、隐私合规清单、回滚开关和版本号。

## 场景执行卡

### WebGL/JS API 地图展示

- 使用浏览器端 ak，配置 referer 白名单；生产、测试、本地调试分环境管理。
- 固定 API 版本和依赖库，记录是否使用 GL；避免多个组件重复插入不同版本脚本。
- 地图容器必须有宽高，初始化在 DOM 可见后执行；隐藏 tab、弹窗、SSR hydration 后要触发重绘或延迟初始化。
- 加载失败要显示可操作错误：网络失败、ak 未授权、referer 不匹配、控制台服务未启用、脚本被 CSP/广告拦截、超时。
- 高频检索、路线、地理编码和坐标转换要尊重官方频率限制，前端做防抖，服务端做限流与退避。
- 卸载时移除覆盖物、事件监听、定时器和地图引用，避免 SPA 内存泄漏。
- 大量点使用聚合、视口裁剪、按需加载和抽稀；不要把全量 POI/轨迹点一次塞进地图。

### WebService / LBS 服务端调用

- 服务端调用必须以官方当前 WebService/LBS 文档为准，核对服务开通、ak 类型、参数、坐标系、返回 status/message/result 字段层级和配额计费；HTTP 200 且响应体 status=0 才算业务成功。
- 统一封装错误分级：ak 无效、权限/白名单/sn 签名不匹配、服务未开通、参数错误、无结果、超配额、QPS 限流、上游 5xx、网络超时；常见 status 速查：0 成功、1 服务器内部错误、2 请求参数非法、3 权限校验失败、4 配额校验失败、5 ak 不存在或非法、101 服务禁用、102 不通过白名单或安全码不对、200 apikey 不存在、201 apikey 被用户自己禁用、202 apikey 被管理员禁用、211 Mcode 参数不存在或非法、240 service 不存在、301 永久配额超限（天配额）、302 永久配额超限（次配额）、401 当前并发量已经超过 ak 配额、402 当日配额已使用完。
- 坐标转换按官方当前 geoconv 接口分批、限流和校验结果；geocoding/v3 与 place/v2/search 等服务配额独立，超限返回按对应 status/message 分级处理。
- 对 403、429、5xx、超时和百度业务 status 非成功分别给出用户提示、重试退避、熔断、降级和告警；禁止只凭 HTTP 200 判成功。
- 后端代理要鉴权、限流、缓存、脱敏日志和请求审计；前端不得直连高权限 WebService ak。
- 批量地理编码、逆地理编码、路线或坐标转换要控制批大小、并发、去重、缓存期限和失败补偿，不做无界队列。

### 地点检索与 POI

- 明确检索类型：关键词、周边、矩形区域、城市内、详情；选择 JS API 还是服务端 WebService。
- 用户输入要限长、去抖、过滤明显无效查询；服务端代理要鉴权和限流。
- POI 结果要处理空结果、多义词、城市歧义、坐标系、营业状态变化和详情字段缺失。
- 禁止批量爬取、囤积、转售、离线复刻百度 POI 数据；缓存只用于合理业务性能和合规期限。

### 地理编码与逆地理编码

- 地址入库前区分用户填写地址、结构化地址、百度解析结果和人工校正结果。
- 地理编码要传城市/行政区上下文，处理置信度、模糊地址、多个候选、无结果和行政区变更。
- 逆地理编码要核对官方返回字段版本，处理道路、门址、小区、商圈、行政区层级缺失；不要把返回文本当作永久标准地址。
- 坐标传入前确认是否为 BD-09；其他坐标系必须使用官方转换或可信链路转换。

### 路线规划与导航跳转

- 明确出行方式：驾车、步行、骑行、公交、货车或业务自定义；核对对应 API 支持和限制。
- 起终点、途经点和避让策略必须带坐标系说明；跨城市、跨省和海外场景要单独验证。
- 路线结果要处理无路、限行、绕行、时间变化、费用字段缺失、多方案排序和实时路况不可用。
- App 导航跳转要检测百度地图客户端安装状态，提供 Web/H5 或其他合规兜底，不强制用户授权无关权限。

### 坐标转换

- 每个接口边界都写明输入和输出坐标系；裸 lng/lat 参数必须改名或补注释。
- WGS84 GPS 点用于百度展示前需转换；GCJ-02 来源点用于百度展示前需转换为 BD-09；百度返回点给 GCJ-02 平台前需反向转换或使用官方/可信能力。
- 官方转换通道优先核对当前 WebService geoconv 文档：新接口 `geoconv/v2` 使用 `model` 表示转换方向（1=amap/tencent to bd09ll，2=gps to bd09ll，3=bd09ll to bd09mc，4=bd09mc to bd09ll，5=bd09ll to amap/tencent，6=bd09mc to amap/tencent）；旧 `geoconv/v1` 项目再核对 `from/to` 取值（常见 from: 1=WGS84，3=GCJ-02，5=BD-09LL；to: 5=BD-09LL，6=BD-09MC），禁止把 BD09LL 和 BD09MC 写反；JS API 用 `new BMapGL.Convertor().translate(points, fromType, toType, callback)`，并按当前文档确认常量、批量上限和回调结果。
- 避免多次重复转换导致漂移；存储原始坐标和转换后坐标时要记录来源、转换时间、算法/接口版本。
- 批量转换后抽样对照已知地标，检查经纬度顺序（百度全部接口为 lng,lat 顺序）、单位、火星坐标/百度坐标混淆和境外点处理（境外点 GCJ-02 偏移规则不适用，必须按 WGS84 直传或走对应海外服务）。

### Android SDK

- 使用 Android 专用 ak，绑定包名和 release 签名 SHA-1 指纹（百度目前主要采用 SHA-1，不是 SHA-256；release SHA-1 通过 `keytool -list -v -keystore release.jks -alias xxx` 获取，注意冒号分隔并转大写）；debug、release、渠道包、加固包分别验证授权，加固或重签后必须重新核对 SHA-1。
- 隐私合规：按当前 Android 地图 SDK 文档，必须在 Application.onCreate 中先调用 `SDKInitializer.setAgreePrivacy(context, true)`，再调用 `SDKInitializer.initialize(context)`；定位 SDK 还要在 LocationClient 实例化前调用 `LocationClient.setAgreePrivacy(true)`。用户未同意隐私政策前严禁初始化，否则地图黑屏、定位无回调、SDK 主动崩溃或上报合规违规。
- 检查 Manifest 权限（ACCESS_FINE_LOCATION/ACCESS_COARSE_LOCATION/ACCESS_BACKGROUND_LOCATION，Android 12+ 还需 ACCESS_COARSE 与 NEARBY_WIFI_DEVICES 视场景）、运行时权限、隐私政策、后台定位声明。
- 地图、定位、检索、路线、鹰眼对象要跟随 Activity/Fragment 生命周期 onResume/onPause/onDestroy 启动、暂停、释放；MapView 必须调用 onSaveInstanceState 与 onDestroy 释放 native 引用。
- 定位频率、轨迹采样和覆盖物刷新要控制电量与内存；弱网和系统定位关闭要有兜底。
- 发布前用目标 Android 版本、主流厂商 ROM、不同定位权限状态和 release 包验证。

### iOS SDK

- 使用 iOS 专用 ak，绑定 Bundle ID；开发、TestFlight、App Store 包分别验证。
- 隐私合规：在 BMKMapManager start 之前调用 `+[BMKMapManager setAgreePrivacy:YES]`（定位用 `+[BMKLocationAuth setAgreePrivacy:YES]`），否则地图不显示、定位不回调；用户未同意前不得调用。
- Info.plist 必须包含 `NSLocationWhenInUseUsageDescription`、`NSLocationAlwaysAndWhenInUseUsageDescription`（涉及后台定位）等用途文案；iOS 14+ 还需处理 `NSLocationTemporaryUsageDescriptionDictionary` 精确定位临时授权；隐私清单 PrivacyInfo.xcprivacy 必须声明数据采集类目（位置、设备 ID 等）。
- 处理精确/模糊定位（iOS 14+ accuracyAuthorization）、仅一次授权、拒绝授权、系统定位关闭、后台定位和前后台切换。
- 释放 BMKMapView、BMKLocationManager、BMKSearch delegate、轨迹监听，避免循环引用和内存泄漏；BMKMapView viewWillAppear 调用 viewWillAppear、viewWillDisappear 调用 viewWillDisappear 否则地图渲染异常。
- 发布前覆盖真机、弱网、App Store 隐私问卷和崩溃/卡顿监控。

### 小程序与移动 Web

- 小程序原生 map 组件由宿主平台提供，不等同于百度 JS API；微信小程序无法加载 BMapGL 脚本，百度地图能力主要通过 WebService 在小程序后端代理调用，前端只能用宿主 map 组件做展示。
- 小程序位置通常来自宿主定位能力，微信 wx.getLocation 默认返回 GCJ-02（type:'gcj02'），调用百度 WebService 前必须转换为 bd09ll（coord_type=bd09ll 或先调 geoconv/v1）；百度返回的 BD-09 点用于微信 map 组件展示前必须反向转回 GCJ-02。
- 小程序中高风险 WebService 调用走自家后端，避免暴露服务端 ak、绕过域名限制或失去审计；request 合法域名必须在小程序后台白名单加入 api.map.baidu.com 或自家代理域名。
- 移动 Web/H5 依赖 HTTPS、浏览器定位授权、WebView 宿主权限和系统定位开关；拒绝授权时提供手动选点。

### 轨迹与鹰眼

- 先确认是否真的需要长期轨迹；能用一次定位或短期路线就不要启用持续采集。
- 鹰眼数据模型：service（控制台创建，得到 service_id）→ entity（终端，按 entity_name 唯一）→ track point（轨迹点，按 loc_time 上报）→ 围栏 fence（圆形/多行政区/多边形/线形）；entity_name 命名规则只能英文数字下划线连字符，长度 1-128；track 上报字段必须包含 loc_time（unix 秒）、coord_type_input（坐标系：bd09ll/gcj02/wgs84）。
- 服务、终端、轨迹点、围栏和纠偏参数要在服务端管理，客户端只上传最小必要点；管理类接口（addentity/deleteentity/addfence/searchhistory）必须服务端调用并配 sn 签名，禁止暴露在 App/Web。
- 轨迹点包含个人敏感信息，必须加密传输、鉴权访问、分级权限、审计、保留期限（鹰眼默认存储 3 个月，超期需购买扩容包）和删除机制。
- 处理 GPS 漂移、断点、补传、乱序、重复点、停留点、里程误差、低电量和后台限制；开启服务端纠偏 `process_option=need_denoise=1,need_vacuate=1,need_mapmatch=1` 可显著降低噪点和绑路误差。
- 禁止把鹰眼当作无限制位置仓库或导出工具；调用量、存储、entity 数量和查询都要纳入配额与成本监控。

### 行政区与区域业务

- 行政区划名称、adcode、边界和层级会变化；不要把一次返回当作永久真相。
- 区域筛选要明确按行政区、商圈、圆形半径、多边形还是配送围栏；不同语义不能混用。
- 行政边界展示大量点线面时要压缩、抽稀、缓存和按视口加载。
- 发布前用直辖市、省市县、开发区、跨区地址和边界点样本验证。

- 官方门禁：引用的 API/SDK 名称、版本、参数、返回码、配额、计费和限制已对照百度地图开放平台当前文档。

- 测试门禁：至少覆盖目标浏览器/真机、开发/测试/生产 ak、debug/release 包、坐标样本、失败路径和回滚。

## 反例库

- 反例：前端直接使用服务端 WebService ak 调地点检索。修正：前端用浏览器 ak 或调用自家后端，服务端 ak 配 IP 限制和限流审计。
- 反例：数据库只有 lng、lat，半年后不知道是 BD-09 还是 GCJ-02。修正：字段名和元数据标注坐标系，迁移时抽样校验。
- 反例：GPS 设备点直接画到百度地图，点位整体偏移。修正：确认 WGS84 输入，按官方/可信转换链路转为 BD-09 后展示。
- 反例：JS API 加载失败时页面空白。修正：加超时、错误分类、降级 UI、重试按钮和监控埋点。
- 反例：一次渲染十万 marker 导致浏览器卡死。修正：聚合、抽稀、视口加载、服务端裁剪或专用图层。
- 反例：小程序宿主 map 的 GCJ-02 点直接传百度逆地理编码。修正：确认宿主返回坐标系，转换为百度接口要求后再调用。
- 反例：轨迹服务持续采集但隐私政策只写“用于地图展示”。修正：明确轨迹、后台、保留期限、删除方式和用户控制。
- 反例：发布只测本地 localhost referer。修正：测试生产域名、CDN、WebView、灰度环境和 referer 限制。
- 反例：把百度 POI 全量缓存成自有库。修正：只做合理期限业务缓存，遵守官方授权与条款。
- 反例：Android debug 包可用，release 包报未授权。修正：分别绑定并验证 debug/release 签名和包名。

## 相邻技能边界

- 地图 GIS 核心开发 / map-gis-core（map-gis-core）：通用坐标、投影、空间索引、几何关系、瓦片策略和跨地图架构；百度平台细节仍由本技能处理。
- JavaScript / TypeScript 开发 / javascript-typescript-development（jsts）：复杂多边形、缓冲区、相交、包含、拓扑修复等几何计算；百度地图只负责展示或服务调用。
- amap-gaode 、 Google Maps Platform / google-maps-platform（google-maps-platform） 、 mapbox-maplibre 、 esri-arcgis 、 leaflet-openlayers：相邻地图平台；只有迁移、互操作或坐标转换边界涉及百度时联动。
- Vue 开发 / vue-development（vue） 、 性能工程 / perf-engineering（pfe）：前端组件化、性能、构建和首屏优化；百度 JS API 加载、覆盖物和 ak 规则由本技能给门禁。
- API 工程 / api-engineering（api）：服务端代理、接口契约、限流、缓存、错误码和幂等；百度 WebService 语义由本技能定义。
- Web 安全 / web-security（wsec） 、 代码审计 / code-audit（aud）：ak 泄露、位置隐私、日志脱敏、权限越权和合规审计；百度地图风险点由本技能输入。
- 测试验证 / test-engineering（tst）：坐标样本、端到端、弱网、权限拒绝、配额失败和真机矩阵测试。
- Apple 全链路开发与发布 / apple-development（appl） 、 Android 开发 / android-development（andr） 、 Kotlin 开发 / kotlin-development（kt） 、 Flutter 开发 / flutter-development（fltr）：iOS、Android、Kotlin、Flutter 工程实现；百度 SDK 合规、权限和坐标边界需同步。
- 微信小程序 / wechat-miniprogram（wcmp）：微信/小程序宿主位置和 map 组件能力；百度 WebService/坐标互通由本技能约束。
- 发布部署 / release-engineering（rls） 、 可观测性 / observability（obs）：发布、灰度、监控、告警、回滚和调用量看板；百度 ak、配额、错误码和隐私证据纳入发布门禁。
