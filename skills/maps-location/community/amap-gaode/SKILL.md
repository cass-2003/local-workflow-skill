# 高德地图

首次自称：高德地图（amap-gaode，兼容 slug: amap-gaode）。

## 定位/适用范围

本技能用于在产品、前端、后端、移动端、小程序和测试发布流程中接入或审查高德地图能力，覆盖：

- 高德 Web JS API，包括地图展示、图层、覆盖物、插件、定位、逆地理编码、路径规划、POI 搜索、行政区划。
- @amap/amap-jsapi-loader，包括异步加载、版本固定、插件按需加载、失败降级和性能控制。
- 高德 WebService API，包括地理编码、逆地理编码、路径规划、行政区域、搜索、天气、输入提示等 HTTP 服务。
- Android SDK 与 iOS SDK，包括地图、定位、搜索、导航或路线、权限、隐私合规声明和包名/Bundle ID/key 限制。
- 小程序地图与位置能力，包括微信/支付宝等宿主限制、高德服务端接口配合、定位授权和坐标体系处理。
- 常见业务链路：定位、逆地理编码、路径规划、POI、行政区、坐标转换、地图选点、附近搜索、轨迹展示、地址补全。

不适用：

- “map” 指 source map、hash map、roadmap、mind map、site map、map/filter/reduce、数据映射或配置映射，而非地图位置服务。
- 只是在学习高德文档、做只读问答、onboarding、README 梳理、依赖清单识别，且没有修改、调试、测试或发布动作。
- 仅出现 amap/高德依赖、key 名称或 README 描述，但任务目标不是修复、接入、迁移、测试或发布高德能力。
- 用户是在咨询高德导航 App/车机等消费端使用方法，不涉及开放平台工程接入。
- 只要求接入百度、腾讯、Google、Mapbox、OSM 等相邻地图供应商；除非明确要求迁移到高德、从高德迁出或做跨供应商互操作。
- 不涉及高德的通用 GIS 算法实现，应转交 map-gis-core 或 jsts。
- 仅做 UI 样式组件而无地图平台能力，应优先使用 vue、pfe、appl 等对应技能。
- 仅做通用 API 网关、安全审计、测试发布，应与 api、wsec、tst、aud、rls、obs 配合。

## 铁律

1. 坐标体系必须先定性再实现：高德中国境内业务默认 GCJ-02；GPS 原始点通常是 WGS84；百度坐标是 BD-09；禁止混用坐标而不转换。
2. 禁止把服务端 WebService key、签名私钥、安全密钥、移动端安全码、企业账号信息写入前端、小程序包、App 可反编译资源或日志。
3. key 必须按平台隔离：Web、WebService、Android、iOS、小程序、服务端代理分别使用独立 key，并配置 referer、IP、包名、SHA1、Bundle ID 等限制。
4. WebService 启用数字签名时，签名必须在可信服务端生成；前端只能调用自家后端，不直接暴露签名私钥。
5. 定位能力必须经过用户授权、隐私告知、拒绝授权兜底和最小化采集；移动端发布前必须完成高德 SDK 隐私合规声明与初始化时序检查。
6. 路径规划、POI、行政区、逆地理编码等结果必须处理配额、限流、失败、空结果、模糊结果和行政边界变化。
7. 禁止把官方示例代码直接复制进生产；必须按业务平台、权限、安全、性能、可观测和测试证据重构。
8. 发布前必须有真实设备或目标浏览器证据，不得只以本地 mock 或单一浏览器截图作为通过依据。
9. 不得通过爬取 POI、瓦片、行政区边界或批量规避配额来替代授权接口；不得隐藏高德 attribution、版权标识、许可说明或服务条款约束。
10. 任何“修一下地图问题”必须先归因到 key/权限/坐标/网络/配额/SDK 版本/宿主限制之一；证据不足时先止步提问，不盲改。

## 快速总则

- 先确认平台：Web JS API、Loader、WebService、Android、iOS、小程序、混合 App 或服务端代理。
- 先确认数据流：用户位置从哪里来、坐标是什么、是否要转坐标、是否要落库、是否要回传第三方。
- 先确认 key 策略：每个平台独立 key，最小权限，限制来源，区分开发/测试/生产，纳入轮换和泄露响应。
- 先确认能力边界：展示地图用 JS API；批量或敏感查询优先服务端 WebService；高频移动定位优先原生 SDK。
- 先确认合规：隐私政策、权限文案、SDK 初始化、位置采集频率、后台定位、儿童/敏感行业要求。
- 先确认成本：日调用量、QPS、配额、计费项、缓存策略、降级策略和告警阈值。
- 先确认发布证据：官方文档核对、成熟工程实践核对、真机/浏览器测试、失败路径测试、监控埋点。
- 先确认停止条件：若缺少平台、key 类型、坐标来源、权限状态、目标域名/包名/Bundle ID 或发布环境，不进入实现阶段。
- 先确认证据闭环：每个改动要能给出配置证据、运行证据、异常证据和回滚证据，避免二次返工。

## 强制流程

1. 需求澄清
   - 标明业务场景：地图展示、选点、附近搜索、路线、轨迹、配送、行政区筛选、地址录入、考勤、风控等。
   - 标明平台与端：浏览器、H5、WebView、Android、iOS、微信小程序、支付宝小程序、服务端。
   - 标明国家/地区：中国大陆、港澳台、海外；跨境场景要确认坐标、数据出境和地图服务覆盖。

2. 官方能力核对
   - 对照高德开放平台官方文档确认 API 名称、参数、key 类型、限制项、配额、计费、插件、SDK 版本、隐私合规要求和废弃状态。
   - 分清 Web JS API、@amap/amap-jsapi-loader、WebService、Android/iOS SDK、小程序宿主能力的边界；不能用一个平台的示例推断另一个平台可用。
   - 官方示例只用于理解能力，不作为生产结构；成熟实践优先强调隔离 key、服务端代理、限流、缓存、可观测和合规证据。

3. 坐标与数据设计
   - 明确输入坐标体系、输出坐标体系、存储坐标体系和展示坐标体系。
   - WGS84 GPS 点进入高德展示或高德服务前要转换为 GCJ-02；高德返回的中国境内坐标按 GCJ-02 处理；BD-09 进入高德前先转 GCJ-02。
   - 经纬度顺序必须显式标注：高德多数接口 location 参数按 lng,lat；UI、数据库、第三方设备常写 lat,lng，联调时先查顺序再查算法。
   - 数据库字段命名要体现坐标系，例如 gcj02_lng、gcj02_lat、wgs84_lng、wgs84_lat、bd09_lng、bd09_lat，避免 lng/lat 裸字段歧义。
   - 跨境或海外点不得默认套中国偏移；先确认高德服务覆盖、海外权限和是否需要保留 WGS84 原始值。

4. key 与安全设计
   - Web JS API 配 referer 白名单；移动端配包名、SHA1 或 Bundle ID；WebService 配 IP 白名单和签名。
   - 本地开发、测试、生产使用不同 key；CI、日志、错误上报和客户端包中不得出现服务端 key 或签名私钥。
   - 需要跨端共享能力时，通过自家后端网关做鉴权、限流、审计、缓存和签名，不让客户端直连高风险接口。
   - 先在高德控制台确认 key 的服务平台、启用服务、限制条件和计费套餐，再写代码；10009、10012、403 大多是 key 类型或服务权限不匹配。

5. 权限与合规设计
   - Web 定位需 HTTPS、浏览器权限提示、拒绝授权兜底和手动输入方案。
   - Android/iOS 需在隐私政策、权限声明、SDK 合规初始化、后台定位说明、权限用途弹窗中保持一致。
   - 小程序需在宿主平台配置位置权限用途，处理用户拒绝、系统关闭定位、宿主 API 限制和审核文案。

6. 性能与稳定性设计
   - Web JS API 使用 loader 异步加载、版本固定、插件按需加载、地图实例销毁、事件解绑和大数据量聚合/抽稀。
   - WebService 使用服务端缓存、去抖、批量控制、重试退避、错误码分类、QPS 保护和熔断降级。
   - 移动端控制定位频率、电量消耗、后台运行、轨迹采样、地图覆盖物数量和生命周期释放。

7. 测试与发布
   - 测试成功、失败、空结果、限流、key 错误、referer/IP/包名/Bundle ID/SHA 限制错误、坐标偏移、权限拒绝、离线/弱网、跨浏览器和真机差异。
   - WebService 必测 status/info/infocode、403、429、5xx、超时、签名错误、配额不足、空数组和部分字段缺失。
   - 移动端必测用户同意前后 SDK 初始化、权限拒绝、精确/模糊定位、系统关闭定位、后台限制、低端机内存和弱网耗电。
   - 发布前提供 key 限制截图或配置说明、隐私合规核对、调用量预估、监控告警、回滚方案、版本证据和至少一组坐标黄金样本。

8. 一次通过止步条件
   - 无法确认平台、坐标系、key 类型、限制配置、隐私授权或目标发布环境时停止实现，只输出缺口问题。
   - 发现服务端密钥在客户端、未授权采集位置、绕过签名/配额/版权标识、批量抓取 POI/瓦片时停止并给阻断结论。
   - 证据不能覆盖正常路径和关键失败路径时，不宣称可发布；只能标注为待验证。

## 场景执行卡

### Web JS API 地图展示

- 使用独立 Web key，配置 referer 白名单，开发域名和生产域名分开。
- 白名单必须覆盖真实访问来源：生产域名、预发域名、WebView 外壳域名、CDN 域名和本地调试域名分环境维护；不要用通配符兜底生产问题。
- 必须明确 JS API 主版本：1.4.x 与 2.0.x 的命名空间、插件名、构造参数和事件名不完全兼容，不能用 1.4 文档实现 2.0，反之亦然。
- 固定 API 到具体小版本，避免隐式升级导致样式、插件或类型行为变化；插件清单按场景显式声明。
- JS API 2.0 启用安全密钥 securityJsCode 时，生产优先走代理服务器转发；明文方式只允许临时开发和受控测试，不得进入公开前端包。
- 代理转发要限制目标 host、路径前缀、key 映射和调用频率；不能做成可访问任意 URL 的地图代理。
- 地图容器必须有明确尺寸；组件卸载时销毁地图实例、覆盖物、事件监听和定时器。
- 覆盖物数量大时使用聚合、虚拟化、瓦片或服务端裁剪，禁止一次渲染海量 marker。
- 需要定位时先判断 HTTPS、浏览器权限和用户拒绝兜底。

### @amap/amap-jsapi-loader

- Loader 负责按需异步加载，不负责解决 key 泄露；仍需 referer 限制和环境隔离。
- loader 配置要集中管理，避免多个组件重复加载不同 key、不同版本或同时混用 1.4 与 2.0。
- version 字段必须固定到具体小版本字符串，不允许写 latest 或省略；plugins 列表必须与 JS API 主版本匹配。
- 插件按场景加载，避免首屏加载定位、搜索、路线、行政区等全部插件。
- 加载失败要有超时、错误提示、重试或降级页面，不让业务流程卡死。

### WebService API

- 服务端持有 WebService key 和签名私钥，客户端调用自家 API。
- 对用户输入做校验和归一化，限制关键字长度、city/adcode、坐标范围、分页大小、出行方式和 bbox，避免把任意字符串无节制转发给高德接口。
- 服务端代理必须有业务鉴权、租户/用户限流、缓存键规范、超时预算、重试退避和审计日志；日志只保留必要字段并脱敏位置详情。
- 逆地理编码、POI、行政区、路径规划要缓存可缓存结果；用户实时位置要按隐私和时效控制缓存。
- 处理 info、status、infocode、空数组、部分成功、配额不足、服务不可用等结果，不只判断 HTTP 200。
- 常见 infocode 必须分类处理而非统一报错：10001 key 不正确或过期；10003 日访问量超限；10004 单位时间访问过频；10005 IP 白名单错误；10006 绑定域名无效；10007 数字签名失败；10008 SHA1/package 安全码失败；10009 key 与平台不符；10012 服务权限不足；10014/10019/10020/10021 QPS 超限；10044 账号维度日调用量超限；20000/20001 参数错误；20011 海外权限不足；20800/20801/20802/20803 路径规划点或路线不可用。
- 403 优先查 key 类型、签名、IP 白名单、referer 误用和服务开通状态；429 优先查 QPS、日配额、缓存击穿和重试风暴；5xx/超时要退避、熔断和降级。
- 对高频接口配置限流、熔断、告警和调用量看板。

### Android SDK

- 使用 Android 专用 key，绑定包名和 SHA1，区分 debug/release 签名。
- SHA1 要分别记录 debug、release、加固后、渠道包和 CI 签名；包名、applicationId、flavor、签名证书变更后先更新控制台再发版。
- 检查 AndroidManifest 权限、隐私政策、SDK 合规初始化顺序和用户同意时机。
- 隐私合规初始化时序固定：先调用 MapsInitializer、AMapLocationClient、ServiceSettings 或对应 SDK 模块的 updatePrivacyShow/updatePrivacyAgree，再设置 ApiKey，再创建任何地图、定位、搜索实例；顺序颠倒会被 SDK 拒绝初始化或在审核中判违规。
- Android 12+ 精确/模糊定位、后台定位、前台服务通知、权限二次申请和系统定位关闭都要覆盖；用户拒绝后不得继续启动定位链路。
- 定位频率按业务最小化，后台定位必须有清晰用途、前台服务或系统要求支持。
- Activity/Fragment 生命周期中释放地图、定位客户端、搜索对象和监听器。
- 发布证据需覆盖目标 Android 版本、主流厂商定位权限差异和弱网场景。

### iOS SDK

- 使用 iOS 专用 key，绑定 Bundle ID，区分开发、TestFlight、生产包。
- Info.plist 权限描述必须和产品文案、隐私政策一致，至少核对使用期间定位、始终定位、临时精确定位用途和后台模式是否真的需要。
- 隐私合规初始化时序固定：先调用 AMapServices updatePrivacyShow/updatePrivacyAgree，再 setApiKey，再使用地图、定位、搜索类；以及 AMapLocationManager/AMapSearchAPI 等子模块各自的隐私接口也要在使用前调用。
- 初始化 SDK 与用户同意时机符合高德和 App Store 隐私要求。
- 前后台定位、精确/模糊定位、一次性授权、拒绝授权和系统关闭定位都要有兜底。
- 释放地图、定位、搜索和 delegate，避免内存泄漏。

### 小程序

- 使用宿主平台允许的位置能力和地图组件能力，不能假设 Web JS API 能完整运行。
- 宿主平台 key、合法域名、request 域名、地图插件或原生 map 组件能力要分别核对；微信、支付宝、抖音等小程序不能互相套用配置。
- 小程序端不保存 WebService 私钥；需要高德 WebService 时走服务端代理。
- 配置位置权限用途说明，处理用户拒绝、再次授权入口和审核要求。
- 注意宿主地图坐标体系和高德返回坐标的转换关系，避免选点偏移；wx.getLocation/my.getLocation 必须显式声明 type=gcj02 与高德服务对齐，不要默认 wgs84 后直接传给高德。
- 对地址补全、POI、逆地理编码做输入防抖和服务端限流。

### 定位

- 明确来源：浏览器定位、高德定位插件、Android/iOS 原生定位、小程序定位、外部 GPS 设备。
- 明确坐标：GPS 常见 WGS84，高德展示和中国境内服务通常使用 GCJ-02。
- 必须处理无权限、无信号、超时、低精度、模拟定位、后台限制和用户手动输入。
- 定位失败不能无限重试；按错误类型进入手动选点、城市级定位、最近一次授权缓存或业务降级，并记录失败率。
- 只采集业务必要精度和频率；敏感位置数据落库要有脱敏、保留期限和访问审计。

### 逆地理编码

- 输入坐标必须标注坐标系并按高德要求转换。
- 输出地址要区分格式化地址、省市区、乡镇街道、道路、门牌、AOI/POI 附近信息。
- 业务决策不得只依赖格式化地址字符串，应保存结构化行政区编码和坐标。
- 批量逆地理必须走队列、缓存、限速和失败重放；用户实时链路只做必要查询，避免每秒跟随定位调用。
- 对边界附近、农村地址、地下空间、园区楼栋等模糊结果提供人工修正。

### 路径规划

- 明确出行方式：驾车、步行、骑行、公交、货车或多策略。
- 参数要体现业务约束：避让、途经点、车牌限行、时间、城市、货车属性等。
- 结果要展示距离、耗时、收费、路线摘要和不确定性，避免承诺绝对到达时间。
- 起点、终点、途经点先做坐标系和服务范围校验；20800/20801/20802/20803 不应重试风暴，而应提示选点、道路吸附或人工调整。
- 路线点大量渲染时要抽稀，移动端注意内存和绘制性能。

### POI 与输入提示

- 搜索词要做去抖、最小长度、城市限制、类型限制和结果分页。
- 不同城市重名 POI 必须结合 citycode、adcode、坐标距离或用户选择确认。
- POI ID、名称、地址、坐标、类型和行政区要分开存储；名称变更和停业要可刷新。
- 输入提示不能直接当最终地址；必须让用户确认 POI 或补全门牌，服务端保存确认来源、adcode 和坐标系。
- 用户输入地址场景要允许手动补充门牌、楼层、房间号和备注。

### 行政区

- 行政区数据会变化，不能把一次性导出的行政区作为永久真理。
- 使用 adcode、citycode、层级、边界、多边形和中心点时要说明用途差异。
- 行政区边界用于展示或粗筛可以，精确归属要结合业务规则和最新数据。
- 行政区缓存要有版本、来源、更新时间和回滚；省市区联动以 adcode 为主，不以中文名称作为唯一主键。
- 大边界渲染要简化几何或按层级加载，避免前端一次加载全国高精度边界。

### 坐标转换与联调排错

- 至少准备 3 组黄金样本：国内城市中心点、边界附近点、海外或无需偏移点，记录 WGS84/GCJ-02/BD-09 预期关系。
- 黄金样本要同时覆盖原始设备点、数据库点、接口入参、接口返回和地图截图；只看截图不能证明坐标链路正确。
- 排错顺序固定为坐标系、经纬度顺序、key 类型、限制配置、权限状态、网络错误、API 返回码、SDK/插件版本、宿主平台限制。
- 禁止用手工加减经纬度、前端硬编码偏移量或“看起来对齐”的截图替代坐标转换证据。

### 弱网、缓存与降级

- Web JS 资源、WebService、定位 SDK 和小程序请求都要设置超时；弱网下优先保证手动输入、最近城市、缓存 POI 或静态兜底页可用。
- 缓存要区分公开地理数据、用户实时位置和敏感位置；实时位置缓存必须短 TTL、最小字段、受权限约束，不得跨用户复用。
- 重试只针对短暂网络和 5xx，且必须退避、限次、可取消；key 错误、权限错误、配额错误和参数错误不得重试。
- 地图 provider 故障时要有关闭地图入口、切换静态列表、隐藏非关键覆盖物或暂停高频任务的回滚方案。

## 验证门禁

- 坐标门禁：每个入口、存储字段、接口参数、地图展示都标明 GCJ-02、WGS84 或其他坐标系，并有转换测试。
- key 门禁：所有 key 按平台隔离，referer/package/IP/Bundle ID/SHA1 限制已配置，服务端签名私钥未出现在客户端。
- 配额门禁：估算日调用量、峰值 QPS、计费项、缓存命中、告警阈值和超额降级。
- 权限门禁：用户同意前不初始化或调用会采集个人信息的 SDK 能力；拒绝授权有可用兜底。
- 隐私门禁：隐私政策、移动端权限文案、SDK 合规声明、数据保留期限和第三方共享说明一致。
- 性能门禁：首屏加载、地图实例数量、覆盖物数量、路线点数量、定位频率和内存释放有证据。
- 测试门禁：覆盖正常、异常、弱网、限流、key 错误、403/429/5xx、权限拒绝、坐标偏移、真机和目标浏览器。
- 发布门禁：提供官方文档核对、配置截图或说明、版本号、灰度方案、监控看板、回滚策略和禁用/切换地图能力的应急方案。
- 证据门禁：至少包含 key 限制证据、WebService infocode/status 证据、权限拒绝证据、弱网或离线证据、坐标黄金样本和低端设备性能证据；缺项必须标注不可发布。

## 输出要求

- 输出必须先说明平台、API 类型、坐标体系、key 类型、权限合规影响和是否需要服务端代理。
- 给方案时必须包含：坐标处理、key 安全、权限/隐私、配额计费、性能、错误处理、测试发布证据。
- 给审查意见时按严重级别标注：阻断、必须修、建议修、观察项。
- 给需求拆解时按端拆分：Web、服务端、Android、iOS、小程序、测试、运维或发布。
- 给数据库或接口建议时字段名必须体现坐标系，不能只写 lat/lng。
- 若引用官方能力，只做概念性对照和核验提醒，不复制官方示例代码。
- 若信息不足，必须列出最少必要问题，不得凭空假设 key 限制、坐标来源或隐私授权状态。
- 输出结论必须给出“可继续实现/需补证据/阻断停止”之一，并说明缺失证据会导致的返工风险。

## 安全边界

- 不帮助绕过高德 key 限制、签名机制、配额计费、referer/package/IP/Bundle ID 校验。
- 不提供隐藏 SDK、规避隐私弹窗、绕过系统定位权限、未授权后台定位的实现建议。
- 不建议采集超出业务必要范围的位置、轨迹、家庭/工作地点或敏感 POI。
- 不把客户端泄露 key 的风险描述为“只要混淆即可”；混淆不是安全边界。
- 不把第三方地图结果用于法律、执法、医疗急救、金融风控等高风险决策的唯一依据。
- 不承诺地图数据、行政区、路线耗时、POI 状态绝对准确；必须保留误差和人工纠偏通道。
- 不帮助批量抓取、缓存或再分发 POI、瓦片、路网、行政区边界来规避授权、配额、计费或服务条款。
- 不帮助移除、遮挡或误导高德版权、审图号、attribution、数据来源和许可说明。

## 反例库

- 反例：前端直接调用高德 WebService 并携带服务端 key 和签名私钥。正确做法：服务端代理签名，前端只调用自家接口。
- 反例：GPS 设备上报 WGS84 后直接在高德地图画点，发现整体偏移再手工加减经纬度。正确做法：使用明确坐标转换链路。
- 反例：所有端共用一个 key，未配置任何 referer、包名、IP 或 Bundle ID 限制。正确做法：按平台和环境隔离并限制来源。
- 反例：App 启动即初始化定位 SDK，用户尚未同意隐私政策。正确做法：合规声明和用户同意后再初始化相关能力。
- 反例：地址输入每次键盘输入都请求 POI，无防抖无限流。正确做法：最小字符数、防抖、城市限制、服务端限流和缓存。
- 反例：发布只验证本机浏览器地图能打开。正确做法：验证目标浏览器、移动端真机、权限拒绝、key 错误、弱网和配额异常。
- 反例：行政区边界文件长期写死在前端包内。正确做法：版本化、按需加载、定期更新并说明边界用途。
- 反例：路线规划结果当作配送承诺时间。正确做法：结合业务缓冲、实时路况、历史数据和异常兜底。
- 反例：为了省调用量批量抓取 POI 或地图瓦片后自建缓存服务。正确做法：使用授权接口、合规缓存策略和配额治理。
- 反例：前端报 403 后把服务端 key 改到浏览器里试运行。正确做法：核对 key 类型、referer/IP 限制和服务端代理签名链路。
- 反例：用户拒绝定位后仍通过后台 SDK 或其他渠道获取位置。正确做法：停止采集并提供手动选择、地址输入或城市级兜底。
- 反例：把 latitude,longitude 直接拼成高德 location 导致点位跑偏。正确做法：确认接口要求 lng,lat，并在字段名和测试里固定顺序。
- 反例：Android release 包换了签名证书后继续用 debug SHA1 的 key。正确做法：为 debug/release/渠道签名分别配置 key 或限制。
- 反例：把 10003、10044、429 都当网络波动自动重试。正确做法：识别配额/限流错误，停止重试并降级、告警或限流。
- 反例：小程序默认拿 wgs84 位置后直接调用高德逆地理。正确做法：请求 gcj02 或服务端转换后再调用高德能力。

## 自检清单

- 是否确认平台：Web JS API、Loader、WebService、Android、iOS、小程序或混合架构？
- 是否确认坐标来源、目标坐标、存储坐标和转换链路？
- 是否区分 GCJ-02、WGS84、BD-09，并避免裸 lat/lng 字段？
- 是否按平台和环境拆分 key，并配置 referer、IP、包名、SHA1、Bundle ID 等限制？
- 是否避免在客户端暴露 WebService key、签名私钥或服务端安全配置？
- 是否估算配额、QPS、计费、缓存和超限降级？
- 是否完成浏览器、小程序、Android、iOS 的定位权限和拒绝授权兜底？
- 是否完成隐私政策、SDK 合规初始化、移动端权限声明和第三方共享说明？
- 是否处理空结果、模糊结果、错误码、弱网、限流、超时和服务不可用？
- 是否控制 loader、插件、覆盖物、路线点、行政区边界和定位频率的性能成本？
- 是否有真机、目标浏览器、生产域名或预发域名、发布配置和监控证据？
- 是否验证 key 错误、referer/IP/包名/Bundle ID/SHA 限制、403/429/5xx、离线/弱网和回滚路径？
- 是否准备坐标黄金样本，并排除经纬度顺序、GCJ-02/WGS84/BD-09 混用和海外点误转换？
- 是否把官方文档作为能力核对依据，同时采用成熟工程实践补足安全、合规、性能和可观测？
- 是否存在误触发：source map、hash map、roadmap、mind map、site map、map/filter/reduce、只读学习或相邻供应商任务？

## 相邻技能边界

- 地图 GIS 核心开发 / map-gis-core（map-gis-core）：负责通用坐标、投影、空间数据建模、地图平台无关 GIS 架构；本技能负责高德平台接入和高德限制。
- JavaScript / TypeScript 开发 / javascript-typescript-development（jsts）：负责几何拓扑、缓冲区、相交、包含、简化等空间算法；本技能只处理高德行政区、边界加载和平台调用。
- Vue 开发 / vue-development（vue）：负责 Vue 组件封装、响应式生命周期和前端工程结构；本技能提供高德地图实例、loader、权限和性能约束。
- API 工程 / api-engineering（api）：负责通用后端接口设计、网关、鉴权、限流；本技能规定高德 WebService 代理、签名、错误码和配额语义。
- 测试验证 / test-engineering（tst）：负责测试策略、自动化和质量门禁；本技能规定高德相关坐标、权限、弱网、真机和发布证据。
- 代码审计 / code-audit（aud）：负责审计与合规评审；本技能提供位置数据、第三方 SDK、调用日志和发布证据审查点。
- Web 安全 / web-security（wsec）：负责 Web 安全、密钥泄露、CSP、请求防护；本技能强调高德 key、referer、签名和客户端暴露风险。
- Apple 全链路开发与发布 / apple-development（appl）：负责 Apple 平台通用开发；本技能负责 iOS 高德 SDK、Bundle ID、定位权限和隐私声明。
- Android 开发 / android-development（andr）：负责 Android 平台通用开发；本技能负责 Android 高德 SDK、包名、SHA1、权限和生命周期释放。
- Kotlin 开发 / kotlin-development（kt）：负责 Kotlin 语言与 Android 实现细节；本技能只给高德平台规则和合规门禁。
- Flutter 开发 / flutter-development（fltr）：负责 Flutter 跨端工程；本技能用于 Flutter 集成高德插件或原生桥接时的平台规则。
- 微信小程序 / wechat-miniprogram（wcmp）：负责微信/小程序工程能力；本技能负责小程序位置、高德服务端代理和坐标边界。
- 性能工程 / perf-engineering（pfe）：负责前端性能优化；本技能关注地图 loader、覆盖物、行政区边界、定位和路线渲染性能。
- 发布部署 / release-engineering（rls）：负责发布、灰度、回滚和变更管理；本技能提供高德 key、配额、合规和真机证据发布门禁。
- 可观测性 / observability（obs）：负责监控、日志、指标和告警；本技能定义高德调用量、错误码、配额、定位失败率和地图加载失败率指标。
