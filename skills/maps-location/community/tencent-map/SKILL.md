# 腾讯位置服务

首次自称：腾讯位置服务（tencent-map，兼容 slug: tencent-map）。

## 定位/适用范围

本技能用于在真实项目中一次性做好腾讯位置服务相关开发、排障、测试与发布，覆盖从需求澄清、Key 与权限配置、坐标与隐私合规、接口选型、代码实现、性能控制到上线证据的完整闭环。

适用平台与能力：

- Web：腾讯位置服务 JavaScript API GL（新版 TMap，WebGL 渲染，入口 `https://map.qq.com/api/gljs?v=1.exp&key=...`，命名空间 `TMap`）与旧版 JavaScript API v2（`qq.maps.Map`，2D Canvas，入口 `https://map.qq.com/api/js?v=2.exp&key=...`，已停止新功能维护），新项目默认选 TMap GL；含地图展示、覆盖物、信息窗体、选点、几何计算、点聚合、轨迹、路线展示、服务类库调用。两套 SDK 的类名、事件名、Marker/Polyline 配置项不互通，迁移时必须重写而非 alias。
- 服务端：WebService API，含地点搜索、关键词提示、地址解析、逆地址解析、路线规划、距离矩阵、行政区划、区域/多边形检索、坐标转换、静态图等。
- Android：腾讯地图 Android SDK、定位、检索、覆盖物、路线/轨迹展示、3D/卫星/路况/热力图、隐私合规接口。
- iOS：腾讯地图 iOS SDK、定位、检索、覆盖物、路线/轨迹展示、3D/卫星/路况、隐私合规与 Bundle 绑定。
- 小程序：微信小程序/QQ 小程序的腾讯位置服务能力，含地图组件、位置授权、选点、搜索、逆地址解析、路线规划、城市/区县能力。
- 跨端：H5、App、小程序、后端之间的坐标、Key、签名、配额、错误码、降级与证据链对齐。

优先触发场景：

- 用户要接入、修改、调试、测试、发布腾讯地图/腾讯位置服务相关功能。
- 需求涉及 GCJ-02 坐标、POI、路线规划、逆地址解析、地图选点、行政区划、Key 安全、配额或签名。
- 项目文件出现 Tencent/TMap/qq.maps/QQMapWX/apis.map.qq.com/lbs.qq.com/腾讯地图等实质代码或配置证据。

不适用：只读学习、泛泛介绍、普通微信业务、SEO、站点地图、source map/hash map/roadmap/mind map、map/filter/reduce，或仅涉及 Google/高德/百度/天地图/Mapbox/ArcGIS/Leaflet 且没有腾讯迁移/互操作需求。

触发前必须排除误召回：

- 只有 README、依赖清单、截图、教程链接、控制台下载说明或 onboarding 文案，且用户没有要求修改/调试/测试/发布腾讯位置服务功能时，不触发本技能。
- 出现 map、maps、location、route、district 等普通变量名但无 TMap、腾讯位置服务 API、Key、SDK、域名、坐标或业务动作证据时，不触发本技能。
- 只问概念学习或官方文档入口时只做轻量回答，不进入开发流程；一旦涉及落地改动、排障、发布证据或迁移互操作，再启用本技能。
- 相邻地图供应商任务默认不触发；只有明确从/到腾讯迁移、双供应商坐标互通、腾讯接口替换或兼容腾讯地图展示时才触发。

## 铁律

1. 坐标铁律：腾讯位置服务国内业务默认以 GCJ-02 火星坐标为主线处理；不要把 WGS-84、BD-09、GCJ-02 混用。任何入库、传参、展示、路线计算、逆地址解析前必须标注坐标系。
2. Key 铁律：前端 Key 只用于可公开但受限制的场景；高价值 WebService 调用必须优先放后端代理，并启用签名、IP 白名单或服务端限制。
3. 限制铁律：Web 关注 Referer/域名限制，小程序关注合法域名与 appid/平台限制，Android 关注包名/签名证书，iOS 关注 Bundle ID，服务端关注 IP 白名单与签名。
4. 隐私铁律：定位前必须完成隐私政策告知、系统权限申请、最小化采集与用户拒绝路径；移动端 SDK 按官方隐私合规接口要求在初始化/调用前完成合规配置。
5. 配额铁律：所有搜索、逆地址解析、路线规划、行政区接口必须设计状态码处理、限流、缓存、重试与降级，不允许把配额错误当作普通网络失败忽略。
6. 安全铁律：禁止绕过 Key、签名、Referer、IP、包名、Bundle、域名限制；禁止泄露 Secret Key；禁止把签名密钥写入前端、小程序包或 App 包。
7. 数据铁律：禁止抓取 POI、瓦片、路况、轨迹等数据用于离线复制、批量爬取或规避服务条款；禁止隐藏腾讯地图署名/版权/Logo。
8. 签名铁律：签名只用于证明服务端合法请求，不是前端防盗链；签名算法、SecretKey、白名单与接口路径必须按官方口径核对，禁止自创兼容逻辑或客户端代签。
9. 证据铁律：上线前必须留下配置截图/配置项摘要、测试坐标、请求样例、错误码处理、隐私授权、性能与配额证据；缺证据即视为未完成。
10. 计费铁律：上线前必须确认调用接口、QPS、日配额、商业授权、计费口径、余额/欠费告警和超限降级；不能把测试 Key、免费额度或控制台默认额度当生产容量。

## 快速总则

- 先判平台，再判调用链：Web/TMap、小程序、App SDK、后端 WebService 的 Key 类型、限制方式和权限模型不同，不能复用一套配置假设。
- 先判坐标，再写代码：输入来源是 GPS/WGS-84、腾讯/GCJ-02、百度/BD-09、第三方地址还是用户选点；输出用于展示、入库、导航还是搜索。
- 前端只做展示与轻量交互：高频、批量、带密钥或可被滥用的搜索/逆解析/路线/行政区调用放后端。
- 小程序必须双配置：腾讯位置服务 Key 权限 + 小程序后台合法域名/定位权限/隐私声明；缺一不可；用 `gcj02` 获取地图可用坐标，区分平台内置地图组件、选点 API 与腾讯 JS SDK 数据接口。
- 移动端必须双验证：Debug/Release 包名签名或 Bundle 配置都要验证，不能只测调试包。
- 路线规划必须定义交通方式、坐标顺序、起终点来源、途经点数量、避让策略、结果为空时 UX 与降级。
- 地点搜索必须定义城市/区域限制、类别、分页、去重、排序、用户输入防抖、结果可信度与无结果提示。
- 逆地址解析必须定义是否需要 POI、行政区、道路、门牌、位置描述精度，以及坐标偏移异常处理。
- 地图选点必须定义拖图/点选/搜索回填的单一真源，避免地址、经纬度、POI ID 三者互相覆盖。
- 行政区/区域检索必须区分行政区划、城市区县、AOI/多边形范围、乡镇街道轮廓，不要用文本包含替代空间关系。
- 弱网默认降级：展示上一次授权位置、最近搜索结果或静态兜底，不在用户移动中持续重试；缓存命中必须标注时效和坐标系。

## 强制流程

### 1. 需求与平台判定

- 明确用户动作：开发、修 Bug、联调、测试、发布、迁移、性能优化还是安全整改。
- 明确平台：Web/TMap、后端 WebService、Android、iOS、微信小程序、QQ 小程序、跨端混合。
- 明确能力对象：定位、地图展示、地图选点、地点搜索、关键词提示、逆地址解析、地址解析、路线规划、距离矩阵、行政区/区域检索、坐标转换、静态图。
- 明确数据流：用户输入、系统定位、数据库坐标、第三方坐标、腾讯接口返回值、前端展示、服务端存储。
- 明确上线环境：开发、测试、预发、生产是否使用不同 Key、不同限制、不同配额。
- 明确计费面：哪些调用消耗腾讯配额或产生费用，哪些是微信内置能力或本地缓存；估算峰值 QPS、日调用量和失败重试放大倍数。
- 明确停止条件：缺少平台、Key 类型、坐标系、调用链、权限模型或测试环境时，不直接改代码，先补齐最小事实；发现真实密钥泄露、绕限制、爬取、隐藏署名或隐私绕过诉求时立即转安全整改/拒绝。

### 2. 坐标与数据契约

- 为每个经纬度字段写清：longitude、latitude、坐标系、精度、来源、更新时间、是否用户授权、是否可持久化。
- 统一经纬度顺序：接口文档要求是 lng,lat 还是 lat,lng 必须逐项核对；UI、数据库、日志不能混写。
- 国内腾讯地图展示、搜索、逆解析、路线规划优先使用 GCJ-02；来自 GPS 的 WGS-84 必须先转换或标注后再调用。
- 涉及百度坐标时显式做 BD-09 与 GCJ-02 转换；不要用肉眼偏移修正。
- 坐标转换必须集中封装，禁止在多个页面散落魔法公式。
- 存量数据迁移必须抽样校验城市中心、道路点、边界点、港澳台/海外点。

### 3. Key、签名与权限配置

- 为每个平台申请独立 Key：Web、WebService、Android、iOS、小程序、测试环境、生产环境尽量隔离。
- Web Key：配置允许域名/Referer；本地开发域名单独配置；不要使用通配过宽的生产 Key；Referer 不是密钥保护，涉及费用、配额或敏感业务仍走后端代理。
- WebService Key：服务端调用优先启用 IP 白名单；涉及高价值接口或服务端调用时启用签名校验；签名密钥只存服务端密钥管理或环境变量。
- Android Key：绑定包名与签名证书指纹；Debug 与 Release 分别验证；多渠道包、加固包、应用市场重签名要单独确认。
- iOS Key：绑定 Bundle ID；企业包、TestFlight、App Store 包使用同一或分离 Key 时要有明确策略。
- 小程序 Key：确认 WebServiceAPI 权限、微信/QQ 小程序平台配置、合法 request 域名 `https://apis.map.qq.com`、appid/插件/SDK 版本差异。
- 配额与计费：按接口维度配置日配额、QPS、预算告警、欠费告警和熔断阈值；测试环境用低额度 Key，生产环境变更必须有审批和回滚记录。
- 禁止把服务端 Secret、签名计算盐、管理后台账号、未限制 Key 写进仓库、前端环境变量、小程序包或移动端资源文件。

### 4. 接口选型与错误处理

- 地点搜索：优先限定城市、区域、中心点、半径、类别与分页；对用户输入做防抖和取消过期请求；处理同名 POI、跨城结果、结果为空。
- 关键词提示：只用于输入联想，不等于最终 POI；选择后仍需确认经纬度、地址与城市。
- 地址解析：适合把结构化地址转坐标；必须处理模糊地址、多结果、行政区缺失、解析失败。
- 逆地址解析：适合坐标转地址；必须处理偏移坐标、海上/边界点、无门牌、无 POI、行政区层级不完整。
- 路线规划：按驾车、步行、骑行、公交、货车、新能源等场景选择接口；处理限行/绕行/无路线/跨城/途经点过多/实时路况差异。
- 距离矩阵：适合批量估算，不等于实际导航路线；限制规模、缓存结果并标注时效。
- 行政区划：用于省市区列表、编码、边界；不要用地点搜索替代行政区接口。
- 区域/多边形检索：用于判断点是否在范围内、范围内 POI 或 AOI；注意边界点与坐标系一致。
- 状态码：统一解析腾讯接口 status/message/request_id 与部分接口的 info_code/infocode；status=0 才视为成功，其他常见状态固定 mapping（以官方文档当期为准，禁止只 if status===0）：
  - 参数/请求错误段：`310` 请求参数信息有误、`311` Key格式错误、`306` 请求有护持信息请检查字符。
  - 鉴权失败段：`110` 请求来源未被授权（Referer/包名/Bundle/IP/域名不匹配）、`111` 签名验证失败（sig 计算错误或 SK 不匹配）、`112` IP未被授权、`113` 配额已用完（日配额）、`120` 请求的接口未被授权（Key 未勾选该 WebService 能力）、`190` KEY 不合法或被封禁、`199` 此 Key 每秒请求超限或本次请求未被授权。
  - 限流段：`121` 请求并发数过限、`117` 配额超限/QPS 过限。
  - 业务无结果段：`348`/部分接口返回 0 但 result 为空。
  - 服务端异常段：`5xx` HTTP 层或腾讯网关；status `198` 服务不可用类。
  收到 11x/12x/19x 类禁止盲目重试，需走鉴权排查或降级；收到 5xx/超时类才有限指数退避。
- 异常分层：403/鉴权类先查 Key 与限制；429/配额类先降级与告警；5xx/超时类才有限重试；小程序与浏览器跨域/域名错误要回到平台配置排查。
- 重试：只对短暂网络/服务异常做有限指数退避；鉴权、签名、参数、配额错误不盲目重试。

### 5. 实现与性能控制

- TMap 初始化只做一次，组件销毁时释放监听器、覆盖物、定时器和异步请求。
- 大量点位使用点聚合、视野内加载、分页/分片渲染、WebWorker 或服务端裁剪；不要一次渲染全量 POI。
- 地图拖动、缩放、搜索输入、逆解析联动必须防抖/节流，并取消过期请求。
- 覆盖物状态使用稳定 ID；避免每次状态变化全量重建 Marker/Polyline/Polygon。
- 路线、轨迹、热力图和多边形数据压缩传输；控制点数、抽稀精度和刷新频率。
- 服务端对地址解析、逆解析、行政区、静态配置做缓存；缓存键必须包含坐标系、接口参数、城市、语言/区域等影响结果的字段。
- 弱网缓存只缓存稳定、低敏结果；用户精确坐标、完整地址、轨迹和家庭/工作地点默认不做长期缓存，必须设置 TTL、脱敏和清理机制。
- 日志记录 request_id、status、耗时、Key 标识、接口名、环境、坐标系，但不要记录完整精确轨迹、手机号、身份证、详细住址等敏感信息。

### 6. 隐私、合规与用户体验

- 定位前展示明确用途：如附近门店、配送地址、路线规划；拒绝授权时提供手动选点/输入地址。
- 只在必要时请求精确定位；不需要后台定位时不要申请后台定位。
- 移动端和小程序按平台要求配置隐私政策、权限文案、用户授权流程、SDK 合规初始化。
- 个人位置数据最小化存储：能用临时坐标就不持久化，能用城市级就不用门牌级。
- 跨境、海外、港澳台或敏感区域场景，必须确认腾讯服务能力、坐标系、法规和产品条款。
- 用户可见地图必须保留腾讯地图版权、Logo、审图号/归属要求，不做遮挡或替换。

### 7. 测试与发布证据

- 使用至少三类测试点：一线城市高楼密集区、普通道路/社区、边界/郊区/无 POI 点；路线场景加跨区与无路线样例。
- 每个平台分别测试：开发环境、预发环境、生产 Key；Debug 与 Release；真机与模拟器/浏览器；微信与 QQ 小程序如都支持则分别测。
- 鉴权测试：正确 Key、错误 Key、错误 Referer/IP/包名/Bundle、签名缺失/错误、配额不足。
- 权限测试：首次授权、拒绝授权、二次授权、系统定位关闭、弱网、无网、低精度定位。
- 性能测试：首屏地图加载、拖拽缩放帧率、覆盖物数量上限、搜索响应时间、接口 QPS、内存与电量。
- WebService 负面样例必须覆盖：错误 Key、未授权接口、签名缺失/错误、Referer/IP/包名/Bundle/域名不匹配、403、429、5xx、超时、无结果、参数错误。
- 坐标金样本必须覆盖：已知城市地标、道路交叉口、行政区边界、WGS-84 转 GCJ-02、BD-09 转 GCJ-02、经纬度顺序颠倒的失败保护。
- 小程序定位必须覆盖：wx.getLocation 成功/拒绝/系统定位关闭/弱网、wx.chooseLocation 取消/拒绝、gcj02 与 wgs84 选择差异、合法域名缺失。
- 移动端必须覆盖：低端机、冷启动、前后台切换、权限变更、离线/弱网、Release 包签名或 Bundle 绑定失败。
- 计费与配额证据必须覆盖：接口用量估算、QPS 峰值、日配额、余额/欠费告警、超限熔断、缓存命中率和降级文案。
- 发布证据：配置摘要、关键截图、请求/响应样例、错误码截图、隐私授权截图、性能数据、回滚方案、监控告警链接。

## 场景执行卡

### Web / TMap 地图展示与选点

- 检查 Key 是否绑定正确域名/Referer，本地、预发、生产分离。
- 初始化 TMap 时确认容器尺寸、中心点、缩放级别、地图样式、加载失败兜底；JS SDK 加载失败、Key 错误、网络离线时必须有非阻塞降级。
- 区分 TMap 前端能力与 WebService 数据能力：浏览器不得持有服务端 Secret，不得为高价值搜索/路线/行政区接口直接代签。
- 选点交互以坐标为主真源，地址为逆解析结果；拖图、点选、搜索选择只能有一个最终提交入口。
- 拖动地图时节流逆解析；用户停止拖动后再请求；请求返回前若坐标已变化则丢弃旧结果。
- 覆盖物、信息窗体、路线图层在组件卸载时清理。

### WebService 后端接口

- 后端封装统一客户端：签名、超时、重试、状态码、日志、限流、缓存、熔断集中处理。
- Key 和 Secret 从密钥管理读取，不进入代码库、镜像日志或错误响应。
- 对外接口不要透传腾讯原始错误给终端用户；内部日志保留 request_id 和 status 便于排障。
- 对批量地址解析/逆解析设置队列和速率限制，禁止一次性并发打满配额。
- 对签名请求固定参数排序、编码、路径与查询串规范，并用官方示例或控制台工具交叉验证。腾讯 WebService 签名口径（sig 参数）：取请求 path（如 `/ws/geocoder/v1/`，必须以 `/` 开头和结尾按接口实际路径写）+ `?` + 按参数名 ASCII 升序拼接的 query 串（参数值需 URL 编码，且不要把 sig 自身计入）+ SK（直接拼在末尾，不加任何分隔符），对该完整字符串做 **MD5 32 位小写** 作为 sig 随原请求 query 一起发送。每个 Key 对应一把 SK，SK 仅服务端保存，禁止前端/小程序/App 任何形式持有；POST 表单接口也用同样规则按表单参数排序计算。控制台开启「校验 sig」后未带或错算 sig 会返回 111。
- 生产前压测 QPS 与日配额，确认超限告警和降级文案。

### Android SDK

- 核对包名、签名证书、Debug/Release Key、混淆/加固影响。腾讯地图 Android 控制台绑定格式为「包名;SHA-1 指纹」，SHA-1 必须使用大写冒号分隔的 40 位十六进制（如 `AA:BB:CC:DD:EE:FF:11:22:33:44:55:66:77:88:99:00:AA:BB:CC:DD`），用 `keytool -list -v -keystore xxx.jks` 对 debug.keystore 与正式 keystore 分别取值；MD5 指纹不被接受。一个 Key 可绑定多条 `包名;SHA1`（换行分隔），调试包与正式包、加固后重签名包都要分别加入，否则 SDK 报 110。
- 按官方隐私合规要求在 SDK 初始化和定位前完成用户同意流程，未同意前不初始化会采集设备/位置数据的能力。
- 动态申请定位权限，处理拒绝、仅使用期间、精确/大致位置、系统定位关闭。
- 地图生命周期跟随 Activity/Fragment，暂停/恢复/销毁时释放资源。
- 大量 Marker 使用聚合或分页；列表联动地图时避免频繁全量刷新。
- Release 包、加固包、多渠道包若鉴权失败，优先核对签名证书指纹与控制台配置，而不是更换未限制 Key。

### iOS SDK

- 核对 Bundle ID、测试包与正式包配置、Info.plist 定位权限文案。
- 初始化和定位调用前完成隐私政策展示与用户授权判断，未同意前不启动会采集设备/位置数据的能力。
- 处理 iOS 精确/大致位置、定位服务关闭、权限变更、后台定位边界。
- 地图 View 生命周期与控制器一致，释放 delegate、overlay、annotation 与定时器。
- 真机验证 App Store/TestFlight 包，避免只在 Debug 包通过；Bundle 鉴权失败先查控制台绑定与包标识。

### 微信/QQ 小程序

- 配置腾讯位置服务 Key 的 WebServiceAPI 权限，并在小程序后台加入合法 request 域名。微信小程序使用腾讯 QQMapWX JSSDK 时，request 合法域名固定为 `https://apis.map.qq.com`；同时需要在 app.json 的 `requiredPrivateInfos` 中声明 `getLocation`/`chooseLocation`/`onLocationChange`/`startLocationUpdate` 中实际用到的接口，并在 `permission` 中配置 `scope.userLocation` 的用途说明，2022 年 7 月后未声明会直接抛 `getLocation:fail the api need to declare in the requiredPrivateInfos field in app.json`。
- 检查 app.json/平台后台定位权限与隐私接口；拒绝授权时提供手动地址输入或地图选点。
- 区分小程序地图组件展示能力、平台选点/打开位置 API、系统定位 API 与腾讯位置服务 JS SDK 数据接口能力。`<map>` 组件底层是腾讯地图瓦片但配额走微信侧、不消耗自家 Key；QQMapWX 插件 SDK 走自家 Key+小程序合法域名 `apis.map.qq.com`；平台 `wx.chooseLocation` 走微信内置地图（返回 gcj02 + name/address，不消耗腾讯配额）。三套结果字段名不同，业务层要统一适配，不要相互冒充。
- 获取位置时明确坐标系：`wx.getLocation` 默认 `type: 'wgs84'`（返回 GPS 原始坐标，仅适合需要 GPS 语义如轨迹记录的场景），传 `type: 'gcj02'` 才能直接用于 `<map>` 组件、`wx.openLocation`、QQMapWX 检索/逆解析；`wx.chooseLocation` 返回值固定为 gcj02。混用会出现几百米偏移。
- 搜索和逆解析调用做防抖、分页、错误码提示；不要在 onShow/onLoad 中无条件高频请求。
- 微信与 QQ 小程序如同时支持，分别验证 SDK、权限弹窗、域名、定位行为和审核要求。

### 地点搜索 / 关键词提示

- 搜索前确认城市、中心点、半径、类别和分页策略。WebService 地点搜索 `/ws/place/v1/search` 的 `boundary` 与 `region` 互斥：`boundary` 支持三种形式 `region(城市名,自动扩展0/1,中心点lat,lng)`、`nearby(lat,lng,radius)`、`rectangle(lat1,lng1,lat2,lng2)`，必须二选一传入；`filter` 用 `category=` 或 `tag=` 过滤分类，`page_size` 上限 20，`page_index` 从 1 开始。
- 输入联想只展示候选；最终提交需绑定坐标、POI 名称、地址、城市、adcode。
- 处理同名、多结果、跨城、无结果、用户手动输入不选择候选。
- 缓存热门关键词和稳定结果，但避免缓存用户敏感完整地址过久。

### 逆地址解析 / 地址解析

- 坐标转地址前确认坐标系和经纬度顺序。逆地址解析 `/ws/geocoder/v1/` 的 `location` 参数顺序为 `lat,lng`（纬度在前），坐标系通过 `coord_type` 指定（1=GPS WGS-84、2=Soso/腾讯地图，默认 5=腾讯/GCJ-02）；如需返回周边 POI 必须显式传 `get_poi=1`，并可用 `poi_options=address_format=short;radius=1000;policy=1;page_size=20;page_index=1` 控制召回；不传 `get_poi` 时默认只返回 `formatted_address` 与 `address_component`。
- 地址转坐标前尽量提供城市/行政区上下文，提高命中率。
- 对解析结果标注可信度：精确 POI、道路、门牌、社区、行政区级。
- 出现偏移、空结果或低可信度时引导用户手动修正点位。

### 路线规划

- 明确交通方式与业务目标：最短时间、最短距离、避堵、配送、步行可达、公交换乘。WebService 路线规划 `/ws/direction/v1/` 提供 `driving`/`walking`/`bicycling`/`transit`/`truck`/`motorcycle` 子能力；其中 **truck（货车路线）需在控制台单独申请权限并通过审核** 后才能调用，未开通直接返回 120。`transit` 公交不支持途经点，`driving` 途经点上限 16 个，`waypoints` 用 `;` 分隔，顺序为 `lat,lng`。
- 检查起终点坐标系、途经点数量、城市范围、车辆限制、出发时间。新能源/电动车场景需要 `vehicle_type` 与电量/续航参数；货车需 `truck_type`、轴数、限高、限重、限长、危险品等参数齐全。
- 对无路线、跨区域不可达、限行、接口超限给出用户可理解提示。
- 路线展示要压缩 polyline 点数，避免低端机卡顿。

### 距离矩阵 / 配额计费 / 弱网缓存

- 距离矩阵只用于批量距离或耗时估算，不等同实时导航承诺；必须限制起终点规模、交通方式、坐标顺序、缓存 TTL 和结果时效提示。
- 大批量门店、配送或派单场景先做本地粗筛，再调用距离矩阵精排；禁止全量笛卡尔积请求把配额打满。
- 缓存键必须包含起终点坐标系、交通方式、时间段、城市和影响路线的策略；跨天、路况变化或权限变化后不得复用旧结果冒充实时结果。
- 计费排查先看接口维度调用量、失败率、重试放大、缓存命中、QPS 峰值、Key 归属和控制台余额；不要只看应用日志成功率。
- 弱网/离线时按场景降级：地图展示可保留已加载瓦片或静态提示，搜索/逆解析/路线展示上次结果并标注过期，不自动提交陈旧坐标。

### 行政区 / 区域检索

- 行政区列表和编码使用行政区划接口，不用本地硬编码长期替代。
- 区域内 POI 使用范围/多边形/AOI 能力，避免只按名称或行政区文本过滤。
- 边界判断要统一坐标系并处理边界点、洞、多多边形和数据更新。
- 行政区缓存要有版本或刷新机制。

## 反例库

- 反例：把 WebService Secret 写进前端 `.env`，让浏览器直接签名。正确做法：签名在服务端完成，前端只调用自家后端。
- 反例：用 GPS 原始 WGS-84 坐标直接在 TMap 上打点，发现偏移后手动加减常数。正确做法：统一坐标转换并标注坐标系。
- 反例：搜索框每输入一个字立即请求地点搜索。正确做法：防抖、取消过期请求、关键词提示与最终搜索分离。
- 反例：生产 Key 允许所有 Referer 或所有 IP。正确做法：分环境 Key，并配置最小范围限制。
- 反例：只在 Android Debug 包验证地图可用。正确做法：Release 签名包、加固包、多渠道包都验证。
- 反例：小程序只申请腾讯 Key，忘记配置合法域名和隐私权限。正确做法：Key 权限、平台域名、定位授权三者同时检查。
- 反例：把逆地址解析结果当作精确门牌。正确做法：展示可信度，允许用户修正。
- 反例：一次加载全市十万 POI Marker。正确做法：按视野分页、聚合、抽稀或服务端瓦片化数据。
- 反例：配额超限时前端无限重试。正确做法：停止重试、提示降级、触发告警和配额处理。
- 反例：为美观遮挡腾讯 Logo。正确做法：遵守地图展示版权与归属要求。
- 反例：只看到项目有 sitemap.xml 或 source-map 配置就启用腾讯地图排障。正确做法：确认是否存在腾讯位置服务实质代码、Key、API 或用户要求。
- 反例：把 429 当网络抖动无限退避重试。正确做法：停止高频调用、降级、告警并处理配额或限流策略。
- 反例：小程序用 wx.getLocation 的 wgs84 坐标直接提交腾讯逆解析。正确做法：按地图展示/检索需求使用 gcj02 或集中转换。
- 反例：只验证成功路径截图就发布。正确做法：同时留存鉴权失败、权限拒绝、弱网、无结果、配额和回滚证据。
- 反例：把距离矩阵结果永久缓存给配送 ETA。正确做法：设置短 TTL，标注估算时效，路况或时间段变化后重新计算。
- 反例：用生产 Key 跑压测或批处理导入。正确做法：使用隔离 Key、队列限速、预算告警和人工开关。
- 反例：弱网下自动提交上一次定位坐标。正确做法：只作展示兜底，提交前要求用户确认位置时效和地址。

- 测试验证 / test-engineering（tst）：测试框架与自动化由 tst 处理；腾讯地图测试点、鉴权门禁、坐标与平台证据由本技能提供。
- aud / wsec：安全审计和 Web 安全由 aud/wsec 处理；腾讯 Key 泄露、签名密钥、Referer/IP 限制由本技能提供领域规则。
- appl / andr / kt / fltr：App 平台工程、权限和生命周期由对应移动技能处理；腾讯 SDK 配置、定位合规、包名/Bundle Key 规则由本技能提供领域规则。
- 微信小程序 / wechat-miniprogram（wcmp）：微信/QQ 小程序工程和审核由 wcmp 处理；腾讯位置服务小程序 Key、合法域名、地图组件与 JS SDK 差异由本技能处理。
- 性能工程 / perf-engineering（pfe）：前端性能通用优化由 pfe 处理；地图覆盖物、点聚合、拖拽缩放、服务调用节流由本技能处理。
- rls / obs：发布与可观测性由 rls/obs 处理；腾讯配额、状态码、request_id、Key 环境证据由本技能提供检查项。
