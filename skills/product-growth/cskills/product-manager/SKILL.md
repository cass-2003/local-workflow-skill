# 产品经理

产品经理（product-manager，兼容 slug: pm）负责本技能描述范围内的定位、执行、验证和交接边界；旧短 slug 仅作兼容 alias/URL 主键，不作为规范技能名。

定位：把模糊诉求、业务目标、用户问题和上线压力，收敛成可执行、可排序、可验收、可复盘的产品方案。
铁律：没有目标、用户、范围、证据，不进入方案；没有验收、指标、风险、依赖、关闭证据，不进入交付。

## 快速总则（目标/用户/范围/证据）

- 目标：先定义用户要完成的 Job to be Done、业务目标、成功标准和不成功信号；禁止只写“提升体验”“补齐能力”。
- 用户：区分真实使用者、购买者、管理员、运营、客服、风控、反对者和受影响人；B2B 不把客户公司等同于终端用户。
- 范围：写清 scope、out of scope、MVP、后续版本、需求切片、延期项和停止条件；先跑通最小价值闭环。
- 证据：结论绑定数据、访谈、工单、客服反馈、日志、analytics、竞品、实验、销售机会或合规约束；证据弱必须标需验证。
- 决策：每个取舍写牺牲什么、保护什么、谁确认、何时复盘；不能把职位、声音大小或竞品截图当优先级依据。
- 交付：PRD 必须让设计、研发、测试、数据、运营可开工；验收标准、埋点口径、发布策略、回滚和风险 owner 不得缺省。

## 需求防歧义门禁

任何需求进入方案前，先把模糊词翻译成可验证定义：
- “优化”：当前基线、目标阈值、影响人群、观察周期、不能牺牲的护栏。
- “支持”：具体角色、入口、权限、输入、输出、状态变化、失败反馈。
- “自动”：触发条件、可人工接管点、误判处理、审计日志、撤销和恢复。
- “实时”：允许延迟、刷新机制、失败降级、数据一致性和用户提示。
- “个性化”：使用哪些数据、是否可解释、是否可关闭、冷启动和隐私授权。
- “管理”：增删改查、批量操作、审核、导入导出、权限、日志、回收站。

澄清不完时不要补方案，先输出待确认问题、默认假设和假设失效后的改动影响。

## 场景执行卡

### 1. 需求澄清：一句话诉求变成可判断问题

适用：用户说“加个功能”“优化一下”“竞品有我们也要有”“老板要求本周上线”。
动作：
- 还原 JTBD：用户在什么情境下，为了什么进展，当前替代方案是什么，失败代价是什么。
- 标注来源：老板、客户、销售、运营、客服、用户反馈、数据异常、政策变化、竞品动作；同时标证据强度。
- 拆冲突：收入、留存、成本、风控、体验、合规、品牌之间是否互相牺牲。
- 先问非目标：哪些人群、端、场景、数据、权限、历史包袱本期不解决。
产出：问题定义、目标用户、核心场景、成功指标、范围边界、待验证问题、决策人。

### 1.1 Problem framing：先判定是不是值得做

必须写清：
- 问题陈述：谁在什么场景遇到什么阻碍，现有替代方案为何不够，代价是什么。
- 机会大小：受影响用户数、频次、收入/留存/成本/风险影响、战略窗口和证据可信度。
- 成功定义：用户成功、业务成功、不可接受结果、上线后多久判断。
- 非目标：本期不解决的人群、端、流程、历史数据、运营动作和技术债。
- 约束：时间、人力、平台政策、合规、安全、数据可得性、商业承诺和发布窗口。
- 决策请求：现在需要决定做/不做、先做哪段、投入多少，还是先验证假设。

反例：用户说“做一个看板”就开始画页面；应先确认谁用看板做什么决策、当前决策如何失败、哪些指标改变才算成功。

### 2. 用户研究：把证据从传闻变成可用判断

适用：证据来自少数客户、内部直觉、竞品截图、异常数据，或需要验证需求真实性、易用性、付费意愿。
必须问：
- 研究目标：要验证哪条假设、决策会如何受结果影响、什么结果会停止或转向。
- 招募分层：新老用户、高低频、付费免费、沉默用户、流失用户、购买者、管理员、终端用户、反对者是否覆盖。
- 方法选择：访谈用于动机和语境；问卷用于规模化验证；可用性测试用于任务完成；diary study 用于长期行为；日志用于实际行为校验。
- 样本偏差：重度用户、最大客户、销售转述、内部员工、激励样本不能直接代表全量。
- 合成方法：把原话、行为证据、频次、影响面、付费/留存关联、反例和需验证点分开。
输出：研究问题、样本分层、访谈/问卷提纲、任务脚本、洞察证据表、机会清单、需定量验证项。
反例：只访谈 3 个重度用户就代表全量；应补沉默用户、流失用户、购买者/使用者分离和行为数据校验。
转相邻技能：外部行业事实、竞品深研、正式用户研究材料交 rsch；隐私授权与敏感人群交 law。

### 3. PRD：研发、设计、测试能开工

最小结构：
- 背景与目标：问题、证据、JTBD、业务目标、用户目标、非目标。
- 用户与旅程：persona、用户旅程、入口、关键路径、退出、失败、重试、跨端状态。
- 功能范围：MVP、需求切片、scope、out of scope、版本切分、配置项。
- 规则与交互：业务规则、权限、状态机、异常、通知、文案、运营后台、客服处理。
- 数据与指标：North Star Metric、guardrail metrics、analytics 事件、漏斗、留存、分群、口径。
- 验收与发布：acceptance criteria、风险、依赖、灰度、监控、回滚、客服话术、复盘时间。
禁止：只有页面截图、只有功能清单、没有验收标准、没有指标、没有边界、没有关闭证据。

### 4. 用户故事与验收：可开发可测试

模板：作为某类用户，我想在某场景完成某任务，以便获得某结果。
必须包含：正常路径、异常路径、权限、状态、输入输出、数据一致性、通知、埋点、兼容、可撤销或可恢复。
边界至少覆盖：空、错、重、慢、断、并、撤、权限变更、跨端、历史数据、灰度用户、回滚后状态。
产出：user story 列表、acceptance criteria、edge case、out of scope、测试关注点。

验收标准写法：
- Given/When/Then 适合行为验收；同时补输入约束、状态变化、错误提示和数据落点。
- 每条 AC 只验一件事；避免“并且页面美观、流程顺畅、性能稳定”这种混合句。
- 对列表、表单、支付、权限、审批、导入导出、通知、推荐、AI 输出，必须补失败和回滚状态。
- 研发可据此实现，测试可据此写用例，数据可据此验埋点，运营/客服可据此回答用户。

边界矩阵至少扫一遍：角色、权限、端、入口、状态、时间、数量、格式、网络、并发、历史数据、灰度、国际化、无障碍、降级、回滚。

### 5. MVP、需求切片与产品发现：先验证核心假设

判断顺序：
- 是否直接解决核心 JTBD，而不是只做入口、装修或后台。
- 是否能验证关键假设并收集有效 feedback。
- 是否能在约束内交付，且失败时可回滚、可停止、可复盘。
- 是否把“必须有”与“上线后可学到再做”分开。
发现方法：
- 用 opportunity solution tree 把目标、机会、方案、实验串起来，避免先定方案再找理由。
- 用假设地图拆价值风险、可用性风险、可行性风险、商业风险；优先验证最高风险假设。
- 低成本验证可选原型测试、fake door、concierge MVP、Wizard of Oz、手工服务、定向 beta。
- 不把 fake door 当欺骗式增长；必须控制用户损害、告知边界和补偿方案。
输出：MVP 清单、后置清单、明确不做清单、扩量触发条件、停止条件、假设优先级、验证计划。
反例：首版同时做推荐、报表、自动化和后台；应只保留验证核心假设的最短闭环。

### 6. priority：优先级不是谁声音大

排序依据：用户价值、业务价值、证据强度、研发与运营成本、合规风险、时机窗口、战略契合。
可用 RICE、ICE、MoSCoW、机会评分，但必须说明权重、数据来源、tradeoff 和延期风险。
冲突处理：把冲突项放进同一张决策表，写清推荐项、被牺牲项、保护指标、决策人和复盘点。

RICE 使用门禁：
- Reach：受影响用户/客户/交易/工单数量，写时间窗和分群，不能用“很多用户”。
- Impact：对核心指标的预估影响，写高/中/低背后的证据，不把主观兴奋当影响。
- Confidence：证据强度单独计分；访谈、日志、实验、合同、合规要求分别标注。
- Effort：包含研发、设计、测试、数据、运营、客服、法务、安全、迁移和维护成本。
- 输出必须含推荐排序、延后项风险、机会成本、决策 owner、下次复盘条件。

### 7. roadmap 与 portfolio：路线图绑定目标和决策点

必须写：阶段目标、North Star Metric、里程碑、假设、交付物、依赖、风险、暂停条件、转向条件、扩大投入条件。
portfolio 视角：多产品线要写资源分配依据、平台债/技术债与业务需求的取舍、sunset/下线计划、迁移影响和维护成本。
禁止：把季度愿望清单按月份排列后叫 roadmap；把所有客户定制都塞进平台路线图。

### 8. 指标、埋点、OKR 与增长留存

指标体系：
- North Star Metric：反映用户持续获得核心价值的长期指标，不能用注册数、曝光数直接替代。
- guardrail metrics：投诉、退款、误触、滥用、成本、延迟、失败率、内容安全、隐私风险。
- analytics：事件名、触发时机、去重、时区、样本、分母、统计周期、归因、数据源、看板 owner。
- 增长与留存：区分获客、激活、转化、复购、留存、召回；保留 cohort、holdout 和反指标。
OKR 拆解：
- Objective 写用户/业务结果，不写项目交付物。
- Key Result 区分 lagging indicator、leading indicator、输入指标；上线日期通常不是结果指标。
- 建指标树：公司目标、产品目标、用户行为、功能采用、渠道/运营动作、护栏指标之间要能解释因果假设。
- 复盘节奏：周看输入与异常，月看趋势与分群，季度看结果与战略取舍。
输出：指标定义、埋点口径、数据源、观察周期、误判风险、决策阈值、OKR/指标树、复盘 cadence。
反例：把项目按时上线写成 KR；应写上线后用户行为或业务结果，并配输入指标。

### 9. A/B 实验与实验读数

实验前：写清 hypothesis、目标指标、护栏指标、样本量、分层、周期、最小可检测效果、排除人群、终止条件。
实验中：避免入口、文案、价格、推荐逻辑多变量同时变化；防样本污染、跨端串组、活动干扰和数据延迟。
实验后：区分统计显著、业务显著、短期波动和长期副作用；输出继续、停止、扩大或重做的决策。
不适用：样本不足、强伦理风险、强合规风险、强网络效应、不可逆伤害场景，不强行 A/B test。

### 10. Pricing/Packaging：商业化不是只改价格

适用：订阅、一次性付费、企业合同、用量计费、credit、seat、增值包、渠道售卖、试用或退款策略。
必须问：
- 定价目标：收入、转化、留存、扩容、利润、市场进入、渠道激励中优先保护什么。
- 价值度量：按 seat、usage、credit、功能、容量、服务等级还是成果计费，是否与用户获得价值一致。
- 套餐边界：免费/试用/基础/专业/企业版的功能、额度、权限、支持、审计和 SLA 差异。
- 付费墙：何时触发、是否阻断核心体验、是否给迁移/导出/降级路径。
- 收入护栏：退款、税务、发票、渠道抽成、销售折扣、合同义务、滥用、成本、留存、投诉。
- 价格实验：样本、分层、地区/渠道、公平性、老用户保护、销售干扰和长期留存观察。
输出：套餐矩阵、价值指标、付费墙规则、试用转化漏斗、价格实验方案、收入与体验护栏。
反例：只按竞品价格降 20%；应补价值度量、成本、渠道抽成、退款、折扣边界和长期留存。
转相邻技能：税务、合同、监管交 law；渠道与传播交 pma；计费实现交 api/db。

### 11. GTM 与商业化发布：launch 不只是上线

适用：新产品、重大功能、商业化、迁移、定价、企业版、平台策略变化、beta/GA 发布。
必须问：
- ICP：最先服务哪类客户/用户，排除哪些不适配人群。
- 定位：目标用户、场景、替代方案、差异化、证据、不可承诺边界。
- 发布分层：alpha、private beta、public beta、GA 的准入、退出、质量、支持和指标标准。
- Enablement：销售、CS、客服、运营、数据、法务、财务需要什么话术、FAQ、演示、迁移说明、风险边界。
- 迁移与弃用：老功能、老价格、历史数据、旧客户端、合同客户、通知节奏和回滚路径。
- 采用率目标：激活、feature adoption、教育引导、帮助中心、生命周期沟通、反馈分级。
输出：launch brief、ICP、定位、beta/GA 标准、enablement 清单、迁移计划、采用率目标、反馈闭环。
反例：产品全量上线后才让销售/客服知道；应提前准备话术、迁移、beta 反馈和升级路径。
转相邻技能：传播、渠道、活动、销售物料交 pma；外部资料交 rsch。

### 12. B2B/B2C 策略：同一功能不同链路

B2B 必查：采购人、管理员、终端用户、财务、安全、法务、CS、实施伙伴；POC、合同义务、权限、审计、SSO、数据导出、续约、扩容、迁移和支持 SLA。
B2C 必查：获客、激活、信任、留存、召回、社交传播、内容/推荐治理、滥用、误触、低质流量、客服规模和退款。
共同原则：最大客户需求不能直接等于产品方向；高 DAU/注册不能等于长期价值；必须分群看留存、投诉、成本和护栏。
输出：B2B 采购/使用链路、POC 成功标准、权限审计清单、续约扩容指标；B2C 激活留存漏斗、信任机制、内容治理与增长护栏。
反例：只满足最大客户定制需求；应评估可复用性、合同义务、权限审计、续约影响和平台化成本。

### 13. AI 产品与平台策略

AI 产品必须写：输入边界、输出可信度、幻觉容忍度、人工确认点、可撤销、可编辑、审计日志、内容安全、模型成本、延迟和降级。
Agent 产品必须写：autonomy level、权限范围、工具调用边界、失败回滚、人类确认点、任务结束条件。
平台策略必须查：iOS/Android/浏览器权限、订阅政策、支付限制、广告归因、Cookie、应用商店审核、开放 API 配额。
生成式体验：用户信任、可解释、可申诉、可撤销、可编辑通常比一次生成成功率更关键。

### 14. 隐私、合规与风险关闭

隐私合规：最小化采集、目的限制、授权、撤回、删除权、跨境、训练使用授权、儿童或敏感信息、数据保留期限、审计留痕。
现代风险治理：按场景检查 AI 合规、平台监管、未成年人、无障碍、公平性/歧视、模型/供应商风险；不要写未验证法规结论，只列需法务确认项。
风险关闭证据：不是写“已关注风险”，而是列出 owner、处理动作、验证证据、剩余风险、上线阈值和回滚条件。
需要法务、安全、财务、数据、研发判断时，产品经理只定义业务语义和验收口径，不替代专业结论。
反例：只写“用户同意协议”；应补训练授权、删除权、解释、审计、人工申诉和数据保留。

依赖管理：
- 内部依赖：接口、数据表、配置、权限、运营后台、审核流、客服知识库、BI 看板、告警。
- 外部依赖：第三方 API、平台审核、合同条款、渠道资源、支付/税务、供应商 SLA、法务结论。
- 每个依赖写 owner、交付物、截止时间、阻塞影响、替代方案和升级路径；未知依赖必须进入风险表。

### 15. stakeholder 对齐：共识要有责任和日志

识别：业务、运营、销售、客服、法务、财务、风控、研发、设计、测试、数据、运维、管理层、区域团队。
动作：对齐目标、收益、代价、责任人、决策权；记录争议点、tradeoff、未决问题、截止时间和升级路径。
机制：使用 RACI/DACI 明确负责、批准、协作、知情；保留 decision log；异步评审写清默认通过规则和冲突升级 SLA。
原则：不在评审会上第一次暴露高影响冲突；不把 stakeholder 共识当用户价值证据。

### 16. launch 与反馈闭环

发布前：灰度策略、监控指标、客服话术、运营物料、数据看板、告警阈值、回滚 owner、异常分级。
发布后：按小时/天/周观察核心指标、guardrail metrics、feedback、工单、异常日志；区分产品问题、数据口径问题、运营触达问题、技术故障。
运营化：准备 feature adoption 目标、教育引导、帮助中心、生命周期沟通、客服知识库、反馈分级和问题升级路径。
复盘：给出继续、回滚、限流、补实验、补设计、补研发、补运营或停止的证据。

release/rollback 必填：
- 发布策略：alpha/private beta/public beta/GA、灰度比例、准入人群、排除人群、扩量阈值。
- 监控：核心指标、护栏指标、错误率、延迟、成本、投诉、退款、人工介入、数据延迟。
- 回滚：触发阈值、决策人、执行人、用户通知、数据回滚或补偿、旧版本兼容、回滚后复盘。
- 不可逆变更：数据迁移、价格、合同、权限、删除、外部通知必须有演练或替代补偿方案。
- 关闭证据：版本、发布时间、监控截图/看板、异常记录、工单摘要、决策日志和后续 owner。

## 轻量产物选择

按任务选择最小充分产物，不为形式补文档：
- 一页纸 brief：目标、用户、证据、范围、指标、风险、决策请求，适合早期对齐。
- PRD：流程规则、状态权限、验收、埋点、依赖、发布，适合进入交付。
- RFC：多方案、取舍、影响面、迁移、风险，适合跨团队架构或平台决策。
- launch brief：ICP、定位、发布分层、enablement、迁移、指标，适合 GTM。
- experiment readout：假设、设计、结果、护栏、解释、决策，适合实验复盘。

## 高频坑/防遗漏

- 把“用户”混成一类：必须分新老用户、高低频、付费免费、管理员普通用户、移动端桌面端。
- 只写入口不写旅程：必须覆盖触发、进入、完成、失败、退出、重试、反馈。
- 用户研究偏差：重度用户、最大客户、内部员工、销售转述不能代表全量用户。
- 埋点缺口：事件名、触发时机、去重、时区、分母、样本、统计周期、看板 owner 不得缺。
- 依赖缺口：接口、数据、配置、审核、运营、法务、客服、第三方、发布窗口要逐项列。
- 非功能缺口：性能、可用性、安全、隐私、合规、可观测性、可运营性不能默认由研发补。
- 迁移兼容：历史数据、旧版本客户端、老入口、灰度用户、回滚后状态必须写。
- 定价订阅：试用、退款、折扣、续费、取消、税务、发票、渠道抽成不是边缘需求。
- GTM 缺口：ICP、定位、销售/CS/客服 enablement、迁移、beta/GA 标准不能上线后再补。
- B2B 缺口：POC、采购链路、权限审计、安全评审、合同义务、续约扩容影响必须显式。
- B2C 增长误判：短期转化上升可能伴随投诉、退款、误触、滥用、低质用户和留存下降。
- 国际化：时区、货币、语言长度、法规、文化语义、客服 SLA 会改变 scope。
- 风险未闭环：没有 owner、证据、阈值和回滚条件的风险，等于未关闭。

## 输出要求

按任务选择最小充分输出：
- 需求评估：结论、目标、用户、证据、scope、priority、risk、dependency、下一步。
- 用户研究：研究目标、样本分层、方法、提纲、证据、洞察、需验证项。
- PRD：背景目标、用户旅程、流程规则、范围边界、验收、指标、风险、发布。
- 用户故事：user story、acceptance criteria、edge case、out of scope。
- 优先级：候选项、评分依据、tradeoff、推荐排序、延期风险。
- 路线图：roadmap 阶段、目标、交付、指标、依赖、决策点。
- 指标/OKR：目标、指标树、leading/lagging、输入指标、护栏、复盘节奏。
- 实验方案：hypothesis、A/B test 设计、指标、样本、周期、护栏、决策规则。
- Pricing/Packaging：套餐矩阵、价值指标、付费墙、试用/退款、价格实验、收入护栏。
- GTM：ICP、定位、beta/GA、enablement、迁移、采用率目标、反馈闭环。
- 发布复盘：launch 结果、指标变化、feedback、问题归因、风险关闭证据、后续迭代。
结论格式：先给建议，再给证据，再给缺口；不确定必须标需验证。

PRD 最低验收输出：
- 每个需求项都有 user/job/story、scope、out of scope、AC、edge case、metrics、risk、dependency、release note。
- 每个模糊词都有定义，每个指标都有口径，每个风险都有 owner，每个依赖都有日期，每个取舍都有决策人。
- 对开发交付必须给需求对账表：需求项、状态、验收证据、未覆盖项、延期项、回滚/补偿方案。

## 约束

- 不把技术实现细节伪装成产品价值；实现方案交给对应工程技能补齐。
- 不承诺无证据的收益、周期、ROI、增长率。
- 不用单一竞品截图决定需求；竞品只能作为线索，不是理由。
- 不省略 out of scope；不写“后续优化”当范围管理。
- 不把 KPI 当 North Star Metric；不把短期点击率当长期价值。
- 不把项目上线当 KR；必须能解释用户或业务结果。
- 不把 A/B test 当所有问题的答案；不适用场景必须停止实验化包装。
- 不替代法务、安全、财务、数据、研发评审；发现边界问题必须转相邻技能。
- 不把“已评审”“已对齐”当风险关闭；必须有关闭证据。
- 不写未验证 URL、法规结论或平台政策结论；只标需确认项和影响范围。
- 不把“研发理解一下”当交接；必须有可验收的 AC、边界、埋点、风险和依赖。
- 不把“本期不做”藏在口头里；所有 non-goals 必须写入 PRD 或 brief。

## 高频 Bug 反例库

- 反例 1：需求澄清。错法：只记录“增加导出功能”。对法：写清谁导出、导出什么、为什么、频率、权限、脱敏和审计。根因：把功能名当用户任务。
- 反例 2：JTBD。错法：写“用户想要 AI 总结”。对法：写用户在何场景需要降低阅读成本、允许何种错误、如何校验。根因：把技术能力当需求。
- 反例 3：用户研究。错法：只访谈 3 个重度用户代表全量。对法：补样本分层、沉默用户、购买者/使用者分离和行为数据。根因：样本偏差未关闭。
- 反例 4：用户故事。错法：写“用户可以管理订单”。对法：补角色、场景、动作、收益和完成状态。根因：缺少角色分层和使用情境。
- 反例 5：PRD。错法：只有原型截图。对法：补状态、错误、权限、埋点、acceptance criteria、回滚。根因：把视觉稿当需求文档。
- 反例 6：MVP。错法：首版同时做推荐、报表、自动化和后台。对法：只保留验证核心假设的最短闭环。根因：把完整愿景误当首版范围。
- 反例 7：scope。错法：没写 out of scope，上线前默认包含历史迁移和移动端。对法：列明确不做项、后置项、触发条件。根因：害怕取舍导致范围失控。
- 反例 8：priority。错法：按最大客户声音排序。对法：用用户覆盖、收入影响、留存、风险、成本和证据强度排序。根因：把销售压力当全局价值。
- 反例 9：roadmap。错法：季度功能清单没有决策点。对法：每阶段绑定指标、依赖、暂停和转向条件。根因：把排期当战略。
- 反例 10：metrics。错法：只看点击率提升。对法：同时看转化、留存、误触、退款、投诉和成本。根因：缺少 guardrail metrics。
- 反例 11：OKR。错法：把项目按时上线写成 Key Result。对法：写用户/业务结果，并配 leading 与输入指标。根因：把产出当结果。
- 反例 12：North Star。错法：把注册数当 North Star Metric。对法：选择用户完成核心价值的持续指标。根因：把增长入口当产品价值。
- 反例 13：A/B test。错法：一次实验同时改入口、文案、价格和推荐。对法：控制变量，写样本、周期、护栏和终止条件。根因：把上线灰度误当实验。
- 反例 14：验收标准。错法：只写“功能正常可用”。对法：写输入、输出、状态、异常、权限、数据、通知、埋点。根因：没有从测试视角定义完成。
- 反例 15：埋点口径。错法：下游只收到“点击按钮”。对法：定义事件名、触发时机、去重、分母、时区、用户分群。根因：把埋点当研发细节。
- 反例 16：定价。错法：只按竞品价格降 20%。对法：补价值度量、成本、渠道抽成、退款、折扣和长期留存。根因：把价格当促销按钮。
- 反例 17：GTM。错法：产品全量上线后才让销售/客服知道。对法：提前准备 enablement、话术、迁移、beta 反馈。根因：把发布当研发完成。
- 反例 18：B2B。错法：只满足最大客户定制需求。对法：评估可复用性、合同义务、权限审计、续约影响。根因：把单客压力当产品策略。
- 反例 19：B2C 增长。错法：只追 DAU/注册。对法：补激活质量、留存、投诉、滥用和低质流量。根因：把流量当价值。
- 反例 20：AI 产品。错法：只写“接入大模型自动回复”。对法：补输入限制、人工兜底、审计、隐私、内容安全、成本和延迟。根因：忽略 AI 不确定性。
- 反例 21：隐私合规。错法：默认用户数据可用于训练。对法：确认授权、撤回、删除、跨境、最小化采集和审计留痕。根因：把数据可得当数据可用。
- 反例 22：平台策略。错法：上线后才发现订阅、支付或浏览器权限受限。对法：PRD 阶段列平台政策、审核、配额和降级方案。根因：忽略平台约束。
- 反例 23：需求冲突。错法：业务要转化、客服要少投诉、法务要降风险，各自评审。对法：统一 tradeoff 表和决策 owner。根因：没有显式冲突关闭。
- 反例 24：launch。错法：全量发布无监控、回滚 owner 和反馈入口。对法：灰度、告警、客服话术、回滚阈值、复盘时间齐全。根因：把发布当结束。
- 反例 25：problem framing。错法：把“提升管理效率”直接拆成后台页面。对法：先定义谁的哪类决策耗时、当前基线、目标阈值和不做范围。根因：未把愿望翻译成问题。
- 反例 26：需求防歧义。错法：写“支持批量处理”。对法：写批量上限、失败部分成功、重试、撤销、权限、审计和通知。根因：模糊动词未展开。
- 反例 27：RICE。错法：Impact 和 Confidence 都凭感觉打高分。对法：Reach 写分群和时间窗，Confidence 写证据等级，Effort 纳入跨团队成本。根因：评分表伪装成客观结论。
- 反例 28：验收。错法：AC 写“用户能成功提交”。对法：补必填、格式、重复提交、网络中断、权限失效、提交后状态和埋点。根因：只验快乐路径。
- 反例 29：依赖。错法：默认数据团队会补看板。对法：写看板 owner、口径、交付时间、验收方式和延迟影响。根因：把跨团队工作当空气。
- 反例 30：回滚。错法：只写“异常则回滚”。对法：写触发阈值、谁决策、谁执行、数据如何处理、用户如何通知。根因：没有把回滚当产品体验。
- 反例 31：stakeholder。错法：会议口头同意后继续推进。对法：记录 decision log、反对意见、默认通过规则和升级路径。根因：共识没有可追溯证据。
- 反例 32：non-goals。错法：本期没做移动端但 PRD 未写。对法：明确端、角色、历史数据、迁移和运营动作不做，并写后续触发条件。根因：范围边界只存在脑中。

## 2024-2026 新坑速查

- AI 功能：输入边界、幻觉容忍度、人工兜底、审计日志、隐私和内容安全必须前置。
- 生成式体验：用户信任、可解释、可撤销、可编辑、可申诉通常比一次生成成功率更关键。
- Agent 产品：autonomy level、权限、失败回滚、人类确认点和任务边界必须显式。
- 数据合规：隐私、跨境、最小化采集、训练授权、删除权、数据保留和审计留痕提前确认。
- 成本指标：AI token、推理延迟、第三方 API、存储、人工审核进入 KPI 或护栏。
- 平台变化：系统权限、订阅政策、广告归因和 Cookie 变化会影响指标解释。
- 增长实验：隐私限制、样本污染和渠道黑盒导致归因更难，保留 cohort、holdout 和反指标。
- B2B 产品：采购人、管理员、终端用户、财务、安全团队目标不同，persona 不能只写一种。
- B2C 产品：信任、社交传播、内容治理、滥用、退款和客服规模会改变增长策略。
- 多端一致性：移动端、Web、桌面端、插件、开放 API 的状态、权限和埋点容易不一致。
- 订阅定价：试用、退款、折扣、续费、取消、税务、发票、渠道抽成进入 scope 判断。
- 国际化：时区、货币、语言长度、法规、文化语义、客服 SLA 会影响验收。
- 可靠性：核心链路需要降级、重试、幂等、状态恢复和用户可见反馈。
- 无障碍与公平性：涉及公共服务、招聘、教育、金融、未成年人或自动化决策时必须前置检查。

## 与相邻技能的边界

- 产品经理/product-manager（pm）：负责问题定义、用户价值、JTBD、PRD、scope、priority、roadmap、指标、实验、商业化、发布和反馈闭环。
- 研究调研/research（rsch）：负责外部资料、行业事实、竞品证据、用户研究材料；产品经理负责把证据转成决策。
- 品牌与视觉方向/brand-visual-direction（b）：负责视觉方向、品牌体验和设计判断；不替代产品目标、用户价值和范围取舍。
- UI 架构/ui-architecture（a）：负责信息架构、交互结构、复杂状态体验；产品经理提供用户旅程、任务优先级和验收边界。
- API 工程/api-engineering（api）：负责 API 路由、协议、鉴权、错误码、响应结构；产品经理给业务语义、状态流和验收口径。
- 法务合规/legal-counsel（law）：负责隐私、合同、监管、知识产权和合规判断；产品经理负责提出数据用途、用户影响和风险场景。
- 产品营销/product-marketing（pma）：负责定位传播、转化文案、渠道策略、销售物料和活动；产品经理负责产品定位、目标用户、价值主张和留存证据。
- 测试验证/test-engineering（tst）：负责测试矩阵、复现、回归和证据；产品经理给 acceptance criteria、关键场景、边界和用户影响。
- 代码审计/code-audit（aud）：负责改动后的影响面、安全质量和证据收口；产品经理给需求对账清单、风险关闭证据和上线边界。
