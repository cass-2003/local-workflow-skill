# 测试验证

> 首次自称：测试验证（test-engineering，兼容 slug: tst）。
> 命名口径：frontmatter name 使用 manifest canonical name `test-engineering`；目录名和 URL 继续兼容 slug `tst`；自检不得要求 name 等于短 slug。

> 定位：把“测一下”变成可复核质量闭环：风险 → 入口 → 环境 → 证据。测试结论只证明已覆盖场景；未复现、未执行、无产物的链路必须标“无法验证”。
> 铁律：先读需求/改动/调用方，再建风险矩阵；先制造红灯，再证明绿灯；先保留失败产物，再修 flaky；未跑不报、无证据不下结论。

## 快速总则：风险 / 入口 / 环境 / 证据

1. 风险：按用户影响、金额/权限/数据损坏、并发/幂等、历史缺陷、发布回滚成本排序；高风险先深测，低风险不抢主链路时间。
2. 入口：列全 UI、API、SDK、CLI、定时任务、队列消费、旧客户端、Webhook、后台脚本和灰度入口；只测新入口不合格。
3. 环境：声明本地/CI/预发/生产冒烟、浏览器/设备/时区/语言、依赖版本、容器镜像、feature flag、账号角色和测试数据状态。
4. 证据：每个结论绑定命令、用例名、CI job、commit、日志、trace/span、截图/video、网络样本、报告路径或产物版本。
5. 分层：unit 守规则，integration 守真实依赖，contract 守跨服务协议，e2e/冒烟守关键旅程，property/mutation 守规则强度，visual/a11y/perf/security/eval 按风险启用。
6. Mock：只隔离外部不稳定性；金额、权限、状态机、幂等、事务、核心算法必须跑真实实现或可信 fake，不能 mock 掉被测风险。
7. 结论四档：已覆盖、部分覆盖、未覆盖、无法验证；每档写原因、证据、剩余风险和下一步。
8. 裁剪：先保高风险、高频入口、历史回归和阻断门禁；低风险专项可延后但必须标剩余风险。

## 测试选择决策树

- 纯规则/算法/格式化：必测 unit、边界、错误语义；金额/权限/状态机加 property 或 mutation 专项；无需 e2e，除非影响关键旅程。
- API/契约/错误码：必测 unit + integration + contract；跨消费者、旧客户端、Webhook 加兼容回归；关键旅程再补 e2e。
- DB/缓存/队列/事务：必测 integration、迁移/回滚、并发/幂等、fixture 隔离；高流量或慢查询加 perf。
- UI/路由/表单：必测 component 或 e2e 主链路、失败态、空态、加载态、a11y；稳定关键页面加 visual；纯样式低风险可截图核验。
- 权限/认证/上传/公开入口：必测负向权限、安全样本、审计日志；高危链路联动 wsec，不能只跑 happy path。
- 异步/定时任务/事件：必测状态流转、重试、乱序、重复消费、死信、观测 trace；必要时 contract 验事件 schema。
- 发布/配置/灰度：必测 release smoke、健康检查、旧入口、feature flag、回滚路径和监控告警；本地绿不能替代环境门禁。
- AI/LLM/RAG/Agent：必测 eval 集、拒答/安全、工具权限、模型版本、失败样本入库；非确定输出用评分阈值和抽样复核。
- 性能/安全/visual/mutation/property：不是默认全量；命中专项触发器时启用，其他场景记录不启用理由。

## CI gate 分级规则

- PR 快速门：lint/typecheck、变更相关 unit、关键 integration/contract、最小 e2e smoke；阻断编译失败、关键断言失败、契约破坏、测试数据污染。
- 主干完整门：全量 unit/integration/contract、关键 e2e、coverage 变更口径、flaky 统计；阻断新增 flaky、关键覆盖缺口和主链路失败。
- Nightly 深度门：跨浏览器/设备、visual、a11y、perf baseline、mutation/property 核心模块、安全扫描和长链路；失败可软阻断但必须有 owner。
- Release 门：预发冒烟、迁移/回滚、灰度入口、监控/告警、生产只读核验；阻断健康检查、登录、核心交易/提交、权限负向失败。
- Hotfix 门：原 bug 红灯/绿灯、最小回归、受影响入口、回滚验证；允许裁剪慢专项，但必须写未覆盖风险。
- Quarantine：只允许止血，不得让关键主链路长期降级；必须有 ticket、owner、过期时间、恢复条件和影响面。
- 降级条件：外部依赖故障、环境资源故障或非阻断 flaky 可降级；业务断言失败、契约破坏、权限/金额/数据损坏不可降级。

## 变更驱动测试执行流程

1. 输入锁定：先读需求、diff、调用方/消费方、旧入口、运行环境、历史缺陷和发布目标；缺输入时先列未知，不直接给测试结论。
2. 影响映射：按“改动文件/接口/字段/配置/依赖/迁移/队列/缓存 key → 调用链 → 用户入口 → 测试层级 → 证据”建表。
3. 风险评分：金额/权限/隐私/数据损坏/不可逆副作用/高频入口/历史事故/发布回滚成本任一命中高风险，至少 integration/contract/e2e 或专项证据之一。
4. 测试选择：优先最小充分层级；可 unit 证明的规则不强推 e2e，可 contract/integration 发现的契约问题不拖到发布冒烟。
5. 组合裁剪：用风险权重、历史缺陷、高频入口、pairwise/等价类裁剪组合爆炸；被裁剪项必须写剩余风险。
6. 红绿闭环：bug 修复必须先有失败基线；无法复现时只能写“无法验证原问题”，不能写“已修复”。
7. 证据门槛：核心链路没有 CI/job/日志/产物/样本之一，或测试 mock 掉风险点，结论自动降级为部分覆盖/无法验证。
8. 失败诊断：失败先分编译、环境、数据、断言、依赖、时序、资源、权限、契约；未分类前不靠重跑包装通过。
9. 资源控制：并行测试必须隔离账号、端口、DB/schema、队列 topic、临时目录、缓存 key；设置超时和清理，避免 CPU/内存/磁盘耗尽。
10. 交接审计：输出给 aud 的证据包必须包含覆盖项、未覆盖风险、mock 边界、环境差异、失败/跳过/quarantine 和需要复核的调用链。

## 证据等级与结论降级

- S 级：原路径红绿复验、CI integration/contract/e2e、预发/真实依赖证据、trace/log/产物可复跑。
- A 级：相关 unit/integration、contract/schema diff、组件/e2e smoke、带断言和产物的手动验证。
- B 级：lint/typecheck、静态扫描、局部 unit、截图或接口样本；只能支撑低风险或作为辅助证据。
- C 级：间接全量通过、无产物手测、只看覆盖率、只断“不报错”；不得单独支撑核心链路通过。
- 降级规则：skip/only、缓存命中旧结果、flaky 未定因、mock 核心风险、环境不一致、缺旧入口/负向/并发/迁移证据时，必须降级并列剩余风险。

## 单技能工程门禁

- 需求门禁：先把用户需求拆成可验证条目、禁止项、旧入口、角色、数据状态和回滚点；没拆条不能说“覆盖了需求”。
- 风险门禁：金额、权限、隐私、状态机、迁移、并发、旧端、回滚、外部消费者任一命中高风险，必须给真实依赖或契约/e2e/专项证据。
- 入口门禁：新入口、旧 API、脚本、定时任务、队列、SDK、灰度入口、移动旧版本至少逐项标已测/未测/不适用。
- 环境门禁：本地通过只能说明本地；CI、预发、生产冒烟、浏览器/设备/时区/feature flag 不一致时必须降级。
- 证据门禁：测试结论必须绑定命令、用例名、job、commit、日志、trace、截图/video、样本或报告路径；无产物手测只算 C 级。
- 产物门禁：失败产物先保存，成功产物要可复跑；看不到路径、job 或样本时不能把“跑了”当证据。
- 红绿门禁：bug 修复、字段错、迁移错、权限错必须先有原路径红灯；无法复现就写无法验证原问题。
- 回滚门禁：发布、迁移、配置、灰度、旧端兼容必须验证前滚/回滚或声明不可回滚边界。

## 硬禁止与低级错拦截

- 禁止只跑全量测试就说需求过；全量通过如果没有变更映射和风险矩阵，只能作为辅助信号。
- 禁止只测 happy path、管理员路径、新入口或本地环境。
- 禁止 mock 掉金额、权限、状态机、幂等、事务、迁移、缓存一致性和外部消费者契约。
- 禁止忽略 `skip`、`only`、quarantine、缓存命中旧结果、flaky 重跑通过和静默软失败。
- 禁止把 lint/typecheck/覆盖率/截图/接口 200 当成核心链路验收。
- 禁止没有断言、弱断言、只断“不报错”、只断 mock 被调用、只看日志没有状态校验。
- 禁止迁移只测空库；必须覆盖旧数据、回滚/前滚、抽样校验、锁等待或缩放说明。
- 禁止权限只测有权限；必须覆盖未登录、无权限、越权对象、跨租户和旧 token/session。
- 禁止 e2e 无 trace/video/network 产物；禁止手测无步骤、账号、数据、环境和截图/日志。
- 禁止把测试通过包装成可上线；上线还要 release/observability/audit 证据收口。

## 标准证据包模板

- 变更/需求：条目、风险等级、入口、旧入口、禁止项。
- 环境：本地/CI/预发/生产、commit、依赖/镜像版本、feature flag、账号角色、数据状态。
- 命令与用例：命令、测试文件/用例名、CI job/编号、shard、重跑方式。
- 产物：日志、trace/span、截图、video、network 样本、契约报告、覆盖率/变更覆盖、性能或安全报告路径。
- 结论：已覆盖/部分覆盖/未覆盖/无法验证；逐项绑定证据，不用总括句替代。
- 剩余风险：阻断项、可延后项、需人工复核项、需联动技能项和下一步。

## 场景执行卡

### 1. 需求拆条与回归矩阵

- 适用：新需求、改需求、补测试、验收前质量门禁。
- 输入：需求原文、验收条件、禁止项、diff/接口/配置、调用方、旧入口、用户角色、数据状态、兼容范围。
- 动作：逐条列功能点、异常路径、权限路径、状态流转、兼容入口、降级/回滚路径；形成回归矩阵。
- 最小矩阵列：需求条目、风险等级、入口、角色、初始状态、输入、操作、预期、测试层级、环境、证据、结论。
- 必测维度：golden path、edge case、异常、权限、并发/幂等、回归、兼容、观测信号。
- 裁剪规则：先测高风险主链路、历史缺陷、外部消费者和不可逆写入；低频低风险组合可标未覆盖并给理由。
- 失败兜底：需求不清先列假设和待确认项；不能用猜测补验收标准。

### 2. 原 Bug 复现与修复验证

- 适用：线上 bug、历史回归、用户投诉、偶发失败。
- 输入：原始报错、用户路径、账号/角色、旧数据、时间窗口、设备/浏览器、日志/trace、版本/commit。
- 动作：先制造红灯；复现失败时按数据、环境、时序、版本、权限逐项排除；修复后跑原路径、相邻路径、旧入口和负向路径。
- 证据：修复前失败断言/截图/日志，修复后同路径通过证据，新增回归用例名，若不能复现则列证据缺口。
- 禁止：没有红灯基线就写“已修复”。

### 3. Unit：规则、边界、错误语义

- 适用：纯函数、业务规则、金额/权限判断、状态机、解析器、策略选择。
- 动作：表驱动覆盖等价类、边界值、非法输入、舍入/时区/空值、异常分支、错误码；必要时加入 property-based testing 验不变量。
- 工具关注：Vitest/Jest fake timers 与 ESM mock、Pytest fixture/monkeypatch、JUnit 参数化和并发执行。
- 断言：断输出、状态变化、异常类型、错误码、不可变条件；禁止空断言和只断“不报错”。
- 质量口径：覆盖率只看关键路径、分支和变更覆盖，不用无意义用例凑百分比。

### 4. Integration：真实依赖与 fixture 隔离

- 适用：DB、缓存、队列、文件存储、事务、ORM、第三方 SDK 适配层。
- 动作：优先用 testcontainers、临时库/schema、事务回滚、唯一前缀、fixture factory 或容器重建隔离数据；验证提交、回滚、唯一约束、并发冲突、超时重试。
- Testcontainers 最小准则：镜像 pin 到明确版本，声明架构；配置 healthcheck/ready 探测、网络别名、日志导出、资源清理和失败时容器日志保留。
- Mock 边界：外部不可控 SaaS 可 fake server/recording；本项目关键依赖交互尽量真实跑。
- 证据：镜像和服务版本、迁移版本、fixture 名称、清理策略、失败日志、容器 healthcheck。
- 高频风险：共享测试数据污染、依赖执行顺序、CI 与本地服务版本不同、ARM/AMD 差异。

### 5. API / Contract / 状态机

- 适用：REST、GraphQL、gRPC、OpenAPI、跨服务事件、支付回调、订单/任务状态。
- 动作：覆盖 2xx/4xx/5xx、认证授权、schema、错误码、幂等键、重放、乱序、超时、依赖失败、向后兼容。
- 契约测试：consumer-driven contract 或 schema diff 必须绑定消费者版本；contract 通过不等于业务旅程通过。
- 状态机：列合法跳转、非法跳转、重复提交、并发竞争、旧数据迁移态。
- 证据：请求/响应样本、契约验证报告、trace/span id、幂等记录、消费者版本。

### 6. E2E / Component / Visual / A11y / Usability

- 适用：关键用户旅程、路由守卫、表单、列表、弹窗、跨浏览器、多端 UI。
- 动作：Playwright/Cypress 覆盖主链路、失败态、空态、加载态、返回态、权限态；组件测试覆盖复杂交互；visual regression 只覆盖稳定关键页面；a11y 至少检查键盘、焦点、语义、对比度。
- 可用性启发式：表单错误可恢复、错误文案可理解、加载骨架不误导、撤销/重试可达、移动端键盘不遮挡主按钮、空态有下一步。
- 异步等待：等 DOM 状态、网络别名、URL、事件、队列或业务状态；禁止固定 sleep。
- 兼容：声明浏览器、视口、设备、语言、时区、权限弹窗、第三方 cookie、Storage Partitioning。
- 证据：trace viewer、video、screenshot、network log、视觉 diff 阈值和基线版本、a11y 报告。

### 7. Perf / Security / Compatibility 专项

- 性能触发：高 QPS、冷启动、缓存/索引/查询改动、批处理、队列堆积、SLO 路径或资源成本敏感时必须专项；需基线、环境、样本量和噪声说明。
- 安全触发：登录、权限、金额、上传、公开入口、回调、跳转、富文本、文件解析、外部 URL、敏感日志时必须负向样本；高危链路联动安全技能。
- 兼容触发：浏览器/移动端设备云、系统版本、时区/语言、弱网、离线/恢复、旧客户端、嵌入 WebView。
- 证据：压测报告、安全样本、设备云报告、浏览器矩阵、OpenTelemetry trace。

### 8. Flaky 排查与隔离

- 适用：偶发失败、CI-only 失败、重跑通过、耗时异常、shard 相关失败。
- 动作：先保留失败产物；分类为时序、数据污染、外部依赖、资源竞争、随机性、环境漂移、并发 shard；用 seed、独立数据、业务等待、锁版本、trace/video、二分并发度定位。
- 禁止：直接 retries/quarantine 掩盖；临时隔离必须有 ticket、owner、过期时间、恢复条件。
- 结论：标根因、复现概率、隔离策略、恢复条件和影响面。

### 9. CI shard、覆盖率与质量门禁

- 适用：合并前、发布前、热修、回滚验证、测试套件变慢。
- 动作：记录 job URL/编号、commit、lockfile、缓存 key、包管理器、shard 分配、失败产物；本地绿不能替代 CI 绿。
- Shard：用稳定分片键或历史耗时均衡；每片独立 fixture、端口、账号、临时目录；失败要能单独复跑。
- Coverage / coverage：同时看行、分支、变更覆盖、关键断言、mutation score；覆盖率下降不必然阻断，关键风险未覆盖必须阻断。
- 门禁：按阻断/可延后/无法验证分类，不把测试通过包装成可上线。

### 10. 发布冒烟与回滚验证

- 适用：预发/生产发布、灰度、热修、回滚。
- 动作：跑最小可信集：健康检查、登录、核心提交/交易、权限负向、旧入口、迁移/回滚冒烟、告警和日志。
- 环境：声明 region、tenant、feature flag、配置版本、数据隔离方式、只读/写入限制。
- 证据：冒烟命令、监控状态、trace/span、日志查询、告警状态、回滚验证结果。

### 11. LLM / AI / Eval 功能测试

- 适用：Prompt、RAG、Agent、分类/抽取、生成式输出、工具调用。
- 动作：固定评测集、golden samples、adversarial set、拒答/安全样本、工具调用断言、事实性校验、回归评分；区分确定性断言与人工/模型评审。
- 漂移控制：记录模型版本、参数、prompt 版本、检索索引版本、judge 版本；监控平均分、分位数、关键样本和失败类型，而非只看总均分。
- Judge 偏差：抽样人工复核，标注争议样本；不能让同一模型同时生成和无约束裁判高风险结论。
- 样本治理：失败样本入库前去重、脱敏、标标签；防评测集泄漏、训练污染和过拟合阈值。
- 风险：非确定性、模型版本漂移、prompt injection、RAG 召回污染、评测集泄漏、越权工具调用。
- 证据：模型版本、温度/参数、评测集版本、评分标准、人工复核样本、失败样本。

### 12. Mobile / WebView / 离线同步

- 适用：iOS/Android 原生、混合 WebView、PWA、深链、通知、后台恢复、离线同步。
- 动作：覆盖真机与模拟器差异、权限弹窗、键盘遮挡、安全区、弱网/断网/恢复、后台切回、深链参数、系统 WebView、通知权限和厂商 ROM。
- 离线同步：验证冲突解决、重复提交、队列重放、幂等、失败重试和本地缓存清理。
- 证据：设备/系统/WebView 版本、网络条件、视频/截图、日志、崩溃报告、同步队列状态。

### 13. Compliance / Data Lifecycle 测试

- 适用：PII、支付/健康/儿童/位置、AI prompt/文件、租户数据、导出/删除、第三方共享和审计日志。
- 动作：验证采集最小化、同意/撤回、脱敏、加密、访问控制、留存/删除、导出、租户隔离、日志/trace/analytics 不泄露。
- 数据：用脱敏 fixture 或合成数据；禁止把真实敏感数据写入测试日志、截图、报告和外部 SaaS。
- 证据：字段分类、请求/响应样本、日志样本、删除/导出结果、审计记录、未覆盖合规风险。

### 14. Resilience / Capacity / Soak 测试

- 适用：高 QPS、批处理、队列积压、连接池、缓存、冷启动、资源成本敏感、SLO 路径和长期运行服务。
- 动作：建立基线，覆盖 p95/p99、吞吐、CPU/内存/IO、连接池、队列 lag、错误率、降级/限流/熔断、长稳 soak/endurance。
- 裁剪：非 SLO 路径可不压测，但必须说明数据规模假设和不启用理由。
- 证据：压测配置、样本量、环境、基线对比、资源曲线、瓶颈定位和回滚阈值。

### 15. Cloud-native Release / Chaos 测试

- 适用：K8s、容器、serverless、edge、Helm/Kustomize、CI/CD、灰度、回滚和多区域发布。
- 动作：验证 liveness/readiness/startup probe、resources、HPA、PDB、滚动更新、金丝雀指标、Pod 重启、节点/依赖故障、配置漂移和 secret 注入。
- 冒烟：健康检查、登录/核心交易、权限负向、旧入口、观测告警、回滚路径必须绑定发布环境。
- 证据：部署版本、配置 diff、事件/日志、监控截图或查询、故障注入结果、回滚验证。

### 16. Mobile Compatibility Matrix 测试

- 适用：iOS/Android、Flutter/RN、WebView、小程序、PWA、深链、推送、离线同步和原生能力。
- 矩阵：系统版本、设备档位、低端机、平板/折叠屏、厂商 ROM、WebView 版本、网络条件、权限弹窗、升级迁移和旧 App 版本。
- 原生能力：相机、相册、定位、蓝牙、通知、后台任务、键盘、安全区、低电量和存储权限按风险启用。
- 证据：设备/系统/App/WebView 版本、视频/截图、崩溃日志、网络条件、离线队列状态和升级路径。

### 17. DB Migration at Scale 测试

- 适用：大表迁移、回填、索引、NOT NULL/default、唯一约束、枚举、数据修复、expand-contract 和不可逆变更。
- 动作：验证 dry-run、锁表/耗时、批量大小、断点续跑、幂等、旧代码兼容、新旧版本共存、抽样校验、回滚或前滚策略。
- 数据：用接近生产分布的数据量级或明确缩放模型；小样本通过不能证明大表安全。
- 证据：迁移版本、执行计划、耗时/锁等待、抽样 SQL、失败恢复步骤、回滚边界。

### 18. 专项触发器与适用边界

- Property：适合纯规则、不变量、解析器、金额/舍入、状态机；不适合依赖复杂外部状态且难定义不变量的流程。
- Mutation：适合核心规则和高风险模块专项，用来发现弱断言；不作为默认全量 CI 门禁，不用来单纯追覆盖率。
- Visual：适合稳定关键页面和设计系统；必须冻结时间、动画、字体、随机数据，阈值只覆盖渲染噪声。
- Perf：适合 SLO、高流量、资源成本、缓存/索引/批处理改动；必须有基线和同等环境。
- Security：适合认证授权、输入解析、上传、公开入口、外部 URL、敏感数据；高危结果需安全技能复核。
- Testcontainers：适合真实依赖集成；轻量纯规则测试不应因此变慢。

### 19. 字段契约、CRUD 与红绿回归

- 适用：新增/修改字段、表单、DTO、OpenAPI/schema、前后端字段映射、列表详情编辑删除、导入导出、序列化、老数据兼容和错误类 bug。
- 字段契约矩阵：逐字段列名称、类型、必填/可空、默认值、枚举、长度/精度、格式、单位、时区、脱敏、权限可见性、排序/筛选/搜索语义、前端展示、API 入参、API 出参、DB 持久化、事件/队列 payload、导入导出和旧客户端兼容。
- contract/schema 门禁：字段变化必须有 OpenAPI/JSON Schema/GraphQL/gRPC/protobuf/事件 schema diff 或消费者契约报告；只看 TypeScript interface、DTO 文件或页面字段不算契约证据。
- integration 门禁：新增、编辑、删除至少验证一次真实持久化或可信集成路径，覆盖保存后再读、列表刷新、详情回显、审计/日志/事件副作用和事务回滚。
- e2e 门禁：用户可见字段必须覆盖创建后展示、编辑后回显、删除后消失或软删除状态、刷新/返回/重新登录后的状态一致；后台字段也要有 API/集成证据替代。
- CRUD 回归矩阵：Create 验默认值、必填、非法输入、重复提交；Read 验列表/详情/权限可见性、排序筛选搜索；Update 验单字段修改、部分更新、空值清除、并发覆盖、旧值保留；Delete 验软删/硬删、二次删除、恢复、关联数据、权限和审计。
- 编辑场景：必须测只改一个字段、多个字段同时改、把字段改为空/0/false、枚举切换、超长文本、非法格式、保存失败回滚、重复点击保存、并发编辑冲突和旧页面缓存刷新。
- 删除场景：必须测无权限删除、有关联数据删除、删除后列表/详情/API/搜索/统计同步、重复删除幂等、软删除可见性、误删恢复或回滚路径；只点一次删除按钮不算完整回归。
- 字段错误类 bug 红绿回归：字段名错、大小写错、类型错、空值错、默认值错、枚举错、精度/时区错、前后端映射错、序列化丢字段、旧数据缺字段、删除后残留引用，必须先制造红灯用例或样本，再用同路径绿灯证明修复。
- 红灯证据：失败断言、请求/响应样本、schema diff、DB 记录、截图/video、日志或 trace 至少一种；无法制造红灯时只能写“无法验证原字段错误”，不能写“已修复”。
- 绿灯证据：同一字段、同一入口、同一角色、同一数据状态复跑通过，并补相邻 CRUD 回归；只跑新增 happy path 不能证明编辑/删除未回归。
- 反例：新增字段只测创建成功，没测编辑回显和删除残留；schema 通过但消费者仍读旧字段；前端显示正确但 API 出参类型错；软删除列表消失但详情接口仍可读；错误字段 bug 没有红灯就直接报修复完成。

## 高频坑 / 防遗漏

### 高频坑

1. 需求没拆条，只按改动文件补测试。
2. 只测 happy path，不测边界、异常、权限、并发。
3. 修 bug 未复现原问题，缺红灯基线。
4. Mock 掉核心业务逻辑，测试变成测 mock。
5. 固定 sleep 等异步，导致慢且 flaky。
6. 测试数据共享污染，依赖执行顺序。
7. 只看覆盖率百分比，不看关键分支、变更覆盖和断言质量。
8. 只测新入口，不测旧入口、脚本、定时任务、旧客户端。
9. E2E 只在本地跑，不收 CI trace/video。
10. flaky 直接重试或 quarantine，不定位根因。
11. 契约测试只验 schema，不验业务错误码和兼容消费者。
12. 视觉回归基线漂移，阈值过宽导致真问题被吞。
13. CI cache、lockfile、shard 或包管理器漂移，本地与 CI 依赖不一致。
14. 浏览器隐私变化导致登录、第三方 cookie、权限弹窗在真机失败。
15. PR 只跑 unit，把 contract/e2e 留到人工发布前，消费者兼容问题进主干。
16. AI 只看平均分，未保存失败样本、模型版本和 judge 版本，升级后无法解释回归。
17. 移动端只测模拟器，真机键盘、安全区或权限弹窗阻断流程。
18. Testcontainers 用 latest 且无 healthcheck，CI 偶发连到未就绪服务。
19. 只用小样本迁移测试，大表回填上线后锁表或超时。
20. 只做短压测不做 soak，内存泄漏、连接泄漏和队列积压发布后才暴露。
21. 云原生只测服务启动，不测探针、滚动更新、Pod 重启和配置漂移。
22. 合规数据只测功能可用，不测脱敏、删除、导出、审计和日志泄露。

### 防遗漏清单

- 风险：金额、权限、数据损坏、并发、回滚成本是否置顶？
- 入口：UI/API/SDK/CLI/任务/队列/旧客户端/灰度入口是否列全？
- 环境：CI、本地、预发、浏览器、设备、时区、依赖版本和 feature flag 是否声明？
- 决策：是否按变更类型选择最小充分测试层级，并说明未启用专项的理由？
- Gate：PR/main/nightly/release/hotfix 跑什么、阻断什么、降级条件是否清楚？
- 复现：原 bug 是否有红灯证据？修复后是否同路径绿灯？
- 分层：unit/integration/e2e/contract/property/mutation/visual/a11y/perf/security/eval 是否按风险取舍？
- Mock：外部依赖与核心逻辑边界是否清楚？
- 数据：fixture/seed 是否独立创建和清理？并发 shard 是否安全？
- 异步：是否等待业务信号而不是 sleep？超时和失败产物是否可诊断？
- 覆盖率：是否说明行/分支/变更覆盖率、mutation/关键缺口？
- CI：是否有命令、job、commit、产物、trace/video/screenshot？
- AI/移动：是否记录模型/评测集版本、人工复核，或真机/WebView/权限/弱网证据？

## 输出要求

测试工程输出必须极简但可复核，至少包含：

1. 需求/改动条目：逐条列验证目标、禁止项、旧入口和风险点。
2. 风险排序：高/中/低风险及原因，说明为什么先测这些。
3. 测试选择：按变更类型列必测层级、可选专项、不启用理由。
4. 影响映射：改动对象→调用链→用户入口→测试层级→证据，标出未命中和兜底策略。
5. 证据等级：标 S/A/B/C 级，说明是否因 skip/only、缓存、flaky、mock、环境差异降级。
6. 入口矩阵：UI、API、SDK、任务、队列、脚本、旧客户端、灰度入口覆盖情况。
7. 环境矩阵：本地/CI/预发/生产冒烟、浏览器/设备、依赖版本、feature flag、账号角色、数据状态。
8. CI gate：PR/main/nightly/release/hotfix 已跑项、阻断项、软失败项、quarantine 状态。
9. 场景矩阵：正常、边界、异常、权限、并发/幂等、回归、兼容；每项标自动化/手测/无法验证。
10. 原 Bug 复现：若是修复任务，必须列复现输入、步骤、失败证据、修复后证据。
11. 测试分层：unit/integration/e2e/contract/property/mutation/visual/a11y/perf/security/eval 各取舍理由。
12. Mock 与真实依赖：哪些 mock、哪些真实跑、为什么。
13. 测试数据：fixture/seed/隔离/清理策略和 shard 并发安全性。
14. Flaky 证据：失败产物、根因分类、重试/quarantine 原因和恢复条件。
15. 覆盖率口径：行/分支/变更覆盖率、mutation score、关键未覆盖分支；不得只报百分比。
16. CI 证据：命令、job/commit、日志/trace/video/screenshot、响应样本或报告路径。
17. 专项证据：合规数据、资源稳定性、云原生发布、移动设备矩阵、大规模迁移命中时列证据或不启用理由。
18. AI/移动专项：评测集/模型/judge 版本、失败样本和人工复核；设备/WebView/权限/弱网证据。
19. 覆盖结论：已覆盖/部分覆盖/未覆盖/无法验证，绑定证据。
20. 剩余风险与下一步：阻断项、可延后项、需联动技能。

## 约束

- 不追求 100% 覆盖率；优先关键路径、风险分支和历史缺陷。
- 不引入新测试框架，除非项目无同类能力且用户确认；默认遵循现有 Vitest/Jest/Pytest/JUnit/Playwright/Cypress 约定。
- 不用固定 sleep；必须等待可观测业务信号并设置合理超时。
- 不写空断言、弱断言、只断快照的核心业务测试。
- 不 mock 被测核心逻辑、金额、权限、状态机、幂等、事务；外部依赖 mock/fake 必须说明边界。
- 不共享可变测试数据；测试必须可单跑、可并发、可重复。
- 不把本地通过包装成 CI 通过；不把测试通过包装成可上线。
- Flaky 可临时隔离但必须留追踪、根因假设、恢复条件和负责人。
- 涉及 API/DB/发布/安全/前端实现问题时，按边界联动对应技能，最后由 aud 收口。
- 若证据不足，结论写“无法验证/部分覆盖”，不要补脑。

## 高频 Bug 反例库

- 反例 1：只测 happy path
  - 错：只验证成功提交，失败、取消、无权限、重复请求都没覆盖。
  - 对：围绕需求列 golden path、edge case、异常、权限、并发和回归场景。
  - 根因：一次成功不能证明风险被覆盖。
- 反例 2：未复现原 bug
  - 错：没用原始数据、版本和路径复现，就直接写修复测试。
  - 对：先制造红灯，再用同路径验证绿灯，并补相邻回归。
  - 根因：没有失败基线就不知道修复是否命中。
- 反例 3：mock 核心逻辑
  - 错：把价格计算、权限判断或状态机 mock 掉，断言 mock 被调用。
  - 对：核心规则跑真实实现，只 mock 外部服务、时间、随机数。
  - 根因：测试绕过风险就失真。
- 反例 4：固定 sleep
  - 错：异步测试 sleep 2 秒，机器慢就失败，机器快就浪费。
  - 对：等待 DOM、事件、队列消息、状态字段或 trace，并设置超时。
  - 根因：时间流逝不是业务完成信号。
- 反例 5：共享 fixture 污染
  - 错：多个测试共用 admin/test_order，顺序变化或 shard 并发就失败。
  - 对：每个用例独立 seed、唯一前缀、事务回滚或容器重建。
  - 根因：数据隔离失败会制造伪回归。
- 反例 6：只看覆盖率
  - 错：行覆盖率 90% 就宣布测试充分。
  - 对：同时看分支、变更覆盖、关键断言和未覆盖高风险路径。
  - 根因：覆盖率是信号，不是质量结论。
- 反例 7：漏旧入口
  - 错：新接口通过，旧 API、定时任务、脚本和旧客户端未验证。
  - 对：按调用方/消费方列入口矩阵，至少覆盖高频旧路径。
  - 根因：能力常有多个入口，漏一个就可能线上回归。
- 反例 8：漏权限/并发
  - 错：只用管理员串行测试，普通用户越权和重复提交没测。
  - 对：覆盖未登录、无权限、越权、重复请求、并发竞争、幂等键。
  - 根因：业务事故常来自授权和竞态。
- 反例 9：E2E 只跑本地
  - 错：本地浏览器通过，没有 CI trace/video、跨浏览器、真实环境证据。
  - 对：CI 跑关键 E2E，保留 trace/video/screenshot，并标浏览器矩阵。
  - 根因：本地环境不能代表发布链路。
- 反例 10：flaky 直接重试掩盖
  - 错：失败就 retries=3 或 quarantine，未记录原因和恢复条件。
  - 对：先留产物并分类根因，临时隔离也要 ticket、owner、过期时间。
  - 根因：重试降低噪音但不消除缺陷。
- 反例 11：contract 当业务验收
  - 错：schema diff 通过就认为下单/退款业务链路通过。
  - 对：contract 验协议兼容，业务流程仍需 integration/e2e/状态机回归。
  - 根因：协议兼容不等于业务语义正确。
- 反例 12：visual 阈值吞问题
  - 错：阈值调到 5%，动画/随机数据不冻结，布局错位被忽略。
  - 对：冻结字体、时间、动画和数据，阈值只覆盖渲染噪声。
  - 根因：基线不稳会让视觉回归失去信号。
- 反例 13：CI shard 不隔离
  - 错：分片共用端口、账号、临时目录和数据库，偶发互相覆盖。
  - 对：每片使用独立资源命名、fixture、端口和清理策略。
  - 根因：并发执行放大隐藏共享状态。
- 反例 14：mutation 结果被误读
  - 错：mutation score 低却只补覆盖率用例，未增强断言。
  - 对：针对存活 mutant 补业务断言或边界样本，限定核心模块运行。
  - 根因：mutation 暴露的是断言弱，不只是用例少。
- 反例 15：CI gate 漏层
  - 错：PR 只跑 unit，contract/e2e 留到发布前人工点检。
  - 对：PR 至少跑变更相关 contract 和关键 smoke，主干/夜间补全量与专项。
  - 根因：兼容破坏越晚发现，修复和回滚成本越高。
- 反例 16：AI eval 只看均分
  - 错：模型升级后只报平均分上涨，未保存失败样本、judge 版本和人工复核。
  - 对：记录模型/prompt/judge/数据集版本，按关键样本、失败类型和抽样复核下结论。
  - 根因：均分掩盖高风险样本回归。
- 反例 17：移动端只测模拟器
  - 错：模拟器通过，真机键盘遮挡提交按钮、权限弹窗阻断流程。
  - 对：覆盖真机、系统 WebView、权限、键盘、安全区、弱网和后台恢复。
  - 根因：移动系统差异会改变关键交互。
- 反例 18：Testcontainers latest 无 healthcheck
  - 错：镜像用 latest，容器启动即连库，CI 偶发连接失败或版本漂移。
  - 对：pin 镜像版本，配置 healthcheck/ready，失败导出容器日志。
  - 根因：未就绪和版本漂移会制造 CI-only flaky。
- 反例 19：全量通过冒充覆盖
  - 错：只贴 `test all passed`，没说明需求条目、入口、风险和未覆盖项。
  - 对：把全量结果映射到改动对象、调用链、入口和证据等级。
  - 根因：全量绿不等于本次需求风险已被命中。
- 反例 20：skip/only 混进结论
  - 错：测试套件显示绿，但关键用例被 skip，或本地残留 only。
  - 对：检查跳过、隔离、软失败、缓存和 shard 明细，再给结论。
  - 根因：执行集合被改变后，通过结果不可信。
- 反例 21：mock 掉外部契约风险
  - 错：消费者契约、支付回调、事件 payload 全用随手 mock，没验 schema 和旧消费者。
  - 对：保留契约报告、真实样本或可信 fake server，并绑定消费者版本。
  - 根因：mock 方便但可能屏蔽真正会坏的边界。
- 反例 22：迁移只测空库
  - 错：新 migration 在空库通过，就说数据安全。
  - 对：用旧数据样本、大表缩放、锁等待、回滚/前滚和抽样 SQL 验证。
  - 根因：生产风险来自历史数据和规模，不来自空库。
- 反例 23：权限只测有权限
  - 错：管理员能访问就认为接口正常。
  - 对：覆盖未登录、无权限、越权对象、跨租户、旧 token 和审计日志。
  - 根因：权限缺陷通常藏在负向路径。
- 反例 24：手测无产物
  - 错：口头说“我点过了”，没有环境、账号、步骤、截图、日志或数据样本。
  - 对：保留可复核产物，至少给路径、时间、账号角色和输入输出。
  - 根因：不可复核的手测不能支撑核心结论。
- 反例 25：回滚没验证
  - 错：灰度发布 smoke 通过，没测关闭 flag、回滚版本、旧入口和数据兼容。
  - 对：发布测试绑定回滚路径、监控告警、旧端和配置恢复证据。
  - 根因：能前进不代表失败时能退回。

## 提交前自检清单

- 是否已读取需求、diff、调用方/消费方、旧入口和历史 bug？
- 是否列出风险/入口/环境/证据四要素并给出取舍理由？
- 是否完成改动对象→调用链→用户入口→测试层级→证据的影响映射？
- 是否标注证据等级，并按 skip/only、缓存、flaky、mock、环境差异做结论降级？
- 是否使用测试选择决策树，说明最小充分层级和未启用专项理由？
- 是否明确 PR/main/nightly/release/hotfix 的 gate、阻断项和降级条件？
- 是否有原 bug 红灯或明确说明无法复现的证据缺口？
- 是否覆盖 golden path、edge case、异常、权限、并发/幂等和回归？
- 是否说明 unit/integration/e2e/contract/property/mutation/visual/a11y/perf/security/eval 的取舍？
- 是否检查 mock 边界，未 mock 掉核心逻辑？
- 是否保证 fixture、账号、端口、临时目录、DB/schema 对并发 shard 隔离？
- 是否保留 flaky 失败产物并给出根因分类，而不是只加重试？
- 是否检查 `skip`、`only`、quarantine、软失败和 CI cache 没有污染结论？
- 是否报告覆盖率口径和关键未覆盖分支，而不是只报百分比？
- 是否记录命令、CI job、commit、trace/video/screenshot/报告路径？
- 手测是否有账号角色、数据、步骤、环境和截图/日志/样本路径？
- 迁移、权限、旧端、回滚命中时是否有负向和旧数据证据？
- AI 是否记录模型/prompt/judge/评测集版本、失败样本和人工复核？
- 移动是否覆盖真机/WebView/深链/权限/键盘/弱网/后台恢复？
- 命中合规数据、资源稳定性、云原生发布、大规模迁移时是否有专项证据或裁剪理由？
- 是否把结论标为已覆盖/部分覆盖/未覆盖/无法验证并列剩余风险？

## 2024-2026 新坑速查

- Playwright/Cypress：自动等待不是万能；SPA hydration、service worker、跨域 iframe、下载/弹窗和权限 prompt 仍需显式等待业务信号，trace/video 要进 CI 产物。
- Vitest/Jest：fake timers 与 Promise/microtask、jsdom 与真实浏览器差异、ESM/CJS mock hoist、并发测试共享模块状态常导致假绿或 flaky。
- Pytest：fixture scope 过大、xdist 并发、monkeypatch 泄漏、asyncio event loop 复用、tmp_path/DB 清理不彻底会污染后续用例。
- JUnit：并行执行、Testcontainers 生命周期、静态单例、时区/Locale、事务回滚边界会让本地绿 CI 红。
- Testcontainers：镜像拉取、ARM/AMD 差异、Ryuk 权限、端口竞争、CI Docker-in-Docker 和容器 healthcheck 不稳；必须锁镜像版本并等健康。
- Contract testing：契约只证明生产者/消费者协议兼容，不证明业务流程正确；消费者版本、错误 schema 和向后兼容要纳入。
- Property-based testing：随机生成能扩大边界覆盖，但必须固定 seed、保存最小失败样本并写回回归用例。
- Mutation testing：高耗时且易暴露弱断言；适合核心规则专项，不作为默认 CI 全量门禁。
- Visual regression：字体/抗锯齿/动画/时间/随机内容导致 diff 噪音；冻结时钟、字体、数据和动画，阈值不能吞掉真实布局问题。
- A11y：只跑自动扫描不够；键盘路径、焦点顺序、ARIA 语义、对比度、错误提示要配合手动或组件断言。
- Flake quarantine：隔离只能止血，必须有 ticket、owner、过期时间、恢复条件；隔离列表本身纳入质量指标。
- CI shard：按历史耗时分片会改变执行顺序；共享 fixture、端口、账号和缓存必须隔离，否则会产生 CI-only flaky。
- Coverage：行覆盖率被 getter、快照和无断言用例抬高；变更覆盖、分支覆盖、关键断言和 mutation score 更能说明风险。
- OpenTelemetry/e2e trace：跨服务 E2E 应记录 trace/span id，把失败定位到前端、网关、服务、DB 或队列；采样率不足会导致证据缺口。
- 浏览器隐私变化：第三方 cookie、Storage Partitioning、SameSite、CHIPS、Safari ITP、Chrome 隐私沙盒会影响登录、嵌入页和埋点。
- 移动端设备云：真机权限弹窗、键盘、安全区、低电量/弱网、系统 WebView 和厂商 ROM 会让本地模拟器假绿。
- CI cache/lockfile 漂移：缓存命中旧依赖、lockfile 未更新、pnpm/npm/yarn/bun 版本不同会导致本地绿 CI 红；记录包管理器和 cache key。
- AI eval 漂移：模型、prompt、RAG 索引、judge 或数据集任一变化都可能让分数不可比；必须记录版本并保存失败样本。
- 移动/混合栈：深链、通知权限、后台恢复、离线队列、系统 WebView 和厂商 ROM 会让桌面浏览器证据失效。

## 与相邻技能的边界

- 本技能负责：测试策略、风险/入口/环境矩阵、用例设计、自动化质量、flaky 排查、CI gate、覆盖率口径、回归和冒烟结论。
- aud：负责代码改动最终收口、调用链和影响面复核；本技能提供测试证据，不替代审计结论。
- rls：负责构建物、发布、灰度、回滚和环境门禁；本技能只定义发布前/后冒烟与回归证据。
- obs：负责生产监控、SLI/SLO、告警和事故响应；本技能只要求测试/冒烟中保留 trace、日志和观测证据。
- wsec：负责漏洞复现、威胁模型和修复方案；本技能只覆盖安全负向样本和回归验证。
- api：负责 API 语义、状态码、错误码、版本兼容和契约设计；本技能验证契约和消费者回归，不修改协议边界。
- aie：负责 AI 架构、RAG、Agent、prompt 和评测体系实现；本技能验证 eval 证据、漂移和回归结论。
- msec/andr/appl：负责移动端平台实现和安全细节；本技能只定义真机、WebView、权限、深链和离线同步测试证据。
- 协作：测试发现实现/契约/数据/发布/安全问题时，只输出证据和风险，切到对应技能处理；代码改动完成前必须由 aud 收口。
