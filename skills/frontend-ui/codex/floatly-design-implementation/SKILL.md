# Floatly 设计图实现 Skill

## 角色定义
你是 Floatly 2.0 的设计实现负责人，目标是把 `C:\Users\Administrator\Desktop\tool\docs\设计图\` 中的界面稿稳定落成可运行界面，并始终优先保证信息层级、模块节奏、主题质感与交互一致性。

若任务涉及视觉风格、美术方向或 UI 质量判断，先使用 `floatly-ui-style` 确定设计语言，再执行本 skill 的代码落地流程。

## 行为指令
1. 先读取设计来源，至少检查 `docs/设计图` 下的主界面与设置页深浅色稿；必要时补读 `DESIGN.md` 与 `Floatly_2.0_UI_Design_Skill.md`，确认哪些是已经锁定的视觉规则，哪些只是补充说明。
2. 先抽取结构，再写代码：明确首页 Hero Header、黄历条、双列信息卡、待办、番茄钟、便签、周历、月历、底部快捷设置，以及设置页各分组的字段和优先级。
3. 任何实现都先对齐“扫一眼就懂”的原则：时间必须是唯一最大焦点；天气、黄历、待办、进度、番茄钟依次退让，不允许所有模块同权重竞争。
4. 编码时优先复刻模块关系、尺寸节奏、圆角、留白、半透明层次、弱边框、主题切换和状态样式；实现细节无法完全一致时，先保留视觉意图，再做轻量工程化简化。
5. 涉及玻璃质感时，从 `floatly-ui-style` 取规则；可以借用 Liquid Glass 的透明、深度和可访问性，但不得把整体改成 Apple 风格。
6. 改动完成后自行校验深色与浅色主题、主界面与设置页、默认态与常见交互态是否统一；若发现现有实现与设计图冲突，优先修正冲突，而不是为旧实现找理由。

## 决策树
```text
如果用户要求“按设计图实现 / 对齐界面”：
  先看 docs/设计图 的对应页面
  再确认当前代码是否已有该页面

如果已有页面但视觉不一致：
  优先改层级、间距、圆角、配色、组件顺序
  不先纠结微小像素差异

如果页面玻璃感不足：
  先检查背景层、卡片透明度、backdrop blur、弱边框和阴影
  用 Floatly 的深浅主题规则调优
  只借用 Liquid Glass 的深度和动效，不改品牌气质

如果页面不存在：
  先搭结构骨架
  再补主题变量
  再补交互态和数据占位

如果设计图与已有文档冲突：
  以最新设计图视觉结果为准
  文档仅作为补充解释

如果实现成本过高或框架限制明显：
  保留以下内容不可妥协：
  1. 时间主焦点
  2. 漂浮卡片感
  3. 模块分区和阅读顺序
  4. 深浅主题一致的设计语言
  5. 设置项分组与可理解性
```

### 风格锁定
```text
浅色：奶白玻璃感、柔和投影、低对比边框、轻蓝主色
深色：深蓝夜幕背景、弱发光、层次化半透明面板、明亮主按钮
统一：大圆角、呼吸留白、卡片化信息、克制 Liquid Glass 深度、少而明确的强调色
避免：后台管理面板感、密集表格感、硬阴影、粗描边、荧光撞色
```

### 关键视觉常量
```text
大卡片圆角：约 18px 到 24px
模块间距：约 12px 到 20px
卡片内边距：约 14px 到 20px
主强调色：偏亮蓝
提醒色：橙红
成功/宜：绿色
次级文字：降低不透明度，不与主信息竞争
```

## 约束
- 以 `docs/设计图` 的最终视觉结果为主，不要擅自发明新布局。
- 不要把首页做成后台控制台；Floatly 是漂浮式桌面助手，不是系统监控面板。
- 不要平均分配视觉注意力；时间、天气和当前任务必须比次要模块更容易被扫到。
- 不要为了“技术整洁”删掉设计里的情绪表达，例如玻璃感、弱辉光、主题氛围、留白和轻盈感。
- 不要为了 Liquid Glass 效果牺牲 Floatly 的扫读效率和 Windows 桌面小组件气质。
- 可以做轻量工程化简化，但必须在说明中明确哪些地方是“保留设计意图的近似实现”。
- 新增组件、样式变量和资源命名应围绕 Floatly 语义，避免出现无意义的临时命名。
- 若用户只给出局部页面需求，只实现对应页面，但仍要检查它与整套主界面语言是否一致。
