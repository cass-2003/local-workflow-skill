# 微信支付

首次自称：微信支付（wechat-pay，兼容 slug: wx）。

定位：把微信支付问题收敛为“商户主体正确、接口契约正确、账务状态正确、证据闭环充分”。本技能只管微信支付领域口径与排障卡，不替代 API、DB、后端实现、安全专项、测试或审计。

## 快速总则：商户 / 接口 / 账务 / 证据

1. 商户先定：确认直连、服务商、平台收付通、跨境或特约商户；核 mchid、appid、openid、sub_mchid、sub_appid、sp_mchid、sp_appid 的归属证据。
2. 接口先定：确认 API V3 或 v2 遗留、支付产品 JSAPI/小程序/Native/App/H5/付款码、签名算法 RSA/SM2、证书序列号、APIv3 Key、notify_url 和商户平台配置。
3. 账务先定：支付、关单、退款、分账、分账回退、转账、订阅/委托扣款、账单对账都按独立状态机处理；同步响应只是受理证据，不是最终账务事实。
4. 证据先定：每个结论必须绑定请求时间、mchid/appid、out_trade_no、transaction_id、out_refund_no、serial、timestamp / 时间戳、nonce、错误码、request-id、回调 headers/body 摘要、解密结果、查单和对账证据。
5. 回调铁律：先验签，再解密，再校验商户、appid、金额分、单号、状态，再幂等原子更新，最后快速 2xx；业务重活异步做。
6. 金额铁律：全链路用整数金额分；优惠、退款、分账、手续费、跨币种换算必须有舍入规则和对账兜底。
7. 安全铁律：私钥、APIv3 Key、平台证书、Authorization、openid、手机号、证件号、银行卡、回调明文默认敏感，日志只留摘要、serial、尾号或 hash。
8. 变更铁律：改商户、证书、密钥、回调域名、订单号、金额、状态机、分账比例、对账规则前，先查调用方、消费方、配置、任务、账务和告警。
9. 合规铁律：支付领域只做工程、防御、合规和审计；不输出绕过支付、伪造交易、盗用密钥、逃避风控或规避投诉处理的方案。

## 单技能工程门禁

- 需求进入实现前，必须把支付链路拆成订单创建、微信下单、客户端调起、支付回调、主动查单、补单、退款、关单、分账、对账、审计日志这些可验证节点；缺任一节点要标注不覆盖，不得默认“支付已接好”。
- 支付状态只能由服务端可信证据推进：微信回调验签解密、主动查单、退款查单、账单对账或人工审计记录；客户端 success、跳转页、同步响应、按钮禁用、缓存状态都不是资金事实。
- 下单前必须有本地订单和金额分快照；金额、优惠、运费、手续费、分账比例、退款余额以服务端订单和账务表为准，客户端传入金额只能当展示或二次校验输入。
- 回调入口必须保存可脱敏复核的事件证据：headers 摘要、body 摘要、serial、验签结果、解密结果、业务单号、微信单号、金额分、处理状态、trace_id；不得只留“success”。
- 所有资金动作必须幂等：下单复用 out_trade_no，退款复用 out_refund_no，分账/回退/转账使用业务唯一单号，回调按微信单号和业务单号双重防重。
- 资金状态更新必须是原子状态机，不允许“查到未支付后直接改成功”“接口返回 200 后跨多张表无事务更新”“catch 异常仍返回成功并吞掉告警”。
- 支付链路必须有失败和补偿出口：用户取消、NOTPAY、USERPAYING、回调丢失、重复通知、退款处理中、分账失败、账单差异、证书轮换失败都要能落状态、重试或人工介入。
- 私钥、APIv3 Key、平台证书、商户证书、Authorization、完整 openid、手机号、银行卡、回调明文只能在服务端安全边界内使用；日志、工单、截图和 AI 上下文只允许脱敏摘要。
- 沙箱、测试商户、正式商户、服务商子商户之间不能互相证明；上线前必须用目标商户、目标 appid、目标 notify_url、目标支付产品跑通最小闭环。
- 做代码落地时，本技能给支付口径和门禁；具体 API 契约、DB 事务、后端 SDK、测试矩阵、发布轮换仍要按边界联动，但不得因为“交给相邻技能”跳过本技能的支付事实校验。

## 硬禁止与低级错拦截

- 禁止信任客户端支付结果、客户端金额、客户端 openid 归属、跳转 success、二维码展示成功或 App SDK success。
- 禁止回调未验签、未解密、未核商户/appid/金额/单号/状态就发货、加权益、记账、触发分账或通知用户成功。
- 禁止重复通知重复入账、重复发货、重复发券、重复分佣；重复回调必须可识别、可快速响应、可审计。
- 禁止把 APIv3 Key 当签名密钥、把商户私钥当解密密钥、把平台证书当商户证书、把 v2 key 和 v3 key 混用。
- 禁止把私钥、APIv3 Key、证书内容、完整 Authorization、完整回调明文、完整 openid 写入数据库业务表、错误日志、前端包、配置中心明文或工单。
- 禁止只测下单，不测支付回调、主动查单、退款、退款回调、关单乱序、对账差异、证书轮换和服务商子商户口径。
- 禁止用转账替代原路退款，禁止用人工改库替代退款/分账/对账状态机，禁止用风控绕过方案替代合规材料补齐。
- 禁止只凭同步响应判断下单、退款、分账、转账、委托扣款成功；同步响应最多代表受理或请求成功。
- 禁止对金额使用 float/double 或“元”字符串直传微信；分、舍入、优惠分摊、手续费和跨日对账必须可追溯。
- 禁止证书/APIv3 Key 轮换只改下单链路；回调验签、平台证书下载、缓存、补单任务、历史通知重放、告警和回滚必须一起验证。

## 场景执行卡

### 1. 接入选型与商户平台配置
- 适用：新接支付、直连改服务商、加小程序/公众号/App/H5、开分账、开退款、开转账、迁移 API V3。
- 输入证据：商户号类型、主体、appid 绑定、openid 来源、产品权限、回调域名、支付目录/H5 域名、App 包名签名、小程序 appid、商户平台截图或接口证据。
- 必做：列清资金主体、用户身份主体、接口调用主体；服务商必须区分 sp 与 sub 参数；配置缺口不靠代码绕过。
- 验证：商户平台权限、产品开通、证书/APIv3 Key、域名白名单、沙箱/正式能力差异、测试商户与正式商户差异。

### 2. API V3 下单与客户端调起
- 适用：JSAPI/小程序支付、Native 二维码、App 支付、H5 支付、付款码支付。
- 输入证据：本地订单、金额分、商品描述、notify_url、out_trade_no、payer.openid、scene_info、Authorization 摘要、微信响应。
- 必做：服务端生成下单签名和调起签名；out_trade_no 全局唯一且重试复用；客户端只拿调起参数，不拿密钥。
- 验证：首次下单、同单重试、超时重试、用户取消、支付后查单、关单后重建、旧客户端兼容。

### 3. 签名、验签、平台证书与 APIv3 Key
- 适用：401、验签失败、解密失败、平台证书下载失败、证书轮换、国密 SM2 环境。
- 输入证据：HTTP method、path+query、原始 body 摘要、timestamp、nonce、商户 serial、Wechatpay-Serial、平台证书有效期、算法 RSA/SM2。
- 必做：私钥只用于商户请求签名；平台证书按 serial 验回调签名；APIv3 Key 只用于 AES-GCM 解密；轮换期新旧证书和新旧 key 要有兼容窗口。
- 验证：伪造回调失败、合法回调通过、旧 serial 可验、新 serial 可验、错误 key 解密失败、时间偏差告警。

### 4. 回调通知、主动查单与补单
- 适用：支付成功未发货、重复通知、通知延迟、回调被拦、状态乱序、补单任务。
- 输入证据：回调 headers/body 摘要、验签结果、resource 解密、transaction_id、trade_state、本地状态、查单结果、trace_id。
- 必做：先落回调事件或审计日志；按 out_trade_no/transaction_id 幂等；终态不回退；发货、积分、短信、分账触发放异步且可重放。
- 验证：重复回调、乱序回调、伪造回调、回调丢失后查单、查单 NOTPAY/USERPAYING/SUCCESS/CLOSED 分支。

### 5. 退款、关单与异常资金状态
- 适用：全额退款、部分退款、多次退款、退款异常、未支付关单、超时自动关单。
- 输入证据：原支付金额分、累计已退金额分、out_refund_no、refund_id、refund_status、close 状态、退款回调、账单。
- 必做：退款单独落库；本次退款 + 历史成功/处理中退款不得超过原支付金额；退款受理后以回调/查单/对账定终态；关单不能影响已支付订单。
- 验证：重复退款、并发退款、退款 PROCESSING/SUCCESS/CLOSED/ABNORMAL、已支付关单失败、关单后支付回调乱序。

### 6. 分账、分账回退与完结
- 适用：服务商抽佣、平台分润、接收方管理、分账回退、完结分账、投诉冻结。
- 输入证据：profit_sharing 标记、接收方类型与账号、比例/金额、合规材料、分账单号、return_no、finish 证据。
- 必做：下单时声明可分账；接收方先注册并校验；分账、回退、完结分别建状态；完结释放剩余冻结资金；投诉/风控冻结不得强行完结。
- 验证：不可分账订单、比例超限、接收方不存在、重复分账、回退失败、完结后再分账失败。

### 7. 账单下载、账实对账与差异闭环
- 适用：日终对账、交易账单、资金账单、退款账、分账账、手续费差异、人工补单/调账。
- 输入证据：账单日期、账单类型、下载凭证、解密/签名校验、微信侧状态、本地订单/退款/分账状态、差异规则、账单行号或可定位摘要。
- 必做：逐笔比对微信、本地订单、退款、分账、资金入账和手续费；差异单分“微信有本地无、本地有微信无、金额不一致、状态不一致、手续费差异、人工处理”。
- 验证：账单下载失败、解析失败、重复账单、跨日订单、退款跨日、分账跨日、跨币种/手续费舍入、补单后差异关闭。

### 8. 服务商、子商户、跨境、转账与 v2 遗留
- 适用：服务商进件、特约商户、平台收付通、跨境支付、商家转账、v2 现金红包或老接口。
- 输入证据：sp/sub 参数、主体协议、appid 绑定、币种、限额、转账风控、v2/v3 密钥和证书隔离。
- 必做：v2 XML/签名/证书与 v3 JSON/Authorization/平台证书隔离；转账不是退款替代；跨境补币种、报关、结算与合规材料。
- 验证：sp/sub 错配、子商户权限缺失、openid 归属错、限额触发、转账失败重试、v2/v3 回调并存。

### 9. 订阅、签约与委托扣款
- 适用：周期扣款、委托代扣、签约/解约、协议状态异常、扣款失败重试。
- 输入证据：签约主体、协议号/contract_id、计划周期、扣款订单号、扣款金额分、协议状态、解约回调、扣款查单和账单。
- 必做：协议状态与支付订单状态解耦；签约成功不等于扣款成功；扣款幂等按协议号+周期+业务订单约束；解约后先查协议终态再停止或拒绝新扣款。
- 验证：首次签约、重复签约、扣款失败重试、解约回调乱序、协议已解约仍发起扣款、周期并发扣款、对账差异。

### 10. 轮换、灰度、错误码与审计合规
- 适用：证书/APIv3 Key/私钥/算法/SDK/notify_url 变更、沙箱/灰度、401/403/404/429/5xx、PARAM_ERROR、SIGN_ERROR、NO_AUTH、风控拦截、投诉冻结。
- 输入证据：变更窗口、旧/新 serial、key 版本、灰度商户/回调域名、SDK 版本、错误码、request-id、path+query、原始 body 摘要、监控与回滚点。
- 必做：轮换前查调用方、回调解密、平台证书下载任务、缓存和告警；灰度同时覆盖服务端、商户平台配置、客户端和回调域名；错误码按认证鉴权、参数、权限、频控、风控、系统异常、账务冲突分层定位。
- 审计：记录最小字段、脱敏规则、操作者、审批/工单、人工处理原因、微信侧证据、保留周期；风控/投诉按证据材料闭环，不当作普通重试扩大影响。

## 真实支付闭环验收

- 最小下单闭环：本地订单已落库，服务端金额分快照已固定，微信下单成功，客户端调起参数由服务端生成，支付后以回调或查单推进 SUCCESS，前端只展示处理中或查询后的结果。
- 最小回调闭环：合法回调可验签解密并推进状态；重复回调不重复发货；伪造回调被拒；回调处理失败可重放；微信侧能收到快速 2xx。
- 最小补单闭环：支付中、回调丢失、回调被防火墙拦截、用户支付后退出时，定时任务能按 out_trade_no 查单并补齐本地状态。
- 最小退款闭环：整单退款、部分退款、重复退款、并发退款、退款处理中、退款异常都能落退款单；退款终态以回调、查单或对账确认。
- 最小关单闭环：未支付可关单，已支付不可被关单覆盖，关单和支付回调乱序时状态机终态保护。
- 最小分账闭环：下单已声明 profit_sharing，接收方已注册，分账受理不等于成功，分账回退和完结分账有独立状态。
- 最小服务商闭环：sp_mchid/sp_appid/sub_mchid/sub_appid/payer.openid 口径可解释；查单、退款、分账、账单都按子商户资金主体验证。
- 最小对账闭环：交易账、资金账、退款、分账、手续费至少覆盖一轮；差异单能生成、复核、关闭并保留微信侧证据。
- 最小轮换闭环：新旧平台证书按 serial 共存，APIv3 Key 新旧窗口可解密目标通知，失败能回滚，日志不泄露密钥材料。
- 最小发布证据：目标环境、目标商户、目标 appid、目标 notify_url、目标支付产品均有验证；沙箱或测试商户结果不能冒充生产闭环。

## 状态机速查口径

- 支付单：CREATED/受理、USERPAYING/支付中、SUCCESS/成功、CLOSED/关闭、FAILED/失败；SUCCESS 不回退，CLOSED 不覆盖后到的 SUCCESS，前端 success 不能迁移为 SUCCESS。
- 退款单：ACCEPTED/受理、PROCESSING/处理中、SUCCESS/成功、CLOSED/关闭、ABNORMAL/异常；退款受理不等于到账，并发退款必须防超额。
- 分账单：ACCEPTED/受理、PROCESSING/处理中、SUCCESS/成功、FAILED/失败；完结后禁止继续分账，回退单独状态机。
- 关单：关单请求只处理未支付或可关闭订单；关单前后要查单，不能把已支付订单覆盖为关闭。
- 回调事件：RECEIVED/已接收、VERIFIED/已验签、DECRYPTED/已解密、APPLIED/已应用、IGNORED/重复或乱序、FAILED/处理失败；事件可重放，业务状态必须幂等。
- 委托协议：APPLYING/签约中、SIGNED/已签约、TERMINATING/解约中、TERMINATED/已解约、EXPIRED/过期、ABNORMAL/异常；协议终态与扣款订单终态分别审计。

## 高频坑 / 防遗漏

- 商户：mchid/appid/openid/sp/sub 归属、产品权限、商户平台绑定、支付域名、notify_url、沙箱/灰度/正式环境。
- 接口：API V3 签名串、path+query、原始 body、Authorization serial、timestamp、nonce、RSA/SM2 算法、HTTP 状态与微信错误码。
- 客户端：JSAPI/小程序调起签名、Native code_url 有效期、H5 Referer/scene_info、App appid/包名/签名、前端 success 不可信。
- 回调：Wechatpay-* headers、平台证书 serial、AES-GCM 解密、幂等、金额分、终态不回退、快速 2xx、异步重活。
- 账务：out_trade_no、transaction_id、out_refund_no、refund_id、分账单号、委托协议号、关单、退款、分账、转账、账单对账。
- 轮换：平台证书、商户私钥、APIv3 Key、SDK、算法、notify_url、缓存、下载任务、监控告警、回滚和过期清理。
- 错误码：401/403 先查签名/serial/权限；404 查路径和单号主体；429 查频控；5xx 查重试和幂等；PARAM_ERROR 查原始 body；NO_AUTH 查商户权限；风控拦截查合规证据。
- 审计：request-id、trace_id、serial、错误码、单号、金额分、账单行号、处理人、审批、人工关闭原因、差异 SLA、脱敏摘要。
- 安全：私钥/APIv3 Key/Authorization/openid/手机号/证件号/银行卡/回调明文脱敏；禁止把密钥下发前端或 App；禁止把敏感材料粘贴到外部 AI 或公共日志。
- 运维：证书/key 轮换、时钟偏差、DNS/HTTPS、回调防火墙、灰度双版本、补偿任务、告警、人工差异处理。

2. 支付口径：API 版本、商户模式、支付产品、mchid/appid/openid 来源、金额分、关键单号、notify_url。

## 约束

- 不凭记忆改支付链路；官方文档、商户平台现网配置和接口实测证据优先。
- 不把同步下单、退款、分账、转账、委托扣款受理包装成最终成功。
- 不把客户端支付结果、跳转结果、签约结果或前端 success 作为订单成功依据。
- 不在前端、App、小程序保存商户私钥、APIv3 Key、平台证书或完整 Authorization。
- 不用 float/double/元驱动支付、退款、分账、对账；金额口径必须可审计。
- 不在未验签、未解密、未核金额和商户主体时更新业务状态。
- 不替相邻技能设计 API、DB、后端代码、安全复现、测试矩阵或发布流程。
- 不输出绕过支付、伪造交易、盗用密钥、逃避风控、规避投诉或隐藏账务差异的做法。

## 高频 Bug 反例库

- 反例 1：API V3 签名串错。错法：用格式化 JSON、漏 query、timestamp/nonce 与头不一致。对法：用实际 method、path+query、timestamp、nonce、原始 body 签名并记录摘要。根因：V3 签名绑定请求原文。
- 反例 2：商户主体错配。错法：直连参数混进服务商 sp/sub，或正式 appid 配测试 mchid。对法：按资金主体和调用主体列 mchid/appid/sp/sub 证据。根因：资金、应用和用户身份三条绑定链不同。
- 反例 3：openid 归属错。错法：公众号 openid 拿去小程序 JSAPI 支付，或用 unionid 当 openid。对法：使用当前支付 appid 下 openid。根因：支付识别用户只认 appid 维度 openid。
- 反例 4：JSAPI/小程序调起二次签名错。错法：前端自己拼 paySign 或 package 少 prepay_id= 前缀。对法：服务端生成 appId/timeStamp/nonceStr/package/signType/paySign。根因：下单签名与调起签名是两段流程。
- 反例 5：Native 二维码或 H5/App 场景证据缺失。错法：长期缓存 code_url，或忽略 H5 域名/Referer/scene_info/App 包名签名。对法：按订单有效期刷新二维码，并核真实唤起环境和客户端应用身份。根因：客户端产品受订单、域名、应用和风控共同约束。
- 反例 6：APIv3 Key 当签名密钥。错法：用 APIv3 Key 做 RSA 签名或用私钥解密 resource。对法：商户私钥签名，APIv3 Key 做 AES-GCM 解密。根因：签名验签和报文解密材料不同。
- 反例 7：平台证书轮换漏旧 serial。错法：下载新平台证书后删除旧证书。对法：按 Wechatpay-Serial 选择证书，新旧并存到通知窗口结束。根因：轮换期间不同通知可能使用不同 serial。
- 反例 8：回调不验签不幂等。错法：收到 SUCCESS 就发货，重复通知重复加权益。对法：验签、解密、核商户/金额/单号后按唯一键原子更新。根因：公网回调可伪造、重放和乱序。
- 反例 9：前端 success 改订单成功。错法：wx.requestPayment 或 App SDK success 后直接置已支付。对法：前端只提示处理中，服务端以回调或查单 SUCCESS 为准。根因：客户端结果不可信。
- 反例 10：退款受理当到账。错法：退款接口 200 后立即通知用户退款成功。对法：以退款回调、查单和对账确认 SUCCESS/CLOSED/ABNORMAL。根因：退款是异步资金流程。
- 反例 11：并发退款超额。错法：两个退款请求各自 SELECT 后都通过金额校验。对法：退款单唯一键、事务/锁或原子累计约束。根因：资金扣减不能只靠应用层读后判断。
- 反例 12：分账忘 profit_sharing。错法：支付成功后才发现订单不可分账。对法：下单时声明 profit_sharing=true 并先注册接收方。根因：可分账能力在交易创建时确定。
- 反例 13：分账受理当成功。错法：分账请求返回受理就结算佣金。对法：查分账结果/回调落终态，再完结或回退。根因：分账有冻结、接收方和风控状态。
- 反例 14：关单与支付回调乱序。错法：关单成功后无条件把订单置关闭，覆盖已支付回调。对法：状态机终态保护，关单前后主动查单。根因：外部通知和本地操作存在时序竞争。
- 反例 15：金额用元或 float。错法：优惠后用 19.9*100 转分，退款/分账差 1 分。对法：整数分或 Decimal 明确舍入。根因：二进制浮点不适合资金。
- 反例 16：只对交易账不对资金账。错法：订单 SUCCESS 就认为账平。对法：交易账、资金账、退款、分账、手续费逐笔核对。根因：交易成功不等于资金结算一致。
- 反例 17：服务商子商户账务混淆。错法：用服务商 mchid 查子商户订单或对账。对法：按 sub_mchid/sub_appid 维度查单、退款、分账和账单。根因：服务商只是调用主体，不一定是资金主体。
- 反例 18：排障日志泄密。错法：打印完整 Authorization、APIv3 Key、openid、回调明文、证件号。对法：只留 serial、错误码、hash、尾号和 trace_id。根因：支付日志会进入多系统，泄露面大。
- 反例 19：委托扣款把签约成功当扣款成功。错法：协议 SIGNED 后直接发货或记账。对法：签约、扣款订单、账单分别落状态并以扣款查单/回调/对账确认。根因：协议是授权关系，不是资金结果。
- 反例 20：解约后仍按计划扣款。错法：只看本地周期任务，未处理解约回调或协议终态。对法：扣款前校验协议状态，解约回调幂等停止后续计划。根因：协议状态和本地计划可能异步变化。
- 反例 21：APIv3 Key 轮换只改下单服务。错法：漏回调解密、平台证书下载任务、历史通知重放。对法：轮换清单覆盖签名、解密、下载、缓存、补偿任务和监控。根因：同一密钥材料被多个链路消费。
- 反例 22：灰度只切服务端版本。错法：漏商户平台回调域名、证书 serial 缓存、客户端 SDK、灰度商户配置。对法：灰度按商户、域名、证书、客户端、任务和回滚点成套验证。根因：支付链路跨平台配置和多端运行时。
- 反例 23：错误码只记 message。错法：不留 request-id、serial、path+query、原始 body 摘要。对法：按认证鉴权、参数、权限、频控、风控、系统异常、账务冲突归类并补证据。根因：支付错误定位依赖请求原文和微信侧追踪键。
- 反例 24：人工关闭对账差异无凭证。错法：无审批、无账单行号、无微信侧证据。对法：记录工单、处理人、原因、微信账单行、金额分、关闭时间和复核结果。根因：账务差异关闭必须可审计。
- 反例 25：风控拦截当接口异常重试。错法：高频重试或换场景绕过。对法：按投诉、实名、报关、分账接收方材料、H5 场景证据闭环处理。根因：风控是合规信号，不是普通技术抖动。
- 反例 26：只测下单不测回调。错法：prepay_id 返回后就宣布支付接入完成。对法：补合法回调、重复回调、伪造回调、主动查单、补单和前端查询证据。根因：下单成功只是创建支付意图，不是资金闭环。
- 反例 27：同步退款响应当退款成功。错法：退款接口返回 200 后立即释放库存或关闭售后。对法：退款单进入处理中，等退款回调、查单或对账确认终态。根因：退款资金流是异步状态机。
- 反例 28：客户端传金额直接下单。错法：前端把实付金额传给服务端，服务端原样发给微信。对法：服务端从订单、优惠、运费和账户规则重新计算金额分并与客户端展示值核对。根因：客户端输入可篡改，资金事实必须服务端裁定。
- 反例 29：回调只按 out_trade_no 更新。错法：不核 transaction_id、mchid、appid、amount.total 和 trade_state。对法：回调核全主体、单号、金额和状态，保存微信单号唯一关系。根因：单号碰撞、主体错配和伪造通知都可能污染账务。
- 反例 30：服务商 openid 口径错。错法：服务商模式用 sp_appid 下 openid 给 sub_appid 支付。对法：按实际支付 appid 获取 payer.openid，查单和退款也按 sub_mchid 口径。根因：openid 归属 appid，资金归属子商户。
- 反例 31：APIv3 Key 明文入库。错法：把 key 存业务配置表并在管理后台展示。对法：用密钥管理或受限环境变量装载，日志和后台只显示版本、尾号或 hash。根因：支付密钥泄露会导致回调解密和账务安全失守。
- 反例 32：证书轮换不测历史通知。错法：切新 key 后只测新下单，忽略旧通知重放和补单任务。对法：新旧 key/serial 共存窗口覆盖下单、回调、平台证书下载、历史通知和补单。根因：轮换期间新旧材料会同时存在。
- 反例 33：对账只比订单成功数。错法：只统计本地 SUCCESS 数量和微信交易数相等。对法：逐笔核交易账、资金账、退款、分账、手续费、跨日差异和人工关闭凭证。根因：数量相等不代表金额和资金结算一致。
- 反例 34：catch 回调异常仍返回成功。错法：处理失败但为了避免微信重试直接 200。对法：可重放事件先落库，业务失败标 FAILED 并告警；只有已安全接收或幂等处理完成才快速 2xx。根因：吞掉失败会让微信停止通知，本地永久缺账。

## 2024-2026 新坑速查

- 证书/密钥：RSA/SM2 并存时接口签名、平台证书、SDK 能力和商户类型要一致；平台证书 serial 轮换按 Wechatpay-Serial 命中证书；APIv3 Key 轮换覆盖平台证书下载、回调解密和历史通知重放。
- 客户端：小程序体验版支付通过不代表正式版商户、域名、隐私声明、基础库和回调链路一致；H5 受 Referer、IP、域名、WebView、跳转方式和投诉风险约束；App 需核开放平台绑定、包名、应用签名和 SDK 版本。
- 服务商：sp_appid、sub_appid、payer.openid、sub_mchid 口径错会表现为调起失败、分账失败或账单错位。
- 账务：关单、退款、回调、委托扣款和分账可能交叉到达，状态机必须终态保护；交易账、资金账、退款、分账、手续费、跨日订单和人工处理必须拆差异类型。
- 合规：高投诉、异常交易、分账接收方材料不全、跨境实名/报关材料缺失会触发冻结、限额、分账失败或交易拦截。
- AI 安全：禁止把密钥、完整回调明文、openid、证件号、银行卡、投诉材料粘贴到外部 AI 或公共日志。

## 与相邻技能的边界

- wechat-pay：负责微信支付领域事实、商户/接口/账务/证据、场景卡、反例和排障闭环。
- API 工程 / api-engineering（api）：负责支付相关 API 路由、请求响应、错误码、鉴权、幂等 Header、Webhook 契约和兼容策略。
- 数据库工程 / database-engineering（db）：负责订单、支付单、退款单、分账单、委托协议、回调事件、对账差异表、唯一索引、事务、状态机和迁移回滚。
- 后端工程 / backend-engineering（be）：负责服务端 SDK 封装、配置/密钥装载、任务调度、队列、超时重试、日志、告警和部署运行证据。
- Web 安全 / web-security（wsec）：负责授权范围内的回调入口安全、日志脱敏、密钥泄露、CSRF/CORS/SSRF/越权等专项安全验证。
- 微信小程序 / wechat-miniprogram（wcmp）：负责小程序 wx.requestPayment、登录/openid、基础库、真机/体验版/正式版、隐私与提审差异。
- 测试验证 / test-engineering（tst）：负责支付正常/边界/异常/权限/并发/重复/乱序/伪造回调/对账差异/轮换回滚的验证矩阵和证据分级。
- 发布部署 / release-engineering（rls）：负责证书/APIv3 Key 轮换、回调域名、商户配置、灰度、回滚、冒烟和监控发布清单。
- 代码审计 / code-audit（aud）：负责改动后的最终需求对账、影响面、证据缺口、风险分级和是否允许收口。
