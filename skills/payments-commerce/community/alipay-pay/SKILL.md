# 支付宝支付

首次自称：支付宝支付（alipay-pay，兼容 slug: ali）。

定位：只处理支付宝支付链路的工程排障、证据链、领域坑点、验证口径与交付检查；不替代语言 SDK、数据库、API、发布平台、安全审计或财务系统设计，不做真实支付调用，不索要密钥，不提供绕过风控、绕过用户授权、欺诈交易、套现或规避合规的方案。

## 快速总则

- 先定入口：下单、同步返回、异步通知、查单、退款、关闭、撤销、预授权、周期扣款/代扣、分账、转账、账单、对账、证书轮换、ISV 代调用、支付宝小程序支付或发布回滚；入口不清先不改。
- 先定版本：通用版开放平台接口、v3 接口、官方 SDK 版本、语言运行时、TLS/编码/时区、普通公钥模式或公钥证书模式；禁止把微信 v3、分整数、JSON 回调规则套给支付宝。
- 先定产品能力：普通支付、当面付、小程序支付、花呗分期、预授权、周期扣款/代扣、分账、转账、跨境/Alipay+ 属于不同产品能力；未确认签约、授权、应用上线状态和接口权限前，不把错误归因为代码。
- 先定环境：沙箱与生产必须成套隔离 app_id、商户号/PID、网关、应用私钥、应用公钥、支付宝公钥、应用公钥证书、支付宝公钥证书、支付宝根证书、AES 密钥、notify_url、return_url、app_auth_token 和日志级别。
- 先定证据：请求参数摘要、支付宝响应 code/sub_code、trade_no、out_trade_no、buyer_id 脱敏值、异步通知原始 form、验签材料、订单/退款/分账/账单状态、trace_id、SDK 版本、配置来源和发布时间；未读证据不下结论。
- 支付成功口径：同步返回只做展示或触发查单，最终以异步通知验签通过且业务字段校验通过，或主动查单确认成功为准。
- 回调底线：异步通知必须按 POST form 原参验签，校验 app_id、seller_id/商户号、out_trade_no、trade_no、total_amount、状态和幂等锁，成功后返回纯文本 success。
- 金额底线：支付宝 total_amount/refund_amount 使用元字符串，禁止 float；订单金额、用户实付、商户实收、优惠/红包/补贴、手续费、分期贴息、退款退券、分账、在途资金、挂账差异必须进入资金流水与账务口径。
- 状态机底线：支付、退款、分账、转账、签约、扣款、预授权状态只允许按规则单向推进；终态不可被异步通知、补偿任务或人工操作回退；乱序通知以验签后主动查询、内部版本号和审计记录收敛。
- 收口底线：不能只等回调；必须有查单、退款查询、分账查询、账单下载、逐笔对账、超时重试、补偿任务、告警和人工处理出口。

## 单技能工程门禁

- 需求入口门禁：接到“接支付宝支付/修支付/加退款/处理回调/上线支付”时，先落交易阶段、产品形态、客户端入口、商户模式、资金动作和验收样本；缺任一项只能列待确认，不直接写支付成功逻辑。
- 服务端真相源门禁：订单金额、商品归属、商户主体、分账接收方、退款额度、会员开通、发货或权益发放只信服务端订单和支付宝查单/通知，不信客户端传入金额、tradePay 返回、return_url 参数或前端缓存。
- 签名验签门禁：下单签名、异步通知验签、退款/转账/分账资金接口证书序列号必须分别验证；普通公钥与公钥证书模式不能混配；私钥、公钥证书、根证书和环境变量必须有来源证据。
- 回调工程门禁：notify 路由必须能拿到 POST form 原参；框架 body parser、网关、WAF、反向代理、字符集、排序和转义不能破坏验签材料；验签通过后再做业务字段校验、幂等锁、事务状态迁移和纯文本 success。
- 幂等与状态门禁：out_trade_no、trade_no、out_request_no、out_biz_no、app_id、seller_id 和金额字段必须有唯一约束或等价幂等保护；重复通知、补偿查单、人工重放、队列重试都不能重复入账、重复开通、重复退款或重复分账。
- 资金闭环门禁：上线前必须覆盖下单、同步返回、异步通知、主动查单、退款、退款查询、账单下载、逐笔对账和差异处理；只测下单成功不算支付链路完成。
- 发布门禁：沙箱到生产切换必须逐项核对网关、app_id、PID、证书、notify_url、return_url、app_auth_token、产品签约、应用上线、白名单、日志级别、告警和回滚；生产至少做小金额支付、通知、退款和账单证据。
- 证据门禁：每个支付改动必须能拿出脱敏请求摘要、支付宝响应、通知摘要、验签结果、状态迁移、资金流水、补偿任务和验证记录；没有证据不能报“支付已通”。

## 场景执行卡

### 1. 新接订单支付：APP / H5 / PC / 当面付 / 支付宝小程序
- 先查：产品类型、客户端入口、直连或 ISV、通用版或 v3、SDK 版本、签名模式、网关、app_id、商户号/PID、订单号规则、金额来源、notify_url 可达性、产品签约和应用上线状态。
- 动作：生成全局唯一 out_trade_no；total_amount 使用元字符串；服务端签名；APP 返回 orderStr，H5/PC 返回跳转或表单，当面付返回二维码，小程序走对应 my.tradePay 或插件/服务端产品要求。
- 证据：保存请求摘要、响应 code/sub_code/msg、trade_no/out_trade_no、金额、客户端拉起结果、同步返回参数、查单结果、trace_id 和脱敏配置快照。
- 验证：成功、取消、超时、重复点击、弱网、金额边界、客户端版本差异、沙箱到生产切换、notify_url 域名和 TLS。

### 2. 异步通知 / 同步返回 / 查单补偿
- 先查：通知是否 POST form，路由是否透传原始参数，中间件是否改写 body/编码，验签用支付宝公钥还是支付宝公钥证书，success 响应是否纯文本。
- 动作：同步返回不改最终成功；异步通知先验签再做幂等锁、业务字段校验和状态机推进；通知丢失、超时或验签异常时按 out_trade_no 主动查单补偿。
- 证据：原始通知参数摘要、sign/sign_type、验签结果、证书序列号、幂等命中、状态迁移、响应文本、支付宝重试次数、查单补偿结果。
- 验证：重复通知、乱序通知、伪造签名、金额不符、订单不存在、处理超时、通知丢失、查单成功但回调未到、终态保护。

### 3. 签名 / 验签 / RSA2 / 公钥证书 / 轮换
- 先查：sign_type 是否 RSA2，普通公钥模式还是证书模式，应用私钥、应用公钥、支付宝公钥、应用公钥证书、支付宝公钥证书、支付宝根证书是否成套匹配。
- 动作：私钥只在服务端安全配置；证书模式保留 cert_sn、alipay_root_cert_sn、支付宝公钥证书序列号；轮换必须灰度、双读或快速回滚；沙箱与生产证书完全隔离。
- 证据：签名原串摘要、验签参数集合、证书序列号、证书有效期、SDK 版本、配置来源、失败 sub_code、容器镜像/环境变量版本。
- 验证：下单签名、回调验签、证书过期、证书不匹配、根证书漏配、网关切换、编码/换行/时区差异、SDK 升级兼容。

### 4. 退款 / 关闭 / 撤销
- 先查：订单状态、可退金额、已退金额、退款号 out_request_no、关闭 close 与撤销 cancel 边界、退款查询入口、退款资金流入账规则。
- 动作：未支付关闭用 close；处理中按产品语义撤销；已支付退款用 refund；每个退款意图有稳定 out_request_no，部分退款每笔独立幂等号；失败先查退款状态再重试。
- 证据：退款请求、支付宝退款响应、refund_detail_item_list、退款查询、原订单状态、剩余可退金额、账务记录和对账差异。
- 验证：全额退款、部分退款、多次部分退款、重复提交、超退、退款处理中、退款失败补偿、退款到账与账单差异。

### 5. 分账 / 转账 / 结算 / 资金流水对账
- 先查：产品签约、分账关系、转账权限、公钥证书要求、out_biz_no/out_request_no、接收方身份、结算周期、账单下载权限、账单类型和内部科目口径。
- 动作：分账前校验可分账金额和接收方；转账/资金类接口按要求使用公钥证书；账单下载后逐笔核对订单、退款、手续费、分账、补贴、跨日交易和补差，并输出差异归因。
- 证据：分账/转账请求、支付宝单号、状态查询、账单文件摘要、差异列表、补偿任务、人工处理记录、风控或权限 sub_code。
- 验证：幂等重试、金额边界、接收方异常、账单缺行、账单延迟、跨日退款、手续费差异、分账回退、总额平但逐笔或科目不平、人工调账回放。

### 6. ISV 服务商 / 授权 / 多商户
- 先查：平台 app_id、商户 app_id/PID、app_auth_token、授权有效期、商户配置隔离、回调归属、账单归属、接口授权范围和审计要求。
- 动作：代调用必须传 app_auth_token；按商户隔离订单号、证书、账单、通知处理、幂等锁和审计日志；授权失效走重新授权、暂停交易或人工处理。
- 证据：授权记录、代调用请求摘要、商户维度订单、通知验签、权限错误、账单归属、审计日志。
- 验证：单商户、多商户并发、授权失效、商户切换、错商户回调、错账单归属、配置串线。

### 7. 周期扣款 / 代扣 / 签约支付
- 先查：产品签约、用户授权协议、协议号、签约场景、扣款周期、扣款上限、可取消路径、签约通知、扣款通知、解约入口、商户主体和合规展示；新接口或政策变化需官方验证。
- 动作：签约与扣款分离建模；扣款使用稳定幂等号；失败按协议状态、余额、风控、权限分类处理；用户解约后停止扣款任务；自动续费必须保留协议、提示、扣费记录和取消入口证据。
- 证据：签约请求摘要、协议号脱敏值、签约/解约通知、扣款请求、扣款响应、查单结果、用户授权页面版本、任务执行记录和客服处理记录。
- 验证：首次签约、续费扣款、扣款失败限次重试、用户解约、协议过期、重复扣款防护、通知丢失补偿、合规提示和退款争议。

### 8. 花呗分期 / 优惠补贴 / 营销资金
- 先查：营销产品签约、分期参数、贴息承担方、优惠券/红包/补贴来源、用户实际支付、商户实收、退款退券规则、账单字段和活动有效期；字段以官方文档和响应为准。
- 动作：订单金额、买家支付金额、商户实收、优惠补贴、手续费和贴息分开记录；退款时按支付宝退款明细和账单字段核对退券、退补贴和手续费差异；营销活动变更必须可追溯。
- 证据：下单营销参数摘要、支付宝响应优惠明细、账单优惠/手续费字段、退款明细、活动配置快照、差异单和人工处理记录。
- 验证：无优惠、有优惠、花呗分期、贴息、部分退款、全额退款、活动过期、补贴不到账、账单跨日差异。

### 9. 预授权 / 押金 / 后付费
- 先查：预授权产品权限、冻结金额、可转支付金额、解冻期限、部分转支付、超时释放、押金订单与消费订单关系；预授权能力和字段需按官方资料验证。
- 动作：冻结、转支付、解冻分别建幂等号和状态；超时任务主动释放；转支付失败先查授权状态；押金差额退款或补扣必须有用户协议与账务记录。
- 证据：冻结请求摘要、授权单号、转支付请求、解冻请求、状态查询、超时任务日志、押金/消费映射、账单差异。
- 验证：冻结成功、冻结失败、部分转支付、全额转支付、主动解冻、超时解冻、重复转支付、用户取消和账单核对。

### 10. 超时 / 风控 / 投诉 / 发布 / 回滚
- 先查：网关超时、SDK 超时、重试次数、幂等键、风控 sub_code、投诉/冻结状态、生产开关、回调域名、证书发布窗口、应用上线、接口权限、产品签约、授权范围、回调域名白名单、灰度策略和告警规则。
- 动作：外部请求设置超时；网络失败先查单再决定重试；风控、投诉、资金冻结只做合规闭环，保留交易证据、用户授权、商品/服务履约、客服工单、退款处理、投诉时限和人工复核记录；证书/网关/notify_url 变更必须灰度、小金额验证、可回滚。
- 证据：超时日志、请求耗时、重试次数、查单结果、风控 sub_code、投诉/冻结证据、发布批次、告警、回滚验证和小金额交易单。
- 验证：网关慢、SDK 异常、重复请求、风控拒绝、投诉冻结、证书回滚、notify_url 切换、告警触发、补偿任务恢复。

## 硬禁止与低级错拦截

- 禁止客户端决定支付成功、金额、商户归属、会员开通、发货、退款额度或分账接收方；客户端只负责调起和展示。
- 禁止把 my.tradePay、APP SDK、H5/PC return_url 的成功结果当最终支付成功；必须以验签通知或主动查单确认。
- 禁止回调不验签、不按原始 POST form 验签、验签后不校 app_id/seller_id/out_trade_no/trade_no/total_amount/trade_status。
- 禁止通知处理先改订单再验签，或验签失败仍返回 success；禁止错误响应写成 JSON success、HTML success 或带引号 success。
- 禁止重复通知重复入账；支付、退款、分账、转账、扣款、预授权各自必须有幂等号、终态保护和审计记录。
- 禁止用 float、分整数或客户端金额生成支付宝 total_amount/refund_amount；禁止订单实付、商户实收、优惠补贴和手续费混成一个金额字段。
- 禁止沙箱与生产混用网关、app_id、PID、私钥、支付宝公钥、证书、notify_url、app_auth_token、日志级别和产品权限。
- 禁止私钥、完整证书、完整通知原文、buyer_id、手机号、token、账单明细、app_auth_token 或商户敏感配置进入日志、报错、工单截图和测试快照。
- 禁止退款失败、网络超时、队列超时后换新幂等号盲目重试；必须先查原交易或原退款状态。
- 禁止只测下单不测异步通知、主动查单、退款、退款查询、对账、重复通知、金额不符、伪造签名和生产小金额。

## 高频坑 / 防遗漏

### 高频坑
- 金额按分传、float 计算、字符串精度丢失，导致金额不符或验签后的业务校验失败。
- 异步通知当 JSON 解析，或中间件重排/过滤参数，导致验签材料变化。
- 同步返回直接改为支付成功，跳过异步通知、查单和账务对账。
- 回调只验签不校验 app_id、seller_id、out_trade_no、trade_no、total_amount 和订单状态。
- 返回 JSON、HTML 或带引号的 success，导致支付宝继续重试。
- 应用公钥、支付宝公钥、应用公钥证书、支付宝公钥证书、根证书混淆。
- 沙箱和生产网关、app_id、证书、私钥、notify_url、app_auth_token 串线。
- 通用版与 v3 SDK、接口字段、证书加载方式、错误码处理混用。
- 退款、分账、转账、预授权、扣款共用订单号或幂等号，导致重复扣退、重复冻结或无法追账。
- 日志打印私钥、完整证书、buyer_id、手机号、token、完整通知原文或未脱敏账单。

### 防遗漏清单
- 改下单：查产品、金额元字符串、唯一订单、notify_url、客户端入口、SDK/平台版本、沙箱生产配置。
- 改回调：查 form 原参、验签、公钥/证书、幂等、金额与商户校验、纯文本 success、补偿查单。
- 改退款：查可退金额、out_request_no、退款查询、重复重试、账务状态、退款账单。

## 高频 Bug 反例库

- 反例 1：错法：把支付宝 total_amount 当微信分整数传。对法：使用元字符串并校验小数精度、币种和订单金额。根因：混淆支付宝与微信支付金额单位，且缺少金额业务校验。
- 反例 2：错法：异步通知按 JSON body 解析后再验签。对法：按支付宝 POST form 原始参数集合验签，排除 sign/sign_type 后使用 SDK 规则。根因：中间件改变验签材料。
- 反例 3：错法：用户从同步返回页回来就把订单置为成功。对法：同步返回只展示或触发查单，最终以异步通知或主动查单确认。根因：混淆前端跳转成功与资金成功。
- 反例 4：错法：回调验签通过后不校验 app_id、seller_id、out_trade_no、trade_no、total_amount。对法：验签后继续校验业务归属、金额和状态机。根因：把签名真实性误当业务正确性。
- 反例 5：错法：回调处理成功返回 JSON 或 success 带引号。对法：返回纯文本 success，失败时让支付宝按规则重试。根因：忽略支付宝通知响应协议。
- 反例 6：错法：退款失败后更换 out_request_no 盲目重试。对法：同一退款意图保持幂等号，先查退款状态再补偿。根因：把网络失败等同业务失败，造成重复退款风险。
- 反例 7：错法：多次部分退款复用同一个 out_request_no。对法：每笔部分退款使用独立幂等号并记录剩余可退金额。根因：未区分退款意图与原支付订单。
- 反例 8：错法：把应用公钥配置成支付宝公钥，或只上传公钥不更新证书链。对法：区分应用私钥、应用公钥、支付宝公钥、应用公钥证书、支付宝公钥证书和根证书。根因：证书角色和信任方向混淆。
- 反例 9：错法：沙箱 app_id 配生产网关、生产证书或生产 notify_url。对法：按环境成套隔离配置，上线前生产小金额验证。根因：配置缺少环境维度和发布校验。
- 反例 10：错法：ISV 代调用漏传 app_auth_token 或复用其他商户授权。对法：平台身份与商户授权同时校验，授权失效进入重新授权流程。根因：多商户隔离和授权状态管理缺失。
- 反例 11：错法：转账、资金类接口仍用普通公钥模式。对法：按接口要求使用公钥证书模式并校验证书序列号。根因：以为下单能通代表全部资金接口能通。
- 反例 12：错法：日志打印完整私钥、证书、buyer_id、手机号、token 或账单明细。对法：日志脱敏，仅保留 trace_id、out_trade_no、trade_no、sub_code、版本和环境。根因：排障证据与敏感数据边界不清。
- 反例 13：错法：网络超时后直接再次下单生成新 out_trade_no。对法：先按原 out_trade_no 查单，再按幂等策略重试或关闭旧单。根因：外部超时状态未知却创建新交易。
- 反例 14：错法：账单对账只看总额，不逐笔核对退款、手续费、分账和跨日交易。对法：下载资金账单后逐笔核对并把差异进入补偿队列。根因：把通知成功误当账务闭环。
- 反例 15：错法：证书轮换只替换应用证书，遗漏支付宝公钥证书或根证书。对法：证书链成套更新，记录 cert_sn/alipay_root_cert_sn 并保留回滚。根因：不了解证书模式的链式校验。
- 反例 16：错法：支付宝小程序端直接拼支付参数或暴露私钥。对法：服务端生成交易与签名，小程序只接收受控参数并调用端能力。根因：把客户端调起能力误当服务端支付能力。
- 反例 17：错法：自动续费扣款失败后换新幂等号无限重试。对法：按协议状态和扣款幂等号限次重试，失败进入通知用户、暂停服务或人工处理。根因：把代扣失败当普通网络失败，忽略用户授权和合规边界。
- 反例 18：错法：花呗分期订单退款只退订单本金，不处理贴息、手续费和优惠退回。对法：按支付宝退款明细和账单字段核对买家实付、商户实收、贴息和手续费差异。根因：营销资金口径未入账务模型。
- 反例 19：错法：预授权冻结成功后转支付失败，直接重新冻结。对法：先查授权状态，按原授权单转支付或解冻，禁止重复冻结造成押金异常。根因：未把冻结、转支付、解冻拆成独立状态机。
- 反例 20：错法：投诉或风控冻结时尝试调整参数重提交易。对法：保留履约、授权、交易和客服证据，按支付宝规则申诉、退款或人工复核。根因：把合规事件误当技术失败。
- 反例 21：错法：my.tradePay 返回成功后立即发货或开会员。对法：客户端结果只做展示，服务端等待验签通知或查单成功后再发权益。根因：把客户端调起成功误当资金到账。
- 反例 22：错法：回调验签前先根据 out_trade_no 更新订单状态。对法：先按原始 POST form 验签，再校验业务字段并加幂等锁。根因：状态迁移早于可信性验证。
- 反例 23：错法：重复异步通知每次都新增资金流水。对法：以 trade_no/out_trade_no/事件类型做幂等，重复通知只记录审计不重复入账。根因：缺少通知幂等和终态保护。
- 反例 24：错法：只校 out_trade_no，不校 seller_id 和 app_id。对法：验签后同时校应用、商户、订单、金额和交易状态。根因：订单号命中被误认为业务归属可信。
- 反例 25：错法：退款接口成功就直接减少可退金额，不查退款结果和账单。对法：记录退款意图，结合退款查询、通知或账单确认到账。根因：把接口受理当作资金完成。
- 反例 26：错法：分账前不校验可分账余额和接收方，失败后人工改库。对法：分账前校资金口径和接收方，失败进入查询、补偿和人工审批。根因：分账被当普通业务字段更新。
- 反例 27：错法：生产上线只跑沙箱支付成功，没有生产小金额和通知证据。对法：生产灰度做小金额支付、验签通知、退款和账单下载验证。根因：沙箱通过被包装成生产可用。
- 反例 28：错法：私钥或完整证书为了排障打印到错误日志。对法：只输出配置版本、证书序列号、trace_id 和脱敏摘要。根因：把安全材料当普通配置值。
- 反例 29：错法：账单下载失败时手工把订单标平账。对法：保留差异单，重试下载、逐笔比对并走审批调账。根因：对账任务失败被业务状态覆盖。
- 反例 30：错法：沙箱 app_auth_token、生产 PID 和生产证书混在同一配置组。对法：按环境和商户维度隔离配置，发布前输出配置摘要和回滚点。根因：配置模型缺少环境/商户双维度。

## 与相邻技能的边界

- API 工程 / api-engineering（api）：负责支付 API 路由、响应结构、认证、错误码、回调暴露面和幂等接口契约；本技能负责支付宝字段、签名、通知语义和交易闭环。
- 后端工程 / backend-engineering（be）：负责服务端架构、任务队列、事务边界、配置加载、并发控制和运行时问题；本技能提供支付宝支付专项检查点。
- Web 安全 / web-security（wsec）：负责鉴权、CSRF、SSRF、回调暴露面、密钥管理、日志脱敏和攻击面审计；本技能只定义支付宝验签、敏感字段和风控边界。
- 数据库工程 / database-engineering（db）：负责订单、退款、分账、转账、账单表、唯一索引、事务和状态机；本技能负责 trade_no/out_trade_no/out_request_no/out_biz_no 等支付宝幂等号和资金字段约束。
- 支付宝小程序 / alipay-miniprogram（almp）：负责 AXML/ACSS、my.* API、基础库、端内授权、审核和小程序端支付调起；本技能负责服务端交易、签名、回调和账务闭环。
- 发布部署 / release-engineering（rls）：负责证书、网关、notify_url、环境变量、灰度、小金额验证、回滚、发布窗口和生产开关；本技能定义支付宝发布风险样本。
- 可观测性 / observability（obs）：负责指标、日志、trace、告警、SLO、补偿任务观测和事故复盘；本技能定义支付链路必须保留的证据字段和告警触发点。
- 测试验证 / test-engineering（tst）：负责复现、沙箱、异常通知、回归矩阵、自动化和发布冒烟；本技能提供支付风险样本与验收口径。
- 代码审计 / code-audit（aud）：代码改动完成前最终收口，查影响面、证据、状态机、安全和遗漏；本技能提供支付宝专项审计点。
- 研究调研 / research（rsch）：只在需要查支付宝开放平台最新政策、接口变更、SDK 发布说明和 2024-2026 外部资料时使用；涉及新产品、资金类接口、自动续费、预授权、跨境、SDK 大版本或证书要求变化时必须触发。
- 微信支付 / wechat-pay（wx）：只负责微信支付；不要把微信的分整数、JSON v3、平台证书、回调语义套到支付宝。
- 语言技能：godv、jv、phpd、pyd、jsts 等负责具体 SDK 代码；本技能不替代语言实现。
