# Core 总览

这一层是**工具中立的工作流核心**。任何 Agent runtime（Claude Code、Codex……）都直接消费这层约定，不应在这里出现任何一家 Agent 的私有概念。

## 阅读顺序

| 文件 | 作用 | 何时读 |
|---|---|---|
| `01-workflow.md` | 10 阶段工作流骨架 | 任务跨扫描/执行/验证/交付多个阶段时 |
| `02-state-systems.md` | 四态系统的发现与读写规则 | 需要确认状态从哪里恢复时 |
| `03-routing.md` | 意图 → 执行能力 的路由矩阵 | 任务语义模糊、多条路线像候选时 |
| `04-authority.md` | 多层规则的权威解析 | 项目结构复杂、存在多个规则层时 |
| `05-validation.md` | 验证 / 文档 / 交付 三道闸门 | 准备声明完成、提交、交付时 |
| `06-evolution.md` | 重复模式的保守沉淀机制 | 发现旧路径、漂移、重复解释时 |
| `07-artifact-contracts.md` | 审计、验证、文档和状态产物契约 | 需要自动生成报告、索引、日志或同步状态时 |

## 中立性约束

写这一层时遵守：

- 只用通用词汇：**agent / 状态 / 路由 / 闸门 / 委托 / 沉淀**。
- 不写任何一家工具的入口文件名、配置键、私有命令（那些归 `adapters/`）。
- 不硬编码项目内的具体文件名，只描述"如何发现"。
- 描述**约定和顺序**，不强制特定脚本或工具链。

## 一句话心智模型

> 先看清项目（scan）→ 恢复状态（state）→ 判断意图（intent）→ 解析权威（authority）→ 路由委托（route）→ 干活并留下产物（execute）→ 验证（validate）→ 回写状态与索引（sync）→ 交付闸门（deliver）→ 沉淀（evolve）。
