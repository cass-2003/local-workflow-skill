# 验证闸门

完成声明必须有证据，且证据范围与改动范围匹配。这道闸门防止"完成只靠口头声明"。

## 验证级别

| 级别 | 适用场景 | 期望证据 |
|---|---|---|
| V0 | 纯分析 | 源码 / 文档检查与一致性说明 |
| V1 | 纯文档或纯 workflow 调整 | 路径校验、引用校验、结构一致性检查 |
| V2 | 单模块代码改动 | 局部 build / typecheck / test |
| V3 | 跨模块行为改动 | 多模块验证、接口或契约检查 |
| V4 | 交付面 / 高风险 / 运行时敏感改动 | 更强验证、回归或 smoke check |

## 验证方式选择顺序

1. 项目本地 validation 能力
2. 项目文档里规定的验证命令
3. 仓库原生脚本
4. 最小可行的全局通用验证

## 改 workflow / 框架本身时的专项检查

本次工作主要在改框架核心、能力库或四态约定时，额外检查：

- 引用路径是否真实存在
- authority 层描述是否和真实结构一致
- 是否引入了重复真相源
- 主文档与 reference 是否分工清晰

## 记录规则（始终）

- 验证了什么
- 用了什么命令或证据
- 有什么没验证
- 为什么没做更宽的验证

没有验证时，也要明确写出"未验证"。

## 三道闸门串联

验证闸门只是交付前的第一道。完整交付前过三道：

1. **验证闸门**（本文件）：成果有匹配范围的证据。
2. **文档闸门**（`01-workflow.md` Phase 7 + `02-state-systems.md`）：四态系统已回写。
3. **交付闸门**（`01-workflow.md` Phase 8）：diff 已查、无意外破坏性 Git 动作。
