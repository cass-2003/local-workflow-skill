# 四态系统

> 框架的核心。回答一个项目被多 Agent、多会话持续推进时，最需要的四个问题。

| 系统 | 回答 | 模板 |
|---|---|---|
| 🗒️ 日志系统 | 最近发生了什么 | `templates/LOG.md` |
| 📋 需求系统 | 当前要满足什么 | `templates/REQUIREMENTS.md` |
| 🧠 记忆系统 | 为什么这样设计 | `templates/MEMORY.md` |
| 📊 进度系统 | 现在做到哪里 | `templates/PROGRESS.md` |

## 为什么是这四个

普通 prompt 工程的上下文是**易失**的：会话一结束、工具一切换，Agent 就"失忆"。四态系统给上下文四个**可持久、可交接、可审计**的落点，于是：

- **跨会话**：新会话先读四态系统就能恢复，不靠用户重述。
- **跨 Agent**：Claude Code 写的状态，Codex 也能读 —— 因为它们都是纯 Markdown。
- **可审计**：每次变更回写到对应系统，历史可回看。

## 怎么用

1. **新项目**：把 `templates/` 四个文件复制到项目根（或 `.agent-os/`、`docs/state/` 等你偏好的目录）。
2. **已有项目**：先按 `../core/02-state-systems.md` 的发现规则，找项目里已存在的等价文档，把它们认领为四态系统的承载体；缺哪个补哪个。
3. **每次开工**：先读四态系统恢复状态（工作流 Phase 1）。
4. **每次收尾**：把变化分类回写（工作流 Phase 7）。

## 读写时机（与工作流对齐）

| 工作流阶段 | 对四态系统的动作 |
|---|---|
| Phase 1 State Restore | **读**全部四个，输出 State Restore 记录 |
| Phase 5 Execute | 按需**读**记忆 / 需求作为约束 |
| Phase 7 Docs & State Sync | 把成果分类**写**回对应系统 |

## 不强制文件名

上面的文件名是**默认落地形态**，方便从零起步。框架真正认的是四个**角色**，不是四个固定文件名。一个项目可以：

- 用别的文件名（`CHANGELOG.md` 当日志、`PRD.md` 当需求……）
- 一个文件兼任两个角色
- 把四态拆到子目录里分文件管理

只要在状态恢复时能说清"哪个文件承担哪个角色"即可。
